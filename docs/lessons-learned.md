# Lessons Learned — Bug-Pattern Catalog

Living catalogue of bug *shapes* (not specific bugs). When a new symptom matches a signal here, jump to the rule before debugging.

Format per lesson:
- **Rule** — what to do.
- **Why** — the failure mode that motivated it (one concrete bug).
- **Signal** — what you'll see if you're about to hit this class.
- **Apply** — concrete fix recipe.

## Cube model

### Time dimensions must resolve to TIMESTAMP, not DATE
- **Rule:** any Cube `type: time` dimension must be backed by a SQL expression that yields TIMESTAMP. Cast at the source CTE / column expression, not in a separate dimension wrapper.
- **Why:** Cube wraps time dims with `<col> AT TIME ZONE 'UTC'` and emits `from_iso8601_timestamp(?)` for `dateRange` filters. Trino rejects either operation on DATE — and Cube's generated SQL has no opportunity to coerce after the fact. The cohort retention cube shipped with `MIN(log_date) AS install_date` (DATE), so every `/load` 400'd and `/liveops/cohort` was stuck on "Detecting data path…".
- **Signal:** Cube `/load` returns 400 with `"Type of value must be a time or timestamp with or without time zone (actual date)"`, or any error referencing `AT TIME ZONE` / `from_iso8601_timestamp` on a DATE column.
- **Apply:** `CAST(<col> AS TIMESTAMP) AS <alias>` at the inner SELECT or aggregate. Don't try to patch via the dimension's `sql:` — Cube still emits the `AT TIME ZONE` wrap around your expression, so the inner type still needs to be TIMESTAMP.

### Ratio measures over integer counts must CAST to DOUBLE, not `* 1.0`
- **Rule:** a ratio measure whose **numerator is an integer count** (`count` / `count_distinct`) must force floating-point division with `CAST({numerator} AS DOUBLE) / NULLIF({denominator}, 0)`. Do NOT use the `{numerator} * 1.0 / …` idiom when the numerator is integer.
- **Why:** `mf_users.paying_rate = {paying_users} * 1.0 / NULLIF({user_count}, 0)`. `paying_users` is `count_distinct` → bigint; `bigint * 1.0` is `decimal(_,1)`. Trino decimal division keeps the **dividend's scale** (1 here), so a real rate of 0.69% (`5835/841072`) truncated to scale-1 = `0.0`. Every country/game came back `0.0` because lifetime paying rates are all <5%. The sibling `arpu_vnd` used the same `* 1.0` idiom but was fine — its numerator is a `sum` of a DOUBLE column, so division stayed floating-point. So the bug only bites count/count ratios.
- **Signal:** a percent/ratio measure reads exactly `0` (often the string `'0.0'`) while its component count measures are clearly non-zero in the same row; small ratios all collapse to 0 while large ones (ARPU) look fine. Compiled SQL looks arithmetically correct — the loss is in Trino's decimal scale, not the formula.
- **Apply:** `CAST({count_measure} AS DOUBLE) / NULLIF({denom}, 0)`. Audit every `* 1.0 /` ratio in the model: numerator a `sum` of a DOUBLE column → fine; numerator a `count`/`count_distinct` → must cast. Verify with a live `/load` (a known >0 case should show real decimals, not `0.0`). Note: stale `0.0` may persist in `response_cache` / pre-aggregations until cleared or rebuilt.

### Cross-cube measure dedup relies on a UNIQUE primary key (joins fan out; the PK un-fans)
- **Rule:** every cube's `primary_key` must be unique per physical row. When a cube is aggregated across a one-to-many join, Cube does NOT sum the fanned rows directly — it de-dups the measure on the primary key first (`COUNT(DISTINCT pk)` for counts; a `SELECT DISTINCT (dims, pk)` "keys" CTE re-joined `ON pk` for sums). The `joins:` blob only decides *which rows match*; the **primary key decides how the fan-out is collapsed**. A non-unique PK silently corrupts both — *deflates* counts (real rows hide behind one pk) and *inflates* sums (the keys re-join matches unrelated rows). If no single column is unique, build a composite.
- **Why:** jus `recharge` declared `primary_key: true` on `transaction_id` whose `sql:` was `transid`. In the Netease/JX `etl_ingame_recharge` schema `transid` is NOT row-unique — one payment splits into multiple rows by `role_id` + `prepaid_detail_item_id`, and values are reused across days. The ARPDAU query (`recharge.revenue_vnd` + `active_daily.dau`, joined through `mf_users` on `user_id`) fanned recharge out by each user's active-days; `transid` couldn't dedup it, so revenue inflated ~800× (774B vs the true 601M) while `transactions` deflated (3892 physical rows → 677 distinct transid). ballistar/cfm ran the *identical* query plan but used genuinely unique txn columns (`transaction_id` / `vng_transaction`) → single-cube and combined numbers matched exactly. The bug was invisible in single-cube queries (no join ⇒ no dedup ⇒ correct).
- **Signal:** a measure is correct alone but wrong once another cube is added to the same query; `count` drops and `sum` balloons relative to the single-cube value; compiled SQL (`/cube-api/v1/sql`) shows a `SELECT DISTINCT … <pk>` keys CTE re-joined `ON <pk>` with no date filter on the re-joined table. Fast check: run the SAME measure single-cube vs combined-with-a-fan-out-cube — they must be equal.
- **Apply:** choose a PK that is provably one-row-per-grain; if none exists, use a `CONCAT(COALESCE(...))` composite (jus: `account_id__pay_time__transid__role_id__prepaid_detail_item_id`) and demote the natural id to a plain dimension. Audit dev PKs against `cube-prod` (`/Users/lap16299/Documents/code/cube-prod`, the validated oracle) for any game present in both; for games absent from prod, run the single-vs-combined test before trusting. Match the *uniqueness intent*, not the literal column — dev and prod are different schemas (cfm dev `vng_transaction` ≠ prod `transaction_id`, both unique → leave it). Reload after a PK edit: `docker restart cube-playground-cube-api-dev` (DEV_MODE=false ⇒ no hot-reload).

### Preset / starter-pack YAML must match live `/meta`
- **Rule:** every measure / dimension referenced by a starter dashboard, KPI config, or anomaly metric must be present in the active game's `/cubejs-api/v1/meta` payload — exact path, exact casing.
- **Why:** `user_recharge_daily.recharge_date` was hard-coded across the starter pack, FE KPI config, anomaly config, and a refresh handler — but the cube exposes `user_recharge_daily.log_date`. Result: every cohort/Revenue/Paying-users tile cached `status: broken` with `"'recharge_date' not found for path …"`.
- **Signal:** `Cube /load → 400: …'<member>' not found for path '<cube>.<member>'`. Also `Invalid query format` for any unknown field.
- **Apply:** before editing presets, diff against `curl <cube>/cubejs-api/v1/meta` for each game in the rotation. Treat the preset's measure/dim list as a typed reference into `/meta`. Add a fast sanity script (`scripts/check-metric-drift.ts` already exists for this) and run it on preset edits.

### Cube `/load` rejects unknown query fields
- **Rule:** internal-only hints (renderer flags, compare modes, viz prefs) must be stripped before the query reaches Cube. Treat the persisted query JSON as a superset; the over-the-wire shape is a subset.
- **Why:** the dashboard starter pack stored `"compare":"prev"` on each KPI tile for a deferred compare-prev render. The tile-refresh cron passed the raw JSON to `/load`, which 400'd with `"compare" is not allowed`. Every tile cached broken.
- **Signal:** `Invalid query format: "<field>" is not allowed`. The field is one your renderer added, not a Cube concept.
- **Apply:** maintain an `INTERNAL_QUERY_FIELDS` allowlist next to the refresh job; `delete` those keys before `loadWithContinueWait`. Mirror the same strip on any other code path that ships persisted tile/segment queries to Cube.

### Scope cohort queries by predicate, not by an inlined uid-IN list
- **Rule:** to scope a query to a materialized cohort, AND on the cohort's *predicate filters* — not an `IN (...)` filter listing every member id. Inlining ids puts the whole list into the query text; a million-member cohort serializes to MBs of JSON and Cube 400s with `Query text length (N) exceeds the maximum length (1000000)`. Predicate-scoping is also the only correct option for ratio/average measures, which cannot be re-merged across id batches.
- **Why:** the segment card-runner scoped each preset KPI/card by `{ member: identityDim, operator: 'equals', values: uids }` against the segment's stored `uid_list` (capped at 100k, but "all-users" segments materialized ~1M). A single card query hit 2.3MB → HTTP 400, and every card for that segment dropped out of the precompute cache. The segment's own *size* was already computed from the predicate (not the uids), so scoping cards by uids was both fragile and inconsistent with the displayed size. Fix: pass the segment's `cube_query_json.filters` into `runPresetCards` and AND them onto each card query; the "all-users" case collapses to `filters: []` (tiny, full-cohort-correct).
- **Signal:** `Cube /load → 400: "Query text length (…) exceeds the maximum length (1000000)"`; the failing queries differ only by a giant `values: [...]` array; ratio measures (ARPU, paying-rate) that can't be reconstructed if you tried to batch the id list.
- **Apply:** scope by the cohort-defining predicate, mirroring whatever computes the cohort's authoritative size. Reserve identity-IN filters for genuinely small, explicit id sets (e.g. a paginated visible page), never the full materialized list. Both sides were fixed: server `card-runner.ts` and the FE `use-segment-cube-query.ts` (`scopeQueryToCohort` picks predicate-scoping for predicate segments, identity-IN only for `uidsOverride` pages and predicate-less manual segments). While here, the card-runner was also switched from bare `load()` to `loadWithContinueWait` — pre-agg-backed cards (LTV install-cohort rollup) were transiently dropped on Cube's `Continue wait` response when the rollup was mid-warm.

### Snapshot artifacts must persist their defining slice, or twin surfaces diverge
- **Rule:** when a creation flow materializes a snapshot from a query (static segment from a cohort pick), persist the originating filter as context even though membership doesn't need it. Any surface that later *re-queries* around the snapshot (metric cards, monitors) needs that slice to answer the same question as a live twin.
- **Why:** static-segment creation set `predicate_tree: null` (only Live built it), so detail cards scoped manual segments by identity-IN alone with each card's own window. A static and a Live segment built from the SAME playground pick ("paid on Jun 5", identical 3,967 uids) showed ₫47.94B (lifetime spend of those users) vs ₫1.74B (the Jun-5 slice) — reported as "the frozen list is broken" when membership was in fact identical.
- **Signal:** two segments with 100% identical uid lists but wildly different card numbers; a static segment's Definition tab saying "no predicate to display" for a cohort that clearly originated from a filter.
- **Apply:** build the predicate tree on EVERY query-builder push (`push-modal.tsx` — Live requires the context, static stores it best-effort); server create/PATCH already persist + translate it type-agnostically. FE `scopeQueryToCohort` scopes manual-with-slice as `identity-IN ∧ slice` (with the same date-pin window-drop as predicate scoping). Membership stays the frozen uid_list — the slice is card context only, never a refresh driver (`isPredicateWithQuery` and the refresh route stay predicate-gated).

### A rollup only serves a query if it's a superset — including the EXACT time dimension
- **Rule:** before adding/relying on a rollup, confirm the query's bound time dimension *is* the rollup's `time_dimension`. cfm event cubes expose both `log_date` (DATE) and `dteventtime` (event timestamp); the UI/agent bound their queries on `dteventtime`, but the shipped rollups were keyed on `log_date`. A rollup must be a superset of the query (every measure, every dimension, AND the same time dim) or Cube silently falls through to a full source scan.
- **Why:** 118/122 seeded cfm_vn queries served `raw` despite rollups existing on every event cube — the only mismatch was time dim (`dteventtime` query vs `log_date` rollup). Same logical query keyed on `log_date` routed to `prod_pre_aggregations.*`; keyed on `dteventtime` it scanned `etl_ingame_login` (~5s).
- **Signal:** a query that "should" hit a rollup is slow / `source: raw`; compiled SQL (`/cube-api/v1/sql`) reads `FROM <raw_source_table>` not `FROM prod_pre_aggregations.*`; the rollup's measures+dims match but its `time_dimension` differs from the query's `timeDimensions[].dimension`.
- **Apply:** add a `*_ts_batch` sibling rollup keyed on `dteventtime` (mirror the existing `*_batch` measures/dims + a `*_ts` `rollup_lambda` union), or switch the existing rollup's `time_dimension` if the cube has no `log_date` demand. Keep granularity `day`, `partition_granularity: month`. Verify with the compiled-SQL FROM clause per cube — do NOT trust `usedPreAggregations` (see lambda note below).

### dteventtime-keyed rollups must cap build_range_end at current_timestamp
- **Rule:** when a rollup's `time_dimension` is a tz-carrying timestamp (`dteventtime`), set `build_range_end` to `SELECT LEAST(MAX(dteventtime), current_timestamp) FROM <source>`. `log_date`-keyed rollups (DATE wrapped to midnight UTC) don't need this.
- **Why:** the open (current-month) partition's end is computed in the query timezone. The UI sends `Asia/Saigon` (UTC+7), so an uncapped end lands ahead of UTC "now" and CubeStore rejects the partition with `Internal: second time provided was later than self`, so it never seals. `log_date` sidesteps it because the DATE→midnight-UTC value is never future.
- **Signal:** worker logs `second time provided was later than self`; a `*_ts` rollup never seals while its `log_date` twin builds fine.
- **Apply:** the `LEAST(..., current_timestamp)` cap (already used in the cfm `*_ts_batch` rollups). Confirm clean seal with `cube-dev/scripts/trigger-preagg-build.sh <game>` (reports per-rollup sealed/errored).

### Only additive measures roll up correctly; verify routing by compiled SQL, not usedPreAggregations
- **Rule:** a rollup may only carry **additive** measures: `count`, `sum`, `min`, `max`, `count_distinct_approx`. `avg` and exact `count_distinct` are NON-additive — a day-grain rollup returns wrong numbers when a query re-aggregates across days. Remodel as `sum`+`count` (compute the avg as a ratio measure) or switch to `count_distinct_approx` before rolling up.
- **Why:** `etl_newbie_detail`'s demanded measures are `avg_*` + `distinct_players` — a naive rollup would silently serve wrong averages, so it was deferred rather than shipped. `active_daily`'s rollup is safe because `dau`/`paying_dau` are `count_distinct_approx` and `total_online_time_sec` is `sum`.
- **Signal:** a rollup-backed `avg`/percentile reads plausibly but disagrees with the raw query once the date span > the rollup granularity; `usedPreAggregations` in the `/load` JSON is empty even though the query is fast.
- **Apply:** check each measure's `type:` before adding it to a rollup. For routing proof, read the compiled SQL FROM clause via `/cube-api/v1/sql` — a `rollup_lambda` (`union_with_source_data: true`) always unions live source, so it reports `usedPreAggregations: []` even when sealed partitions serve. Also: after model edits the SERVING instance needs a restart (DEV_MODE=false = no hot-reload) before it knows the new rollup — restart `cube_api`/`cube_api_dev`, not just the worker; and `CREATE TABLE prod_pre_aggregations.…` build lines only log at trace level.

### Local runs TWO cube stacks with separate cubestores — build into the wrong one and `:11000` never serves
- **Rule:** local has two independent cube stacks that both call their workspace `local` but share NO data. (1) **Dev pair** — `cube_api_dev` + `cube_refresh_worker_dev` + `cubestore_dev`, host `:4000`, what `npm run dev:all` / the Vite `:3000` dev server use. (2) **Non-dev / deploy-mirror** — `cube_api` + `cube_refresh_worker` + `cubestore`, served via web `:11000` → `cube_api:4000` (also published on `:17001`), defined in `docker-compose.prod.yml` (+ `docker-compose.local.yml` overlay). Each `cube_api` reads ONLY its own cubestore (`CUBESTORE_HOST`); the two `cubestore`s are distinct Docker volumes (`cubestore_dev_data` vs `cubestore_data`). `trigger-preagg-build.sh` builds into `cubestore_dev` ONLY.
- **Why:** a rollup sealed via `trigger-preagg-build.sh` (dev pair) routes perfectly on `:4000`/`:3000` but is completely invisible to `:11000` — its cubestore was never built. Cost a full debug cycle of "rollup committed + sealed, yet `:11000` still scans raw source."
- **Signal:** same query routes to `prod_pre_aggregations.*` on `:4000` but reads `FROM <raw_source>` on `:17001`/`:11000`; the two `cubestore` volumes differ in size; `docker ps` shows both `cube-api` and `cube-api-dev` running.
- **Apply (full `:11000` rebuild):** (a) `docker restart cube-playground-cube-api` — DEV_MODE=false means no hot-reload, so the serving instance must restart to recompile new model YAML; wait for `/readyz`. (b) recreate the worker UNSCOPED (all games) to backfill its own cubestore, with a fast sweep + trace so builds are monitorable: `CUBEJS_SCHEDULED_REFRESH_TIMER=30 CUBEJS_LOG_LEVEL=trace docker compose -f docker-compose.prod.yml -f docker-compose.local.yml --env-file .env.docker.local up -d --no-deps --force-recreate cube_refresh_worker` (CUBE_REFRESH_GAMES unset = every supported game). (c) monitor worker logs for `CREATE TABLE prod_pre_aggregations.` going quiet (seal-race errors `…after it was successfully created` / `later than self` are benign — they re-seal). (d) verify with `CUBE_HOST=http://localhost:17001 python3 cube-dev/scripts/probe-preagg-routing.py` — routing in the compiled SQL is the only proof. (e) restore normal cadence: re-run the recreate WITHOUT the timer/log overrides (back to 300s/info). Same secret (`local-dev-secret-…`) signs JWTs for both stacks. Partitions are built per timezone — keep `UTC,Asia/Saigon` or the UI's Saigon-tagged queries miss the rollups.

### A partitioned rollup only ever materialises its end-month partition on the prod-mode stack — historical months stay empty
- **Rule:** on the local prod-mode stack (`DEV_MODE=false`, on-demand/query-triggered builds disabled), the scheduled refresh of a `partition_granularity: month` rollup builds ONLY the partition holding `build_range_end` (the current month). Earlier months in `[build_range_start, build_range_end]` are never enumerated for building, so a query on a mature month routes to the batch rollup and returns **0 rows** even though the source table has that month's data. Adding `update_window` (even 400 day) does NOT change it; scoping the worker to one game does NOT change it — the worker logs show every build attempt targeting the same `…batch<end-month>` table and zero attempts at older months. This is distinct from CubeStore corruption (clock-skew `second time provided was later than self`, unhealthy store), which is a SEPARATE failure that a `colima restart` + volume wipe clears.
- **Why:** cfm/jus D30 churn returned 0 for every cohort older than the current month; the cohort-retention rollup was monthly-partitioned, so only June sealed. Probing `active_daily.dau` for Feb–May 2026 returned 0 fast (rollup hit, empty partition) while last-7d returned correct DAU; the worker built `active_daily_dau_by_country_payer_daily_batch20260601` 100+ times and never attempted `…batch20260201/0301/0401/0501`. Source had the data the whole time (the non-partitioned cohort rollup, scanning the same source, reported Feb cohort_size 926k for cfm / 29k for jus).
- **Signal:** a partitioned-rollup-backed query returns 0 rows *fast* for a historical month but correct data for the current month; worker `CREATE TABLE preagg_<game>.…batch<YYYYMMDD>` lines only ever carry the current month's date; compiled `/cubejs-api/v1/sql` reads `FROM preagg_<game>.<rollup>_batch…` (routing is fine — the partition is just absent).
- **Apply:** for a SMALL aggregate (≈one row per day, e.g. cohort retention) drop partitioning entirely — a plain non-partitioned `rollup` builds the full history in one `CREATE TABLE` and serves every period (done for cfm + jus `retention.cohort_retention_batch`). For LARGE/high-cardinality rollups (`active_daily`, `billing_detail`, `mf_users`) non-partitioning is wrong (huge single table, no incremental refresh) — accept that on this stack they serve the recent window from the rollup and deep-historical falls to the `union_with_source_data` lambda (slower, source-direct) or is out of scope for local. After any structural rollup change, `docker restart cube-playground-cube-api-dev` (serving instance won't hot-reload) before the new shape serves.

### A custom-`sql` cube's partition prune must be a scalar subquery, not a CROSS-JOINed anchor value
- **Rule:** when a cube's `sql:` filters a partition column (`log_month`) by a bound derived from a CROSS-JOINed CTE (e.g. `WHERE g.log_month >= substr(a.d - 3d, …)` with `a` an anchor CTE), Trino CANNOT use it for partition pruning — the join value isn't known at plan time, so it scans the FULL table history. Make the prune predicate a self-contained scalar subquery: `WHERE g.log_month >= (SELECT substr(CAST(MAX(...) - INTERVAL '3' DAY AS VARCHAR),1,7) FROM <table>)`.
- **Why:** `user_gameplay_daily` over raw `etl_ingame_game_detail` (~1M matches/day, monthly partitions) took 17s cold (scanning Dec→May, ~150M rows) with a CROSS-JOIN-derived bound; the same query with a scalar-subquery `log_month` bound pruned to 2 months (~7.4M rows) and dropped to 8s — back under the 15s Cube client timeout.
- **Signal:** a custom-SQL mart query is far slower than the equivalent direct behavior-cube query with a static `dateRange` (which prunes); EXPLAIN / scan-row-count covers months you filtered out.
- **Apply:** keep the CROSS-JOIN anchor for SELECT-list window logic, but express every partition-column WHERE bound as an independent scalar subquery. Aggregating-before-join helps join cost but does NOT fix a missing partition prune — the scan dominates.

### Heavy raw-event marts can't back an `event`-window playbook — the anchor probe times out
- **Rule:** a care playbook with `kind:'event'` (relative window like `last 48 hours`) triggers a SECOND Cube query per sweep — `resolveDataAnchor` probes `ORDER BY <member> DESC LIMIT 1` to find the data anchor. On a heavy raw-event mart that probe runs the full cube SQL and times out (15s), then fail-safes to `new Date()` → the window resolves to today (no data) → empty cohort. Model the event as a 1/0 flag dimension on the mart and gate with `kind:'abs', op:'equals', value:1` instead — the trailing window is baked into the mart's own cross-window diff, so no anchor probe is needed.
- **Why:** PB10/17 (clan switch / clan left) on `user_gameplay_daily` returned 0 (anchor probe timed out → window = today, data ends 2026-05-01); switching them to `clan_switched_recent` / `clan_left_recent` flags + abs(=1) made them return 213 / 156 and removed the extra query.
- **Signal:** an `event` playbook reports `cohortSize 0` or `query-failed` while a direct `<member> is set` probe returns a real cohort; sweep latency spikes on that playbook.
- **Apply:** reserve `kind:'event'` for members on light daily marts (e.g. `mf_users.first_recharge_date`). For anything computed over a raw-event mart, emit the recency as a flag column and use `abs`.

### A YAML folded scalar (`>`) collapses base-indent `--` comments onto one line — the comment eats the next statement
- **Rule:** in a `sql: >` block, lines at the block's BASE indent are folded together with spaces; a `--` line comment at that level therefore swallows whatever follows on the folded line (e.g. a CTE keyword), producing `mismatched input 'SELECT'`. Put any `--` comment at a DEEPER indent than the base (inside a clause), where folding preserves the newline — or use `/* … */`.
- **Why:** comments placed between CTEs (`), -- note\n  next AS (`) folded to `), -- note next AS (`, commenting out `next AS (` → Trino syntax error. The same comment moved one line down, just after the CTE's `SELECT`, compiled fine.
- **Signal:** a Cube `/sql` dump shows several source lines joined onto one line with a trailing `-- …`; the parse error points at the token right after where a CTE/clause keyword should be.
- **Apply:** never leave a `--` comment on a foldable (base-indent) line; verify by reading the compiled SQL via `/cube-api/v1/sql`, not just the YAML.

### A surface's identity/enrichment view is a silent per-game dependency the availability check can't see
- **Rule:** before declaring a per-game surface "done", inventory the views/cubes its code reads **directly by name** — not just the members gated by an availability/coverage check. Identity and enrichment views (display name, profile) are queried unconditionally; if the game lacks one, the surface degrades silently (raw ids, blank fields) while every gated feature still reports "available".

### OAuth token precedence: API key out-ranks OAuth unless the env is scrubbed
- **Rule:** the Claude Agent SDK reads creds from env in a fixed order: API key (`ANTHROPIC_API_KEY`) wins over OAuth token (`CLAUDE_CODE_OAUTH_TOKEN`). To force OAuth, spawn the SDK in a subprocess with a clean env that STRIPS every API-key/gateway var (including `ANTHROPIC_BASE_URL`, `ANTHROPIC_API_URL`, `ANTHROPIC_AUTH_TOKEN`, `CLAUDE_CODE_USE_BEDROCK`, `CLAUDE_CODE_USE_VERTEX`). Never mutate the long-lived server process.env. Accept both canonical (`CLAUDE_CODE_OAUTH_TOKEN`) and Vault names (`ANTHROPIC_SUBSCRIPTION_OAUTH_TOKEN`) at the source; inject under the canonical name so the SDK reads it correctly.
- **Why:** the Optimization Advisor agent must run on the subscription OAuth lane (rate-limited, quota-managed) never the API key (shared, easily drained). First implementation passed the API key in `process.env` to the SDK subprocess; the SDK silently upgraded to key-auth, drained the shared quota within hours, and every agent turn then 503'd with a "key exhausted" error. Vault sets the Vault name (`ANTHROPIC_SUBSCRIPTION_OAUTH_TOKEN`), not the SDK canonical name, so the token was visible but ignored (key ranked higher). After hour-long debugs probing upstream service status, the fix was: `buildAgentEnv()` strips API keys → uses OAuth precedence.
- **Signal:** agent turns 503 oauth_unavailable despite Vault provisioning a token (check Vault name matches canonical), OR agent silently switches lanes post-deployment (key-based auth succeeds locally, fails on prod where key is unset). View logs: if the agent logs `"model": "claude-sonnet-4-6"` on turns but your default is a different model, you're using the gateway key (gateway pins sonnet-only). Also: `ANTHROPIC_SUBSCRIPTION_OAUTH_TOKEN` (Vault name) is not the name the SDK reads — it's a confusing mismatch.
- **Apply:** `agent-oauth-env.ts` `buildAgentEnv()` creates a clean child env: copy nothing by default, explicitly allowlist only non-cred vars (workspace, game, debug flags). Strip all `ANTHROPIC_*` except the canonical token. Accept the token at source from either name, canonicalize to `CLAUDE_CODE_OAUTH_TOKEN` before passing to the SDK subprocess. Test locally with API key unset (`unset ANTHROPIC_API_KEY`) to verify OAuth is the fallback — agent should work. Test on prod where API key is unset → agent works. If Vault provides under a different name, the source-acceptance layer resolves it before injection. See `server/src/advisor/agent/agent-oauth-env.ts`.
- **Why:** jus_vn's CS queue showed raw uids. All playbooks were "available" (their `dataRequirements` members existed), but the VIP profile fetcher also queries `user_roles_panel.last_role_name` for the display name — a view jus never had (only cfm/cros/tf did). The availability check only inspects playbook `dataRequirements`, so the missing identity view was invisible to it. The name data existed (`mf_ingame_roles`, ~100% populated) — only the cube + view were missing. Also: the fetcher reads the **view** (`user_roles_panel`), so adding just the `user_roles` cube wasn't enough — the `user_360.yml` view block was the other half.
- **Signal:** a per-game surface renders fallback values (raw uid, em-dash, blank) for one game while another game is fine, and no error/`unavailable` is raised. Grep the surface's fetch code for view/member names it queries without a presence guard.
- **Apply:** port the canonical game's full view set (`views/<game>/user_360.yml`) and backing cubes, not just the playbook marts — see `docs/game-onboarding-data-yaml.md` (Tier 0). After landing YAML, refresh stored snapshots (`upsertVipProfiles`) — existing rows keep stale `NULL` until re-fetched.

### Editing a rollup def opens a serve-ERROR window until the worker rebuilds it
- **Rule:** changing a rollup's measures/dimensions/segments mints a NEW version hash. Queries route to the new version IMMEDIATELY (compiled SQL points at it), but in this prod-mode stack (`CUBEJS_DEV_MODE=false`, API `CUBEJS_REFRESH_WORKER=false`) NOTHING builds on query — only the scheduled refresh worker seals partitions. So between the model edit and the next build, every query for that shape ERRORS. Always trigger a build right after the edit (dev) and right after the deploy-restart (prod), or you ship a regression window on a query that previously worked (slow, raw) and now errors.
- **Why:** extending `mf_users.user_composition_batch` with churn_risk/segments flipped churn_risk queries from raw-Trino MISS to routing at the new rollup version. Until `trigger-preagg-build.sh` sealed it, `/load` returned `"No pre-aggregation partitions were built yet … this API instance wasn't set up to build pre-aggregations"` and `/sql` showed routing=HIT but 0 rows. The routing was correct; the table just didn't exist yet.
- **Signal:** right after a rollup edit, `/sql` shows `usedPreAggregations` populated (HIT) but `/load` errors with "No pre-aggregation partitions were built yet" / returns 0 rows; the same query worked (slowly) before the edit.
- **Apply:** after any rollup-def change, build before declaring done: `cube-dev/scripts/trigger-preagg-build.sh <game>` (or scope the worker to all games via `CUBE_REFRESH_GAMES` for one sweep), confirm SEALED, then verify serve returns real rows. On prod, the deploy-restart must be followed by a build sweep. Verify routing (not serve) is build-independent — good for a fast pre-build coverage check, but never mistake routing-HIT for serving.

### A rollup can serve segment-scoped queries only if it lists those `segments:`
- **Rule:** when cards/queries carry a cube `segment` (e.g. `segments: [mf_users.whales]`), a rollup matches that query ONLY if the segment is named in the rollup's `segments:` list — even when the segment's condition overlaps a dimension already in the rollup (whales ≡ payer_tier='whale'). Cube does not infer the equivalence; it materializes each listed segment as a boolean column and filters on it.
- **Why:** segment-sizing cards are commonly scoped by `whales` / `at_risk_paying` / `paying_lifetime`. The composition rollup had `payer_tier`/`is_paying_user` dims but no `segments:` block, so whale-scoped cards fell through to a raw scan despite payer_tier being present. Adding `segments: [whales, at_risk_paying, paying_lifetime]` made them route; cross-checked rollup approx vs raw exact within HLL tolerance (0.5–1.0%).
- **Signal:** a group-by card routes to a rollup, but the SAME card scoped by a cube segment misses (compiled SQL `FROM <raw_source>`); the rollup has the dim the segment is equivalent to but no `segments:` entry.
- **Apply:** list the segments cards actually use in the rollup `segments:`. Each listed segment must be a row-level predicate (additive-safe). Guard coverage with a live probe of segment-scoped shapes (`cube-dev/scripts/check-mf-users-rollup-coverage.py`).

## Server cache contract

### Empty-placeholder rows must not be served as `200 OK`
- **Rule:** a cache row whose payload has not been populated yet (status `refreshing`, empty `{}` payload) must return `202 warming`, not `200`. `200` is reserved for rows that have ever been written by a real refresh.
- **Why:** `ensurePlaceholder()` inserts a row with `payload_json: '{}'` + `status: 'refreshing'` on cache miss so the cron can pick it up. The route returned `200 + toView(cached)`, the FE's `isCached()` checked only `status === 'fresh' | 'refreshing'`, and `payload.tiles.map(...)` threw silently — the LiveOps strip rendered nothing.
- **Signal:** FE component renders empty (no skeletons, no errors) on a freshly-evicted/cold cache. Console has `console.warn('[useX] cache read failed: payload.tiles is undefined')` or similar swallowed throw.
- **Apply:** add a sentinel — empty `payload_hash` is unambiguous since a real upsert hashes the JSON — and branch on it in the route to send `202 warming`. Defensively, the FE should also guard `Array.isArray(payload.<list>)` before mapping; missing list ≠ "renderable cached row".

### Never cache empty/zero-row data-fetch responses
- **Rule:** when caching the result of a remote data query (Cube `/load`, Trino, anomaly probe, etc.), skip the cache write when `rows.length === 0`. Cache only confirmed-good payloads. Failures and empties are almost always transient and must not be glued in for the cache's TTL window.
- **Why:** chat-service `preview_cube_query` hit Cube during a ~60s blip where every cube returned `{rows: []}`. `load-cache-adapter` cached the empty arrays for 10 min (the default TTL), keyed by `(query, gameId, metaHash)`. The agent's chart emission requires `data.min(1)` (Zod schema in `chart-spec.ts`), so the empty payload caused `emit_query_artifact({ chart })` to fail with `code: too_small`. The model dropped `chart`, retried, and persisted a chartless artifact to `response_cache`. From then on, the question replayed verbatim with no chart even though Cube was back to returning real rows seconds later. Same bug existed in `refresh-cached-artifacts.ts:58`.
- **Signal:** a data card / chart that "used to work" stops working after a brief upstream blip, and stays broken for ~exactly the cache TTL even though hitting the upstream directly returns data. Cache row payload is `{"rows":[]}` / `{"data":[]}` / equivalent empty shape. Downstream consumers fail schema validation (`min: 1`) or render an empty state on a query that should be populated.
- **Apply:** guard every `putCachedX(...)` call with `rows.length > 0` (or the kind-specific "is non-empty" predicate). Apply equally to: live preview reads, scheduled refreshers, any meta/probe call whose empty result might be transient. If an empty result is *legitimately* cacheable (rare — e.g. confirmed "no players in cohort"), cache it under a separate kind/key with a much shorter TTL and an explicit `emptyOk: true` flag, not the same row as non-empty results.

### Destructive ops guarded by a mutex must check the guard BEFORE mutating
- **Rule:** any DELETE / PATCH / write operation that protects against concurrent conflicts (via a mutex, in-flight flag, or version check) must check that guard **before** any mutation. Order: (1) check the precondition, (2) if it fails → abort + error response, (3) only if it passes → mutate. Checking after the mutation is a logic bug that reports false success.
- **Why:** `POST /api/care/cases/reset` deleted all cases for a game (via `clearCases()`) and *then* checked `isSweepInFlight`. A concurrent sweep could have started after the code determined the flag was clear but before the delete, causing the fresh sweep to reference cases that had just been wiped. The response would correctly send `409 sweep in-flight`, but the user would see "reset aborted" while the cases were already gone, invisible to the sweep. This is silent data loss masked as a "prevented conflict."
- **Signal:** an operation reports "conflict detected, changes not applied" but you audit the data and find the mutation already happened; a guard check comes after the destructive operation in the code flow; a concurrent operation's data disappears after it receives a "prevented conflict" response.
- **Apply:** restructure the route as:

```typescript
if (isSweepInFlight) {
  return res.status(409).json({ error: 'sweep_in_flight' });  // Abort first
}
// Only then proceed to mutation
await clearCases(gameId, workspace);
```

The 409 response is now truthful: the conflict was detected and the mutation was prevented. Unit-test both paths: happy path verifies the delete, conflict path verifies nothing was deleted. Integration test: spawn a concurrent sweep immediately after receiving 409 and verify cases still exist.

## Test infrastructure

### ESM imports hoist above top-level `process.env =`
- **Rule:** never rely on top-level `process.env.X = value; import …` to override env *for* the import. ES module imports are hoisted; the assignment runs *after* the module has already read its top-level constants.
- **Why:** `dashboard-starter-pack-seeder.test.ts` (and similar) set `process.env.DB_PATH = "<tmp>/test.db"` at the top, then imported `getDb`. The `const DB_PATH = process.env.DB_PATH ?? "data/segments.db"` in `sqlite.ts` had already evaluated — captured the dev DB path. The test's `DELETE FROM dashboards` then wiped the dev DB. We only noticed because daily-health vanished mid-session.
- **Signal:** an env override "isn't taking effect" only when the module was already imported earlier; the dev DB / cache / file mysteriously loses rows after running tests; module re-imports cached at a different path than the test set.
- **Apply:** (1) move env-reads to lazy resolution at first use, not module top level (`getDb()` does this now via `resolveDbPath()`); (2) use `vi.hoisted(() => { process.env.X = …; })` if you must set env before import; (3) add a regression test that asserts the lazy contract.

### reactflow doesn't render in jsdom; no jest-dom matchers in this repo
- **Rule:** never assert on reactflow-rendered node/edge DOM or pixel geometry in a jsdom test. Unit-test the pure layout/edge logic directly; for component/integration tests, STUB the reactflow board and assert on the props passed to it. Also: `<Handle>` (and any reactflow hook) throws outside `<ReactFlowProvider>` — wrap it. And this repo has **no `@testing-library/jest-dom`** — `toBeInTheDocument`/`toBeVisible` don't exist.
- **Why:** the concept-map integration test first asserted on board node DOM; reactflow measures via `ResizeObserver`/`getBoundingClientRect` (both zero in jsdom) so it never lays out nodes, and `BaseNode`'s `<Handle>` threw without a provider. Separately, `expect(...).toBeInTheDocument()` failed with "Invalid Chai property" because jest-dom isn't set up.
- **Signal:** a reactflow component test hangs/empties, or errors "Seems like you have not used zustand provider as an ancestor"; `toBeInTheDocument is not a function` / "Invalid Chai property: toBeInTheDocument".
- **Apply:** (1) extract geometry into a pure fn (`build-layout.ts`) and unit-test that; (2) `vi.mock` the board, render its props as data-attributes, assert those (focus/node-count/edge-count/active-layers); (3) wrap any direct node render in `<ReactFlowProvider>` (+ `<MemoryRouter>` if it has `<Link>`); (4) use `toBeTruthy()` / `expect(queryBy…).toBeNull()` instead of jest-dom matchers.

### A new translator's tests must use the EXISTING counterpart as oracle, not self-written fixtures
- **Rule:** when writing one direction of a paired translation (FE tree→Cube-query mirroring the server's `treeToCubeFilters`; a reverse of `buildPredicateFromRows`), the test oracle is the **counterpart's behavior**, not fixtures you author alongside the code. Self-written fixtures encode the same misunderstanding as the implementation and pass green.
- **Why:** the FE tree→query mapper forwarded tree ops `in`/`notIn` verbatim — invalid Cube operators (the server maps `in→equals`); and it flattened `OR(AND(a,b),c)` to `or(a,b,c)`, silently widening cohorts. Both shipped with green tests asserting exactly the buggy output. Code review caught them only by diffing against the server translator line-by-line.
- **Signal:** a translator/serializer test suite where every expected value was written by the same author in the same session; no test feeds output of direction A into direction B; operator/shape sets duplicated in two files with no cross-check.
- **Apply:** (1) round-trip property tests (`A(B(x)) ≅ x`) over hand-picked expected outputs; (2) one test that imports the counterpart's operator map (or a shared constant) and asserts set equality — drift then fails loud; (3) for op sets that must be a subset/superset (translatability gates), assert that relation explicitly against the counterpart's exports.

## Frontend data extraction

### Heuristic column pickers must filter by type, not position
- **Rule:** when extracting "the value" from a Cube result row, never pick by `Object.keys()[0]` — Cube always emits the granularity-suffixed time dim first. Filter by what the value should be (numeric for measures, ISO string for dates), not by ordinal.
- **Why:** dashboard `extractKpiValue` accepted "any non-empty string" as the value column. For a `{ "active_daily.log_date.day": "2026-05-11…", "active_daily.dau": "24230", "active_daily.log_date": "2026-05-11…" }` row it returned the date string, rendering `DAU: 2026-05-11T00:00:00.000` instead of `24,230`.
- **Signal:** KPI / single-value tiles show ISO date strings, granularity-bucket keys, or any string that the user clearly didn't intend.
- **Apply:** prefer `typeof v === 'number'` then numeric-parseable strings (`!Number.isNaN(Number(v))`); use the *latest* row, not the first; lock the contract with a unit test that includes time-dim columns in the fixture.

### Cube `format: percent` values are fractions — scale per-series, don't compact small ones to "0%"
- **Rule:** a percent-typed measure pulled from Cube arrives as a **fraction** (`0.0069` = 0.69%), not pre-scaled. Before display, decide the scale from the data (all |v| ≤ 1 → ×100; any > 1 → already 0–100) and format with **adaptive precision**; never run a small fraction through compact/1-decimal formatting.
- **Why:** the scatter y-axis for `paying_rate` (0.0019–0.0069) rendered every tick as `0%`: the formatter appended `%` without ×100 and used `compact, maxFractionDigits 1`, so `0.0069 → "0"`. Axis read `0% 0% 0%` and looked un-zoomed even though the points were distinct.
- **Signal:** a percent axis/column shows repeated `0%` (or values 100× too small) while the underlying numbers are clearly nonzero and distinct; chart looks like it isn't scaling to the data.
- **Apply:** `detectPercentScale(values)` once per chart/column → pass the factor into `formatAxisValue`/`formatReadableValue`; precision grows as magnitude shrinks (42% → 1dp, 0.69% → 2dp). Keep axis, tooltip, and data-table on the same factor. For scatter, also set `domain={['auto','auto']}` so a tight cluster isn't crushed against a 0-anchored axis.

### Re-rendering a persisted Cube load response must normalize to the wrapper shape
- **Rule:** before building a `ResultSet` from a stored/forwarded Cube `/load` response, normalize it to the wrapper shape `{ queryType, results: [{ data, annotation, query }], pivotQuery }`. The playground `ChartRenderer` reads `resultSet.loadResponse.results[0].data`/`.annotation` **directly** (not via the ResultSet API), so a `ResultSet` built from the legacy single-result shape (`{ data, annotation, query }` at top level) constructs fine but throws at render — `loadResponse.results` is `undefined`.
- **Why:** pinned dashboard tiles persisted the raw `cubeClient.load()` response to drive engine parity with the QueryBuilder. The backend returns the **legacy single-result** shape (top-level `data`/`annotation`, no `results[]`), so the tile's `isRenderableLoadResponse` precheck (which only matched `results[0].data`) rejected every response and silently dropped to the rows-only legacy renderer — the exact mismatch the parity feature was meant to remove. `new ResultSet(legacy)` would not have helped: the *renderer* needs the `.results` wrapper.
- **Signal:** a chart that should mirror the QueryBuilder renders through a fallback/heuristic path instead; a stored load response has top-level `data`/`annotation` keys but no `results` array and no `queryType`; an engine path "works" only against one Cube backend's response shape.
- **Apply:** one `normalizeLoadResponse(resp)` that (a) passes a `results[0].data` wrapper through unchanged, (b) lifts a legacy `{ data, annotation, query }` into `{ queryType: 'regularQuery', results: [resp], pivotQuery: {...resp.query} }`, (c) returns null otherwise → fallback. Verify against a **real cached row** that `chartPivot`/`seriesNames`/`loadResponse.results[0].data[0]` all resolve — synthetic fixtures may omit the time dimension and hide series-count differences. Keep the error boundary as the safety net, not the primary guard.

### Cross-source series (ratios) must align by date key, not array index
- **Rule:** when combining two timeseries that come from **different cubes/queries** (e.g. a ratio `numerator/denominator`), pair them on the matching **date**, not by zipping positions. Carry the date alongside the value; never reduce to a bare `number[]` and `num[i]/den[i]`.
- **Why:** `anomaly-detector.divideSeries` sorted each series by day, dropped the dates, then divided by index. It only worked while both cubes returned the same contiguous date set. The business-metrics `paying_rate` (`recharge.paying_users / active_daily.dau`) draws from two cubes with **different freshness** — `recharge` had data to 05-25, `active_daily.dau` only to 05-18. An interior gap or different start in either series silently pairs the wrong days (`paying_users[05-20] / dau[05-19]`) and corrupts every later point; index-truncation also evaluated the *oldest* overlapping days and mislabeled the latest. The sibling LiveOps handler did it right with a date-keyed map.
- **Signal:** a ratio/derived metric looks plausible in steady state but goes wrong after a pipeline lag, a missing day, or when the two sources' date ranges diverge; the "latest day" in an alert isn't actually the newest available date.
- **Apply:** keep `{ date, value }[]` (`rowsToDatedSeries`), build a `Map<date, value>` for the denominator, and divide only on matching dates (`divideByDate`), skipping unmatched/zero days. Cover an interior-gap fixture in tests — the happy path hides the bug because contiguous same-start series align by index too.

### SQLite `datetime('now')` is naive UTC — serialize with a `Z` before the browser parses it
- **Rule:** any timestamp column defaulted from `datetime('now')`/`CURRENT_TIMESTAMP` must leave the API as ISO-8601 UTC with an explicit `Z` (`strftime('%Y-%m-%dT%H:%M:%SZ', ts) AS ts`). Never ship the raw `YYYY-MM-DD HH:MM:SS` string to the FE.
- **Why:** SQLite stores `datetime('now')` as a space-separated UTC string with no timezone marker. The browser's `new Date('2026-06-05 03:20:00')` parses that space-separated form as **local** time, so a row written *now* renders as "&lt;utc-offset&gt; hours ago" — in GMT+7 the segment refresh history showed a just-created refresh as "7 hours ago". The data was real, not mocked; only the parse was wrong, and every consumer (`formatDistanceToNowStrict`, sparkline `new Date(ts).getTime()`) was off by the same offset.
- **Signal:** a freshly written timestamp shows up exactly the UTC-offset number of hours in the past (7h in GMT+7); relative-time labels look mocked/seeded; the bug is invisible to a UTC-machine developer and only appears for offset users.
- **Apply:** fix at the single API boundary (the SELECT), not per `new Date()` call site — one `strftime(... 'Z')` corrects history table + library sparklines at once. Lock it with a route test asserting `ts` matches `/T\d\d:\d\d:\d\dZ$/` and that the parsed instant is <1 min old. Same class applies to `created_at`/`updated_at`/`last_refreshed_at` if ever rendered as relative time.

### Never synthesize Cube member names — and don't trust the SDK's meta shape
- **Rule:** any member name your code *constructs* (`` `${cube}.count` ``) instead of reading from /meta is a latent boot error — cube models name their measures differently (`rows`, `events`, `transactions`). And when you DO read meta, know your source: the cubejs SDK's `meta()` object strips fields the raw `/v1/meta` response carries (`joins`, and reliably only the raw fetch keeps `connectedComponent`). Features keyed on those fields silently degrade while fixture tests stay green.
- **Why:** the segment "Open in Playground" deeplink synthesized `<cube>.count` as its boot measure — no jus cube exposes `count`, so every boot would have thrown a Cube UserError; only a live probe caught it (unit fixtures used the synthesized name). Same day, the predicate member-catalog grouped "joined cubes" from SDK `meta()` joins — which the SDK strips — so joined-cube members never appeared at runtime while the fixture (which included joins) passed.
- **Signal:** a member name built by string template; a meta-driven feature that works in tests but shows nothing/errors live; a fixture whose meta shape includes fields the real response lacks (or vice versa).
- **Apply:** boot queries that don't need a measure should send `measures: []` (a dimensions+filters query is valid). For reachability/joins, fetch raw `/cube-api/v1/meta?extended=true` (pattern: `use-catalog-meta.ts`) and key on `connectedComponent`. Make fixtures mimic the REAL response shape — copy a live /meta excerpt, don't hand-write one.

### `Unexpected token '<', "<!doctype"` on a JSON fetch = the request hit the SPA fallback, not the API
- **Rule:** when an API base URL is derived from `window.location`, strip **all** trailing slashes (`/\/+$/`), not one (`/\/$/`). A leftover slash concatenated with a prefixed base produces a `//prefix/...` path; the Vite dev proxy only forwards paths that *begin* with the exact prefix (`/cube-api`), so `//cube-api/...` falls through to the SPA `index.html` (HTTP 200), and the caller's `resp.json()` then chokes on `<!doctype`.
- **Why:** `buildApiUrl` (`use-cube-api-bootstrap.ts`, inherited from the original cubejs-playground scaffold) normalized the origin with `.replace(/\/$/, '')`. Landing on `http://localhost:3000//#/catalog/data-model` (double slash before the hash) left `http://localhost:3000/`, so `apiUrl` became `http://localhost:3000//cube-api/v1` and the meta fetch path was `//cube-api/v1/meta`. The proxy skipped it, the SPA answered with HTML 200 (`resp.ok` true), and `useCatalogMeta` surfaced *"Failed to load meta: Unexpected token '<', '<!doctype '… is not valid JSON"*. The clean single-slash URL worked, so it looked route-specific when it was URL-shape-specific. Affects every `apiUrl` consumer (Catalog, Explore, Segments, QueryBuilder) — one normalization point, one fix.
- **Signal:** a `fetch(...).json()` throws `Unexpected token '<', "<!doctype "...`; the response is 200 but the body is `index.html`; the URL bar (or a redirect/link) shows a doubled slash; works on a clean URL, breaks on a malformed one.
- **Apply:** normalize with `/\/+$/`; verify the resulting path against the proxy rule (`new URL(url).pathname` must start with the proxied prefix, no leading `//`). Quick triage: `curl` both the single- and double-slash proxy paths — the doubled one returns `<!doctype html>` while the correct one returns JSON. Lock it with a unit test that feeds a double-slash origin through the URL builder.

## Metric resolution

### A resolver's output ref must match downstream validator vocabulary
- **Rule:** when normalizing user input to a canonical reference (e.g. catalog term → cube member), do it at the *read boundary*, not on demand. The resolved ref must match the vocabulary that the consuming validator accepts. Here: glossary loads as `formula.ref` (a cube member name), the /meta gate validates members by their **cube-only** names.
- **Why:** the glossary resolver emitted catalog paths (`business_metrics/revenue`) while the `/meta` gating layer only knew cube members (`revenue`). Every catalog-backed metric query hit the /meta gate's unknown-member path, forcing a mandatory clarification prompt. A live catalog term with one unambiguous cube mapping would still clarify, because the ref mismatch turned the resolver into a silent "always clarify" trap.
- **Signal:** a metric with an unambiguous business term (e.g. `revenue`, `paying_users`) returns a clarification prompt listing sibling metrics; the glossary-matched term appears in the prompt's prose but the answer route never auto-fires. Works fine for native cube members but fails for every catalog-backed term. Cache replay shows stale clarifications persisting even after the live metric is fixed.
- **Apply:** (1) assert the resolver's output ref against the validator's live vocabulary in unit tests, not just in the happy path. (2) When fixing a mismatch, bypass caches (`Bypass-cache` pill, cold cache restart) because the response cache replays entire (prompt + rationale + sibling list) objects, not just refs — stale clarifications stick until the cache TTL expires or is cleared.

## Chat session lifecycle

### Per-mount latches on URL-replace effects break back-to-back chat cycles
- **Rule:** URL-sync effects driven by streaming state must re-fire each chat cycle. A `useRef(false)` latch that flips to true on first replace and never resets will skip every subsequent cycle in the same mount.
- **Why:** the `/chat → /chat/:id` history.replace effect was gated by a per-mount `replacedRef`. After the first new chat in a mount, it locked at `true`; the second new chat created a fresh session server-side but the URL stayed at `/chat`. The `useChatStream` post-done guard (added to stop the prior chat's terminal-state entry leaking into the next submit) then stripped the entry from the null-pinned page after `done`, so the assistant reply was never committed to `committedMessages`. Users reported "new chat got merged into the previous one" — the merge was actually a vanished reply.
- **Signal:** "New chat" works the first time but not the second; URL fails to update from `/chat` to `/chat/:id`; assistant reply visible while streaming then disappears on `done`; committed history loses the just-streamed turn.
- **Apply:** prefer derived guards (`streamSessionId === id`) over once-per-mount latches. If you genuinely need a latch, reset it on the lifecycle event that opens a new cycle (e.g. `id === undefined`). Cover the second cycle in tests — most "new chat" regressions only show on the second run because the first one masks them.

### Never-pruned session aliases leak the latest new chat's id onto older sessions
- **Rule:** when a module-global store resolves a key through an alias map, the alias must either be pruned when its target is repurposed, or the consumer must verify the resolved entry actually matches the requested key. An accumulate-only `realId → sharedKey` map silently maps stale ids to whatever the shared slot now holds.
- **Why:** the chat stream store keys every new chat at the single `__new__` slot and registers `realSessionId → '__new__'` on `session_created`, never pruning. After N new chats, all N real ids alias to `__new__`, whose `sessionId` field only reflects the *latest* chat. Opening an older recent (`/chat/<oldId>`) resolved through the alias to the `__new__` entry, so `useChatStream` returned the latest id; the `streamSessionId === id` URL-replace guard then bounced every recent to the newest session. Same leak also routed `sendTurn` (via `liveSessionIdRef`) into the wrong thread. The sidebar links were correct the whole time — the redirect happened after navigation.
- **Signal:** clicking different recent conversations all land on one id (the most recently created); only chats created *this* browser session misbehave (older/seed ones, having no alias, route fine); a message sent on session A persists into session B.
- **Apply:** in the store selector, honor an alias only when the resolved entry's own id equals the requested id — `if (sessionId !== null && found && found.sessionId !== sessionId) found = streams.get(sessionId) ?? null;` (`use-chat-stream.ts`). This keeps the just-created session's live continuity (ids match) while rejecting stale aliases (ids differ). Fixing only the URL-replace effect masks the bounce but leaves the send-routing corruption — fix at the resolver. Regression-test the *second* session opened after two new chats; the first masks it.

### A new chat-service route is dead until the main server proxies it
- **Rule:** every `/api/chat/*` route the FE calls is served by the main server (`server/src/routes/chat.ts`), which forwards to chat-service via *explicit per-route handlers* — not a catch-all. Adding a route in chat-service is only half the job; add the matching proxy handler (same method + path, forward `X-Owner-Id`, build the upstream URL) or the FE never reaches it.
- **Why:** Phase-03 added `GET/DELETE /api/chat/sessions/:id/focus` to chat-service, but no proxy rule was added on the main server. The FE hits `/api` → vite proxy → main server :3004 → 404 (route unknown). The focus client swallows non-OK as `null` by design, so the header chip silently hid — looked like the feature was unwired or the flag was off, when both backend and flag were fine. Cost an hour chasing the wrong layer.
- **Signal:** a new chat feature does nothing in the UI but `curl` straight to chat-service :3005 returns correct data; the same path through :3004 returns `{"error":"Not Found"}`. Any FE client that defensively returns `null`/`[]` on error will mask this as "empty," not "broken."
- **Apply:** when adding a chat-service route, grep `server/src/routes/chat.ts` and add the twin proxy handler in the same change. Test through :3004 (or the vite proxy), not just :3005 — a green direct hit proves nothing about what the browser sees. Watch the upstream path prefix: most session routes proxy to `/sessions/:id` (no prefix) but focus registers with the full `/api/chat/...` path, so the forward URL must match what chat-service actually registered. Bodyless POSTs (e.g. turn-cancel) must NOT forward `Content-Type: application/json` with no body — the upstream's body parser returns `FST_ERR_CTP_EMPTY_JSON_BODY` 400; only set the content-type header when a body is actually sent. This bit Phase 04's cancel button the same way Phase 03's focus chip got bitten.

### Recorder-only fields in a normalized event must be stripped at the wire edge
- **Rule:** when instrumentation threads optional fields through a normalized event stream to a recording layer (e.g. tool `input`, post-redaction `resultText`), strip those fields before the event reaches the live client wire contract. Leaving optional fields in place converts them from "internal artifact" to "part of the schema," and clients build logic around them. An optional field in the schema is a permanent promise.
- **Why:** agent-run audit recorder added `input` and `resultText` to `tool_call` and `tool_result` events (for persistence only). These fields were marked optional and live clients shouldn't reference them — but leaving them in the SSE stream (the official wire contract) made them visible. Future clients would see them and build replay features around them; future refactors would have to preserve them forever or risk breaking deployed clients. The rule: if a field is *only for the recorder*, it must not exist on the live stream.
- **Signal:** a field appears in a SSE event definition and is documented as "recorder-only" or "internal," indicating confusion about who owns the wire contract. An optional field added for one system (recorder, cache, persistence) that leaks onto a client-facing channel.
- **Apply:** strip recorder-only fields in the route that emits the SSE stream, not in the recorder itself or normalizer. For advisor-run audit, `advisor.ts` route receives events from `agent-runtime.ts` with `input`/`resultText` attached, then deletes them before `res.write(JSON.stringify(event))` — the live client never sees them. This keeps the normalizer generic (no special casing) and the persistence layer complete (no data loss). Verify in an integration test: stream an event SSE client-side and log `JSON.stringify(event)` — should not include recorder-only keys.

## UI / design

### Don't introduce experimental visual directions on existing surfaces
- **Rule:** new pages, redesigned pages, and feature additions must read in the existing design language (`var(--font-sans)`, the established header pattern, semantic tokens). Editorial / display / experimental directions need explicit buy-in *and* their own contained route — they don't ride along with a feature ship.
- **Why:** the Liveops Phase-1 polish shipped a serif/editorial direction (Georgia, 34px display H1, deck-style eyebrow) into an Inter sans-serif app. Visually a different product on `/liveops` vs every other route, and the user flagged it as "spacing odd, font not matching" the first time they opened it.
- **Signal:** new surface uses fonts, padding, header shape, or radii that don't match the closest existing well-formed page. Hard-coded hex colors instead of `--text-*` / `--border-*` / `--*-soft / -ink` tokens.
- **Apply:** before shipping any UI work read `docs/design-guidelines.md` and copy the page header recipe verbatim. If a direction genuinely warrants forking, isolate the fork to one route and ship it behind a "this is intentional" comment + design-review handshake — don't introduce drift through the back door of a feature PR.

### Scrollable flex column: children shrink to fit before overflow-y ever engages
- **Rule:** a `display:flex; flex-direction:column; overflow-y:auto` container must also set `& > * { flex-shrink: 0; }` (or equivalent) on its children. Otherwise flexbox compresses the children to fit the container height and `scrollHeight` never exceeds `clientHeight` — no scrollbar, content just clipped. Children with their own `overflow:hidden` are the worst case: that resets their automatic minimum size from min-content to 0, so one such child silently absorbs the entire height deficit.
- **Why:** the glossary edit-term modal had the canonical dialog scroll recipe (dialog `max-height:92vh; overflow:hidden`, form `flex:1; min-height:0; overflow-y:auto`) across two prior fixes — and still clipped. The collapsible Concept-tier `<Section>` (`overflow:hidden` for its rounded corner) shrank to absorb ~400px of overflow, clipping its fields with zero scrollbar. DevTools-style evidence: `form.scrollHeight === clientHeight` (527) while the last field's `getBoundingClientRect().bottom` (1010) sat far below the form's box bottom (613).
- **Signal:** modal/panel content visibly cut off, wheel does nothing, and in console `el.scrollHeight === el.clientHeight` even though a descendant's rect bottom exceeds the container's rect bottom. Two scroll "fixes" on the same surface that adjust the container but never the children is the historical tell.
- **Apply:** add `& > * { flex-shrink: 0; }` inside the scrollable flex column's CSS (styled-components nesting works). Verify with a headless-browser measurement (scrollHeight > clientHeight, last field's rect inside the form rect at max scroll), not by eyeballing — jsdom can't test layout.

## Agent query construction

### Every `/meta` fetch must send `x-cube-game`, or dev-mode silently serves the default game
- **Rule:** any surface that fetches Cube `/meta` to build a model list (Query Builder, Catalog, New Metric wizard) must send the `x-cube-game: <activeGameId>` header — the same header the `/load` / `/sql` paths already send. The header is what scopes the schema; a `game_id` query param does NOT (Cube reads `securityContext.game` from the minted JWT, not the query string).
- **Why:** `cube-dev` runs `CUBEJS_DEV_MODE=true` locally. In dev mode Cube's `checkAuth` is permissive: a JWT with no `game` claim resolves to an empty securityContext, and `gameFor()` then falls back to `CUBEJS_DEFAULT_GAME || 'ballistar'`. The Query Builder's extended-meta fetch sent only `x-cube-workspace`, so the proxy minted a game-less JWT → every game showed ballistar's cubes (6 + the user_360 view) instead of its own (PTG has only 2). `/load` was correctly scoped (it sent the header), so data was right but the model list was wrong — a confusing split. Catalog had the same latent bug masked by a no-op `game_id` query param.
- **Signal:** switching games leaves the cube/dimension list unchanged; a game with a known-smaller model (PTG, muaw) shows the richest game's cubes; meta looks wrong but actual query results are correct (or vice-versa). The minted JWT carries `{ userId }` with no `game`.
- **Apply:** add `if (gameId) headers['x-cube-game'] = gameId` to the fetch (read `localStorage['gds-cube:active-game']` if React context isn't threaded through). The `playground` service user is authorized for all games, so the claim alone scopes correctly even in dev mode. Don't rely on a `game_id` query param — it's silently ignored. Defense-in-depth: turning `CUBEJS_DEV_MODE` off makes a missing claim throw instead of defaulting, converting this silent class of bug into a loud 500 — but it disables the Cube `:4000` playground UI, so it's a deliberate trade, not a free win.

### Strip views from `/meta` before the agent (or any cube-only surface) sees it
- **Rule:** the schema the chat agent builds queries from must be filtered to cubes only — drop every `type: 'view'` entry. Do it once at the `/meta` fetch boundary so every consumer (tool schema exposure, member-name validation, capability detection) is cube-only by construction.
- **Why:** Cube's `/meta` returns cubes *and* views in the same `cubes[]` array (views tagged `type: 'view'`). `cube-meta-cache.getMeta` cached the raw payload, so `get_cube_meta` handed the agent view members like `revenue_metrics.*`. The agent emitted view-based artifacts; a view is a self-contained namespace that can't join to anything, so opening that artifact in the query builder left every cube greyed out — the user couldn't add a cross-cube dimension (platform, install cohort). The query was correct but a dead end.
- **Signal:** an agent-generated query / "open in builder" link uses a view member (a name that maps to a `views/*.yml` entry, not a `cubes/*.yml` cube); in the builder the whole cube list is disabled and the query can't be extended. Or: you intend "cubes only" but the agent still references a known view name.
- **Apply:** filter `meta.cubes` to `c.type !== 'view'` at the single fetch/caching chokepoint (`cube-meta-cache.ts#getMeta`), keeping untyped entries (older cubes lack `type`). Strip at the source, not per-tool — per-tool filtering leaks the day someone adds a new meta consumer. Note the runtime tail: already-cached view artifacts in `response_cache` replay verbatim, and the meta cache has a TTL — the fix only governs *new* fetches/answers, so clear stale caches or wait out the TTL when verifying.

### "A vs B" questions need a chart shape that shows both metrics
- **Rule:** when a question correlates two metrics across entities ("ARPU vs paying-rate per country"), the chart must plot **both** metrics — scatter (x = metric A, y = metric B, one point per entity). Charting a single metric as a bar silently drops the comparison the user asked for. The chart-type guidance the agent reads must name this shape explicitly; a generic "2 numeric metrics → scatter" row gets out-competed by the dominant "1 categorical + 1 metric → bar" pattern.
- **Why:** for "ARPU vs paying-rate per country" the agent fetched `arpu_vnd`, `paying_users`, `user_count` per country but emitted a `horizontal-bar` of `arpu_vnd` alone — the paying-rate axis vanished. The chart-section menu then offered no scatter toggle (scatter is data-shape-isolated: it needs two numeric axes, which a category×value bar can't supply), so the user had no recovery path. Root cause was guidance, not code: the explore skill's shape table didn't map "1 entity dim + 2 metrics".
- **Signal:** a "X vs Y" / "compare A and B" / correlation question renders a single-metric bar or line; the second metric appears only in the prose summary; the chart menu has no way to reach scatter.
- **Apply:** (1) in the chart-shape guidance (`explore/SKILL.md`, `emit-chart.ts` description) add an explicit row for "1 entity dim + 2 metrics → scatter" with the encoding recipe (`category` = x metric, `value` = y metric, keep the entity label column per row); (2) flag the "A vs B per <entity>" phrasing as a correlation, not a single-metric bar; (3) for derived ratios with no native measure, compute per-row and `emit_chart`; (4) make the scatter renderer label each point with the leftover (non-axis) column and detect x/y units independently — two metrics often differ (VND vs %). Don't expect the menu to "convert" a bar into scatter; the fix is emitting scatter up front.

## Auth / Authz

### Authenticate ≠ authorize — a brokered IdP needs an app-side default-deny
- **Rule:** when login is brokered through a corporate IdP (Keycloak → Microsoft/Entra) with JIT user creation, authentication proves identity for the *whole tenant*. Authorization must live in a writable app store keyed by a stable identifier (lowercased email), default-deny: no active grant ⇒ no access, even with a valid token.
- **Why:** prod login moved to KC→Microsoft. The old role/`allowedGames` claims were derived from KC realm-roles/groups, which are **empty** for brokered JIT users — so every real user would have authenticated successfully and resolved to an empty/viewer grant silently. The fix moved role + workspace/game/feature grants into `user_access` (+ `user_*_access`, `feature_flags`) and made the login callback mint a privileged JWT only for `status='active'` rows; unknown/pending/disabled → `403 ACCESS_PENDING` + an auto-created pending row for the admin queue.
- **Signal:** users can log in but see nothing / everything depending on a claim that's empty; role/group claims you relied on are absent under brokering; "it worked in the local realm" but not in prod.
- **Apply:** resolve role + grants from the DB per request (`getAccess(email)` in `authenticate.ts`), not from the client-held JWT — this also gives mid-session revocation within the cache TTL. Keep an env-gated bootstrap-admin seed (`AUTH_BOOTSTRAP_ADMINS`) so the first deploy isn't locked out. Never trust role/games from the signed token; the JWT carries identity only.

### UI reads grants + features but never mirrors the write-side — gate surfaces to prevent no-op toggles
- **Rule:** when a write-side endpoint (admin grant mutations) lives in a separate UI layer from the read-side (nav filters, visibility toggles), the write-side **must** be gated at the request boundary AND the read-side **must** house the enforcement, or toggles look like they do nothing. The login callback must return the full resolved grant set (`workspaces`, `features`, `allowedGames`) so the FE can gate surfaces on first paint without waiting for a `/me` refresh.
- **Why:** admin Grant UI in Settings wrote grants to the DB, but the FE nav/settings/routes never read them — every checkbox toggle looked like a no-op because the surfaces stayed visible. The workspace `/api/workspaces` endpoint was role-filtered (read-side) but workspace grants in the DB were unused (write-side had no enforcement). Features were granted in the DB but the FE had no `useHasFeature` gate — every feature remained enabled. Root cause: the consuming surfaces (Data nav button, Settings tabs, feature-scoped routes like `/chat`, `/catalog`) didn't check the grant before rendering.
- **Signal:** an admin grant UI writes to the DB (200 OK) but the user's view doesn't change; toggles appear to be cosmetic. A grant/feature exists in `user_access` + shows in `/api/auth/me` after refresh but nav items still render; a disabled feature URL still loads instead of redirecting.
- **Apply:** (1) server-side: filter `/api/workspaces` by `userCanAccessWorkspace(user, w)` (grants-first, role fallback); pass `workspaces` + `features` in the login callback so the FE has the full set on first paint. (2) FE-side: import the resolved user's `features` map from the login callback → Context; gate surfaces with `useHasFeature()` hook (sidebar sections, bottom-nav items, settings tabs); mount `<FeatureRouteGuard/>` to redirect disabled-feature URLs. (3) Test the full path: write a grant, refresh the browser (or toggle the FE's features map in devtools), verify the surface hides. Don't test the write in isolation — confirm the gate fires.

### Scope game access server-side — an FE-only filter is not enforcement
- **Rule:** per-user game (tenant) access must be checked on the server for every request that names a game, and the minted Cube token must carry the *real* user's key so the downstream Cube also enforces it. An FE switcher that hides games is cosmetic — a crafted request bypasses it.
- **Why:** the minted path signed Cube JWTs with a blanket `userId:'playground'` (allowlisted for all games), so per-user game limits existed only in the FE `use-game-context` filter. Anyone could hit `/load` with `x-cube-game=<other>` directly. Fix: `workspace-header.ts` 403s `GAME_FORBIDDEN` when the requested game isn't in the user's grants, and mints the Cube token with `userId=<email>` so cube-dev's `checkAuth` (`auth-db.js` → internal access API) re-enforces it.
- **Signal:** access control that only appears in client code (a dropdown filter, a hidden nav item); a server endpoint that trusts a client-sent game/workspace id without re-checking grants; tokens minted under a shared service principal for per-user data.
- **Apply:** gate at the request boundary, fail closed (deny on missing/ambiguous grant), and align the token `userId` contract across services so the second hop (Cube) enforces independently. Test through the proxy (`:3004`), not just Cube (`:3005`). A migration `AUTHZ_GRANT_FALLBACK` flag may relax empty-grant users to role-based defaults during rollout; flip it OFF once grants are seeded so the gate fails closed.
- **Trap — the upstream game gate keys off the `x-cube-game` *header*, not query/body.** `workspace-header.ts` calls `userCanAccessGame` only when `readGameId` finds an `x-cube-game` header. A new route that takes `game` from a query param or request body (e.g. Drift Center's `GET /drift-center?game=`, `PATCH /:id/repoint {game}`) gets NO upstream game check — `apiFetch` doesn't send `x-cube-game`, so `buildCubeCtxForGame(body.game)` mints a token / fetches `/meta` for an arbitrary game ungated. **Signal:** a handler reads `game` from `req.query`/`req.body` then calls `buildCubeCtxForGame(game)`; the only authz nearby is the write-role gate. **Apply:** re-check `userCanAccessGame(req.user, game)` (403 `GAME_FORBIDDEN`) inside any such route, *before* doing work — don't assume the global preHandler covered it. Skip when `req.user` is absent (AUTH_DISABLED dev), matching the upstream pattern.

### A module-level cache behind a localStorage mirror survives `localStorage.clear()` in tests
- **Rule:** when a store keeps an in-memory cache in front of localStorage (the pref store: server = source of truth, localStorage = synchronous write-through mirror, `getPref` reads cache→mirror), a test that seeds `localStorage` directly can be *shadowed* by a value an earlier test in the same file wrote through `setPref` — the cache wins and `localStorage.clear()` doesn't touch it. Reset the cache between tests, not just localStorage.
- **Why:** migrating prefs/view-state/aliases/drafts off raw localStorage onto `src/hooks/server-prefs-store.ts` made `useLocalStorage` and many hooks read the cache first. Vitest `pool:'forks'` isolates modules per *file*, so the cache persisted across `it`s within a file; suites that did `localStorage.setItem(KEY, …)` then asserted got the prior test's cached value (chat-service settings read `claude-opus-4-6` instead of freshly-seeded `claude-sonnet-4-6`).
- **Signal:** a test passes in isolation but fails under in-file ordering; an assertion reads a value some *other* test set; you cleared localStorage in `beforeEach` but stale data still appears.
- **Apply:** reset in global setup, not per-file. `src/test-setup.ts` (wired via `vitest.config.ts setupFiles`) calls `__resetPrefsCacheForTests()` in a global `beforeEach`, clearing only the cache (not localStorage) so suites relying on cross-test mirror persistence aren't disturbed. Ship a cache-reset hook in the same change as any read-through cache.

### Migrating off localStorage: keep the mirror, move the source of truth
- **Rule:** to make client state device-portable without rewriting every synchronous reader, keep localStorage as a *write-through mirror* under the same keys and put the DB behind it. `setPref` writes mirror (sync) + server (async, best-effort); `getPref` reads cache→mirror so header injection and first paint stay synchronous; boot `hydrateServerPrefs()` lets the server win and one-time-imports legacy local values. The session JWT never moves — it authenticates the very request that loads the prefs.
- **Why:** workspace switching needed aliases/workspace/game/drafts/view-state isolated per owner+workspace and multi-user-safe, but readers like `api-client` header attach and `cube-api-factory` read the workspace id synchronously. Routing them through a server fetch would have meant async refactors everywhere; the mirror kept them untouched while the DB became authoritative.
- **Signal:** "move everything to a DB" requests where some values are read synchronously (request headers, pre-React init like theme/lang), genuinely device-local (anonymous telemetry id, reload-loop guard), or are credentials.
- **Apply:** classify keys into data/view-state (→ DB-backed mirror), session/identity (→ stay local), cross-tab event-bus signals (→ stay local). Back the legacy `useLocalStorage` core itself so every current/future consumer migrates at once. Leave device-local infra keys (`playground_anonymous`, `lastLocationReload`) alone and say so.

### SSRF host allow/deny checks must normalize IPv6 before stripping the port
- **Rule:** when guarding a user-supplied host (SSRF), normalize the literal *before* pattern-matching. A naive trailing-`:port` strip (`/:\d+$/`) corrupts bare IPv6 (`::1` → `::`), so the blocklist misses it and a loopback target slips through. Strip the port only for bracketed IPv6 (`[::1]:p`) or hosts with ≤1 colon (IPv4/hostname); leave bare multi-colon IPv6 intact.
- **Why:** the connector SSRF guard (`connector-host-guard.ts`) blocked `127.*`, `169.254.*`, `localhost`, metadata, and `::1` — but `::1` passed because `.replace(/:\d+$/,'')` rewrote it to `::`, which wasn't in the blocklist. Caught by the host-guard test matrix before ship.
- **Signal:** an SSRF/allowlist test passes for IPv4 loopback but the IPv6 loopback case returns "allowed"; any host normalization that runs a `:port` regex unconditionally on values that may be IPv6.
- **Apply:** branch on bracket form / colon count before the port strip; unit-test every loopback + metadata literal in both IPv4 and IPv6 form. Allow RFC1918 / `*.internal` (legit internal warehouses) — block only loopback + link-local/metadata.

### Prefix-workspace `/meta` is not game-scoped — scope it before the chat agent sees it

- **Rule:** On `gameModel:'prefix'` workspaces (prod cube-dev), Cube `/meta` returns EVERY game's cubes regardless of the `x-cube-game` header. Filter `meta.cubes` to `${gamePrefixMap[game]}_` at the gateway (`cube-proxy.ts`) before any consumer reads it. `game_id` workspaces (local) need no filter — one cube per concept, scoped by a `gameId` dimension at query time.
- **Why:** chat produced **0 artifacts on prod vs 64/100 on local**. Unfiltered prod meta exposed `revenue_vnd` in `ballistar_recharge`, `cfm_recharge`, `jus_recharge`, … so `disambiguate_query` offered several identically-labeled "Revenue Vnd" options across games and never resolved. Local's single `recharge.revenue_vnd` was unambiguous. The readiness service already prefix-filtered for cube *counts* (`workspace-readiness.ts`) — the meta proxy just never applied the same rule.
- **Signal:** a chat/agent feature works on one workspace but loops on disambiguation (or "can't resolve measure") on another; the same measure name appears in N cubes in `/meta`; behavior differs local vs prod with identical code.

## Local infra (colima / CubeStore)

### "CubeStore dormant" is usually colima clock-skew, not a missing worker

- **Rule:** Before concluding pre-aggs aren't built because "no worker," check CubeStore for clock-skew corruption. On a colima VM, a host laptop sleep jumps the guest clock backward; CubeStore's `SystemTime::duration_since` then panics and the metastore goes inconsistent — partitions build then vanish on readback, so `usedPreAggregations` stays empty and plain rollups 404 forever. The build-capable workers (`cube_refresh_worker_dev`→`cubestore_dev`, stack `cube_refresh_worker`→`cubestore`) already exist and run; the store is the failure point, not the worker.
- **Why:** local served 0% from CubeStore for 24h+ despite two healthy workers. Root cause was `cubestore` logging `Internal: second time provided was later than self`, `Unique constraint violation: ...batchNNN... has a key that already exists in Name index`, and `Table with id N not found in cache, completely resetting cache`; the worker logged 27 "successfully created" / 27 "Pre-aggregation table is not found ... after it was successfully created" in one window. Wiping volumes + `colima restart` (clock resync) cleared it.
- **Signal:** `usedPreAggregations` empty on every query; plain `rollup` 404s "instance not set up to build pre-aggregations"; `docker logs <cubestore> | grep "second time provided was later than self"` returns hits; `docker exec <cubestore> date` differs from host `date`. A `rollup_lambda` hides this (serves live tail silently) — always assert on a *plain* rollup or on `usedPreAggregations`.
- **Apply:** `colima restart` (resyncs clock) → stop+rm the cube containers → `docker volume rm cube-playground_cubestore_dev_data cube-playground_cubestore_data` → recreate via `STACK_DEV_CUBE=1 node scripts/stack-local.mjs up -d ...` (wrapper sets the arm64 `CUBESTORE_TAG`; raw `docker compose` pulls the wrong-arch cubestore and wedges silently). RECURS every laptop sleep — `colima restart` at session start before trusting CubeStore, and only rebuild while the laptop stays awake (closing mid-build re-corrupts).
- **Apply:** match on the exact `${prefix}_` boundary (not bare `prefix`, else `cfm` matches `cfmx_*`). Verify no legitimate shared/cross-game cubes exist before filtering (here all cubes carried a game prefix). Re-test the actual NL→artifact flow per workspace, not just `local` — workspace-dependent behavior won't show up in unit tests or a single-workspace smoke test.

### A fixed wide `build_range` over sparse data builds empty-month partitions that stop the WHOLE rollup from serving

- **Rule:** A partitioned rollup (`partition_granularity: month`) builds one partition per month in `[build_range_start, build_range_end]`. A month with zero source rows builds a 0-row partition that never becomes a queryable CubeStore table ("Pre-aggregation table is not found ... after it was successfully created"), and the partition-range loader needs the WHOLE range covered — so a single empty month blocks the entire rollup, even months that built fine. Never hardcode `build_range_start: SELECT DATE '2025-01-01'` against a table whose data is a narrower/recent window. Bind the range to the source's real extent: `SELECT MIN(<col>) ... / MAX(<col>) FROM <source_table>`, mirroring the time-dimension's SQL expression so the bound type matches the partition column.
- **Why:** local CubeStore served 0% for the user/behavior cubes. Root cause was NOT the worker and NOT (this round) clock skew — it was that every rollup ranged `2025-01-01→today` while ballistar `active_daily` only had data in 2026-05/06. The 16 empty leading months each built a 20-byte (0-row) partition; the worker's first scheduled sweep never even finished ("Refresh Scheduler Interval: Previous interval #1 was not finished") because ~18 rollups × ~16 empty partitions saturated it, so the canary cubes were never reached. Fix: `MIN/MAX(col)` build_range on all 18 rollup files + coarsen `partition_granularity: month → year` so any sparse interior months (e.g. install-cohort gaps) collapse into a single non-empty year partition. After that, the sweep finished and ballistar/cfm served from CubeStore on both stacks.
- **Signal:** worker logs `Previous interval #1 was not finished with 300000 interval` (sweep can't complete); cubestore logs `Import downloaded temp://...batchNNN... (20 bytes)` for empty months interleaved with real-size imports for data months; a query lists ALL expected partition patterns then "No pre-aggregation partitions were built yet"; `SELECT measure, granularity=month` over the source shows data confined to a few recent months.
- **Apply:** `partition_granularity: year` trades finer incremental refresh for robustness against sparse months — fine for seed data and small marts; flag the prod trade-off (coarser rebuild unit) since the model is shared local↔prod. `MIN/MAX` returns NULL for a game with zero seed data → that rollup just doesn't build and the lambda serves live (harmless). The build_range SQL runs raw against the datasource (Trino), so it needs the physical table + column, not the Cube member.

### A `rollup_lambda` with `union_with_source_data` serves from CubeStore but reports EMPTY `usedPreAggregations`

- **Rule:** Do NOT use `usedPreAggregations` non-empty as the gate for whether a `union_with_source_data` lambda is served from CubeStore — it reports empty/`{}` even when the batch partitions ARE answering the query (the response is a source+batch hybrid, so Cube doesn't credit a single pre-agg). The decisive signal is **latency + a correlated CubeStore select**: a served query is tens of ms and produces a `Running select completed` line in the cubestore log; a Trino-served query is seconds (cold) and produces no cubestore select.
- **Why:** after fixing build_range, ballistar queries returned real rows in 25–47 ms (never-run slices, so not result-cache) while `usedPreAggregations` stayed `{}`. A control query for `paying_dau` — a measure NOT in the rollup — took 3619 ms (Trino) with no cubestore select. The fast queries each produced exactly one cubestore select; the slow one produced none. That correlation, not the (always-empty) `usedPreAggregations`, proves the rollup serves. The earlier plan/handoffs treated empty `usedPreAggregations` as "dormant," which sent the investigation down the wrong path for hours.
- **Signal:** `usedPreAggregations` empty AND latency is tens-of-ms AND `docker logs <cubestore> | grep "Running select completed"` ticks up per query → it IS served from CubeStore (lambda reporting quirk). Empty `usedPreAggregations` AND multi-second latency AND no cubestore select → genuinely live/Trino. A plain (non-lambda) `rollup` DOES populate `usedPreAggregations` — use one as a canary when you need an unambiguous boolean.
- **Apply:** prove pre-agg serving with a never-before-run query (defeats Cube's in-memory result cache, which also gives tens-of-ms) measured against a forced-Trino baseline, and grep the cubestore select log. A measure missing from the rollup silently routes to Trino — check the rollup's `measures:` list before assuming a slow query means dormancy.

### A glossary term whose `primary_catalog_id` is a dead catalog path leaks into the /meta gate as an unresolvable ref

- **Rule:** `primary_catalog_id` is overloaded — it holds a real cube member for dimension/user terms (`mf_users.country`) but a catalog PATH for metric terms (`business_metrics/revenue`). When a term has no derivable `measureRef` (its catalog YAML is missing), the alias index and metric resolver fall back to `primary_catalog_id`; a path can never match `/meta` (members have no `/`), so it surfaces as *"`business_metrics/x` isn't available in this game's data model"*. Fall back to `primary_catalog_id` only when it looks like a member (`memberOrNull`, in `synonym-resolver.ts`).
- **Why:** the `dolphin`/`minnow`/`whale` payer-tier seed terms pointed at `business_metrics/{dolphin,minnow,whale}_payer` YAMLs that never existed and carried no member binding. After the resolver consolidation added a hard `/meta` validation gate, "ARPPU by payer tier (whale/dolphin/minnow)" hard-clarified instead of resolving. The fix had three legs: guard the path fallback, bind the terms via `default_filter_json` on `mf_users.payer_tier` (the chat `buildFilters` now emits member+op+value from a term's `defaultFilter`), and fix `classifyTerm` — it matched singular `'segment'` while the seed used plural `'segments'`, so the segment values fell through to the metric slot.
- **Signal:** chat says a `business_metrics/<id>` (a path, not a `cube.member`) "isn't available"; a seed term's `primary_catalog_id` points at a YAML that doesn't exist; a glossary category is plural (`segments`/`concepts`) but the classifier checks the singular.
- **Apply:** when adding a segment/concept value term, bind it to a real cube member via `default_filter_json` (member it resolves to must be a dimension/measure — `extractMemberNames` does NOT include Cube segments). Never leave `primary_catalog_id` pointing at a non-existent catalog YAML — set it `null`. Keep `id`s unique (the seed had a duplicate `whale`; `INSERT OR IGNORE` + refresh made which one wins order-dependent). Glossary edits need a server restart (re-seed) + ~30s for the chat-service glossary cache to expire.

### A segment monitor scoped only by uid re-aggregates measures over each member's full history

- **Rule:** A segment is a cohort of *users*, not a saved slice of a result cell. When card/monitor queries are scoped only by `{ identityDim equals uids }`, a measure like `revenue_vnd` sums each member's ENTIRE history (all platforms, all time) — so the monitor diverges from the query-builder cell the segment was created from. Scope card queries by the segment's **predicate filters** (the slice) too, not just the uid list. `count`-type measures match either way (they just count the uid set), which masks the bug — only additive/ratio measures (revenue, LTV, ARPU) visibly diverge.
- **Why:** a segment built from an iOS·W20 cell (revenue 1.53B, 1,823 users) showed the right user count but a much larger revenue on its monitor tab. `card-runner.scopeQuery` (server) and `scopeQueryToSegment` (FE) both appended only the identity-uid filter; the predicate's `os_platform=iOS` + `recharge_date in [W20]` constraints — already stored as `cube_query_json.filters` — were dropped, so revenue covered those users' Android + all-time recharges. Fix: AND the predicate filters onto every card alongside the uid scope (both layers; their intersection is the cohort and the slice gives measures the right window).
- **Signal:** a segment's user count matches the source query but a money/ratio metric doesn't; the monitor number is suspiciously round-trippable to "all-time, all-platform"; scoping code appends an identity filter and nothing else.
- **Apply:** `cube_query_json.filters` IS the predicate translated to Cube filters — pass it straight through as the slice scope (`refresh-segment.ts` → `runPresetCards`, and `segmentSliceFilters()` on the FE). Tell the user the monitor reflects the slice (creation modal + monitor banner via `SliceScopeNote`), since "cohort of users" vs "saved slice" is a genuine mental-model trap. Manual (non-predicate) segments have no `cube_query_json` → uid-only scope is correct for them.

### Predicates built from query-builder rows accumulate redundant structure — simplify before display + save

- **Rule:** `buildPredicateFromRows` emits a literal transcription of the query (outer dateRange) + per-row OR-of-AND, which for a single selected cell yields single-child `OR(AND(…))` wrappers and an outer dateRange overlapping the per-row bucket. Run a pure resolver (`simplify-predicate.ts`) that flattens nested same-op groups, unwraps single-child groups, dedupes identical siblings, and intersects overlapping **absolute** inDateRange windows on the same member within an AND. Apply at build time (clean tree is persisted) AND at editor hydration (pre-existing verbose trees render concise).
- **Why:** saving a segment from one iOS·W20 cell produced `AND[ recharge_date in [05-01,06-01], OR[ AND[ os_platform=iOS, recharge_date in [05-11,05-17] ] ] ]` — three nested levels for what is just `os_platform=iOS AND recharge_date in [05-11,05-17]`. Logically equivalent, visually confusing in the editor.
- **Signal:** an editor "Define who belongs" view shows a 1-condition OR wrapping a 1-condition AND; two date ranges on the same time member where one contains the other; duplicate leaves from re-running the builder.
- **Apply:** only intersect absolute `[start,end]` pairs — never relative literals ("this week"), and keep disjoint windows as-is (a non-overlap is the user's intent to see, not silently rewrite). Keep the root a `GroupNode` (callers/render expect it). Updating tests that asserted the old verbose shape is correct, not gaming — the concise tree is the new canonical form.

### A secondary Cube query (compare / side-by-side) that scopes game by token is silently wrong — game rides the `x-cube-game` header

- **Rule:** The cube proxy (`cube-proxy.ts`) is server-authoritative: it **drops the client `Authorization` header** and mints the upstream token itself from `x-cube-workspace` + `x-cube-game` (`workspace-header.ts`). So any client-side CubeApi that wants a specific game MUST forward `x-cube-game` on its transport headers — fetching a per-game token client-side does nothing (the proxy throws it away). The main query already does this (`QueryBuilder.tsx`); secondary clients built via `makeCubeApi` must pass the game id too.
- **Why:** "Compare → Other game (ptg)" showed byte-identical numbers to the active game (ballistar). `makeCubeApi` set `x-cube-workspace` but not `x-cube-game`, so the proxy minted a game-less token → cube-dev served the **default** game's data for the comparison. The `deriveCompareQuery('game:')` filter-swap was also a no-op because game isn't a query filter. Even `prev`-period compare was affected — it only looked right because the active game happened to be the default. Fix: `makeCubeApi(token, apiUrl, gameId?)` forwards the header; `runCompareLoad` passes the target game (`game:<id>`) or active game (`prev`).
- **Signal:** a comparison/secondary/duplicate-query feature returns data identical to the base, or "works" only when the active game is the default; client code fetches a per-game token and assumes it scopes the data; a CubeApi is built without `x-cube-game`.
- **Apply:** scope secondary Cube queries by the **header**, not the token. When comparing across games, the other game's schema may differ — (1) a missing measure/cube makes the WHOLE query 500 (`Cube 'X' not found`) and hides every column, so intersect the query with the target game's `/meta` and compare only shared members, flagging the rest N/A; (2) dimension vocabularies drift (`IOS` vs `ios`), so normalize merge keys (trim+lowercase) or rows won't align. Both are real product gaps the old fake-identical behavior masked.

### Server-side config in logical Cube names breaks on prefix workspaces — translate at egress, don't rewrite each surface

- **Rule:** Anything that hardcodes a logical Cube member (`mf_users.user_count`, `active_daily.dau`) only works on `game_id` workspaces. On `prefix` workspaces (prod cube-dev) cubes are game-namespaced (`ballistar_mf_users`), so those members 404. Don't fix each surface ad-hoc — route logical names through one translation layer (`cube-member-resolver.ts`: `physicalizeQuery` outbound, `logicalizeRows` inbound, `physical/logicalCube`, prefix from `resolveGamePrefix(gameId)`). Keep server config logical + readable; the resolver physicalizes before `/load` and logicalizes responses so consumers keep reading logical keys. Strict no-op when prefix is null → local unchanged.
- **Why:** the segment-monitor revenue fix forwarded the predicate's own (already-physical, FE-originated) slice filters fine, but the `mf_users-hub` preset's logical card measures, the LiveOps `KPI_CONFIG` strip, and the hardcoded `ANOMALY_METRICS` path all hardcode logical names and silently failed on prod. The lesson is the *class*: every workspace-difference bug here traces to logical-vs-physical member naming with no central translator. Data-driven surfaces that already derive names from live `/meta` (LiveOps cohort grid via `findRetentionCube`, identity map seeded from meta, the business-metric anomaly planner) are inherently workspace-agnostic and need no change — copy that pattern for new features.
- **Signal:** a feature works locally but renders empty / 404s `'<member>' not found` on prod; code contains a string-literal `cube.member`; `pickPresetForCube(cube)` or a `cubeNames.has('active_daily')` check fails on prod because the stored/meta cube name is prefixed.
- **Apply:** make the resolver idempotent on the `${prefix}_` boundary (so already-physical inputs pass through un-double-prefixed) and a no-op on null prefix (so local is untouched and existing tests stay green). Match presets by `logicalCube(storedCube, prefix)`. `physicalMember` is for qualified `cube.field`; use `physicalCube` for bare cube names. Background jobs have no request workspace → use the default workspace (`resolveGamePrefix`). Prefer deriving names from live `/meta` over hardcoding when designing a new surface — then no translation is needed at all. The FE bundle can't import the server module, so the FE live-fetch path (`use-segment-cube-query.ts`) routes through a thin mirror (`src/lib/cube-member-resolver.ts`, same contract); prefix comes from `resolveGamePrefix(workspace, activeGameId)` reading the workspace's `gameModel`/`gamePrefixMap`. Physicalize the outbound query, logicalize the `rawData()` response — server card cache covers the warm path, this covers cold-cache live refetch.

### Tenant scope (`x-cube-game`) must propagate from the fetch wrapper, not per-call-site

- **Rule:** the gateway `fetch` wrapper (`api-client.ts → apiFetch`) is the single chokepoint that already attaches `x-cube-workspace` from stored state; attach `x-cube-game` the same way. Any `/api/*` endpoint whose result depends on the active tenant (e.g. `/api/identity-map`, which the suggester resolves against the tenant's `/meta`) breaks if it relies on each call site to remember the game header — and one always forgets.
- **Why:** the segment row-picker never appeared on prod `/build`. It's gated by `/api/identity-map` returning an identity field for the query's cube; `identityMapClient.list()` went through `apiFetch`, which sent `x-cube-workspace` but **not** `x-cube-game`. On the multi-tenant `local`/`game_id` prod cube (`CUBEJS_DEV_MODE=false`), a game-less minted token → empty `/meta` → zero identity suggestions → no picker. The cube *query* path worked (it forwards `x-cube-game`), so data rendered while the picker silently didn't — and local dev's laxer cube returned full `/meta` game-less, hiding the gap. This is the FE-wide generalization of the per-query `x-cube-game` lessons above: the fix isn't stricter enforcement (cube `checkAuth` already fails closed), it's reliable *propagation* of the tenant *selection*.
- **Signal:** a feature works locally but is empty/missing on prod only on a `game_id`/minted workspace; the cube query returns rows but a sibling `/api/*` call (identity map, readiness, prefs-derived) comes back empty; the failing call goes through `apiFetch` and you can make it work by hand-adding `-H "x-cube-game: <id>"`.
- **Apply:** read the active game synchronously from a cycle-free module (`active-game-storage.ts → getActiveGameId()`, reads the `gds-cube:active-game` pref) and attach it in `apiFetch` next to the workspace header — don't import the React game-context (it pulls `segments-client → api-client` into a require cycle). Invalidate any per-workspace cache (e.g. `use-identity-map`) on `gds-cube:game-change` too, since the resolvable cube set differs per tenant. Verify against prod: `curl /api/identity-map` game-less vs `-H x-cube-game:<id>` should differ on a `game_id` workspace and be identical on a `prefix` workspace (shared schema ignores the claim).

### ticket_communications_centralized.created_date_unix is milliseconds — divide by 1000 before from_unixtime
- **Rule:** when reading `ticket_communications_centralized.created_date_unix` for timeline assembly, divide by 1000 (UNIX epoch milliseconds) BEFORE `from_unixtime`. Reading it raw produces a timestamp ~57,881 years in the future.
- **Why:** CS ticket detail reader piped `created_date_unix` directly into `from_unixtime` without scaling. A communication created 2026-06-05 (ms: 1749032400000) became 1749032400000 seconds = year 57881. Every ticket appeared 57k years old; recharge-window joins failed (communication date > cohort window by millennia); timeline was backwards.
- **Signal:** CS ticket timestamps show years in the 57k+ range; communications appear outside the recharge window they match; from_unixtime on a `*_ms` / `*_unix` column produces year-57881 dates.
- **Apply:** `from_unixtime(created_date_unix / 1000)` or `from_unixtime(CAST(created_date_unix / 1000 AS BIGINT))` (Trino requires integer arg). Verify by querying a known recent ticket date — it should land in 2026, not future.

### login_info≠uid alone false-positives on security account-takeover signal — require BOTH login mismatch AND account/security AI label
- **Rule:** when signaling potential account takeover via login-info mismatch (`login_info ≠ user_id`), require **both** a mismatch **AND** a confirmed account/security AI label on the ticket. A mismatch alone produces false positives: legitimate Web/CS-portal logins use a username (a value genuinely different from the game's numeric uid), so ~5–15% of tickets will have `login_info ≠ uid` without any compromise.
- **Why:** securityFlag raised on every Web user who contacted CS (Web login is a username; game uid is numeric — they never match). The flag became noise; every ticket for Web platform lit red. Root cause: the initial implementation flagged mismatch alone as evidence of hijacking. In reality, login_info carries the authentication method/principal name (often human-readable), while uid is the numeric account id — they differ by design for non-game-client logins.
- **Signal:** CS tickets for certain platforms/login methods trigger the security flag despite no actual compromise; securityFlag high-rate on Web vs in-game logins; the flag appears on cohorts with zero reported account issues.
- **Apply:** compute securityFlag as `login_info ≠ uid AND (ai_label ILIKE '%account%' OR ai_label ILIKE '%security%')` — the AND ensures a mismatch only signals when paired with AI-detected suspicious content. Backfill existing tickets by re-running the signal derivation; UI banners on past tickets will update. Test against known compromised + known-legitimate Web tickets to verify F+ rate.

### vitest 2.1.9: a mock-touching beforeEach (mockReset/mockClear) makes a handled rejected-promise in a click handler surface as unhandled rejection — drop the hook or set impl per-test
- **Rule:** a `beforeEach` that calls `mockReset()` / `mockClear()` on a mocked module **transforms handled promise rejections into unhandled ones**. A click handler that catches and logs a rejection (handler is correct) will report as an unhandled rejection on the test that runs after a mock reset, failing the test. Move the reset into a global setup or set the impl per-test instead of using a blanket beforeEach hook.
- **Why:** a member-360 care-history form test had a click handler that catches and logs a Recharge API error (`catch(e) { setError(e) }`). The test suite's `beforeEach` did `vi.mocked(rechargeApi).mockReset()` on shared mocks. After running a prior test that exercised the handler, the reset moved the mock config (removing the implementation, leaving an undefined return) but left the handler's `.catch()` in the promise chain. The *next* test's recharge call → undefined → rejected promise → the global rejection listener sees it before the handler's .catch fires (or at least reports it that way) → test fails with "unhandled promise rejection: undefined is not a function." The handler was fine; the test framework + mock lifecycle caused the report.
- **Signal:** a test fails with `unhandledRejection` or `UnhandledPromiseRejectionWarning` for a handler that looks correct; a prior test in the same file exercises the same handler fine; adding `await vi.runAllTimersAsync()` before the assertion or removing the `beforeEach` mock reset makes the test pass. The error only appears when the handler's test runs after a prior test + reset.
- **Apply:** keep mocks per-test by setting the impl inline (`vi.mocked(api).mockImplementation(…)` per test) instead of a shared `beforeEach` reset, or move all resets to a global `config.setupFilesAfterEnv`. If you must use beforeEach, reset sparingly — only the modules that genuinely need a clean slate for test isolation. Cover the "handler survives reset" scenario in a regression test that explicitly runs the handler *after* a sibling test + reset within the same file.

## Performance / event loop

### A list route that JSON.parses a large per-row column starves the single event loop
- **Rule:** a list/index endpoint must never `JSON.parse` (or do unbounded synchronous CPU on) a column that can grow large, once per row. Node is single-threaded and better-sqlite3 is synchronous, so the parse blocks *every* concurrent request — a trivial `/health` poll suddenly takes seconds. List views return summary columns (counts); the heavy payload is fetched per-item on the detail route.
- **Why:** `GET /api/segments` did `SELECT *` then `hydrateSegment` per row, which `JSON.parse`'d each segment's `uid_list_json`. Two seed cohorts carried ~2.2 MB each (1M+ uids); ~7.5 MB of JSON parsed synchronously per call ≈ 180 ms of pure main-thread block. `...row` also shipped the raw blob *and* the parsed array (double payload). Under concurrent clicks/polls the blocks serialized — a load probe showed `/api/health` p99 going 0 ms idle → 2172 ms under 30 connections, throughput pinned at 55 req/s; the event-loop monitor logged `maxMs` 1131–1343 ms blocked. Fix: list passes `includeUidList=false` (skip the parse, never ship `uid_list_json`); detail route still hydrates full uids. Result: 182 ms → 7 ms single-request, 55 → 629 req/s, probe p99 2172 → 235 ms.
- **Signal:** a *trivial* endpoint (health, notifications, a static read) returns 504 / multi-second `responseTime` while the server is busy — the giveaway is the slow endpoint being cheap, so the time is queueing behind a blocked loop, not its own work. `[event-loop] blocked over threshold` warns fire (see `runtime-observability.ts`). A list response is suspiciously large; one column dominates its bytes.
- **Apply:** grep list/index handlers for `JSON.parse` of a `*_json` column inside a `.map`/loop; gate it behind an `include*` flag the detail route sets true and the list sets false, and destructure the raw blob out of the spread so it's never serialized. Confirm with the load harness (`load-test/`, latency-under-load: probe a cheap endpoint while driving the heavy one) that the cheap path's p99 stops ballooning. Any FE list code that consumed the full array (here the bulk CSV export) must fetch it per-item on demand instead. Both services ship `[slow-request]` + `[event-loop]` warns so the next instance is greppable in `logs/dev-all.log` rather than hunted by hand.

### A wedged in-stack Cube stays "running" — only a healthcheck makes it visible
- **Rule:** the stock `cubejs/cube:latest` image ships no `HEALTHCHECK`. When Cube's single event loop blocks (refresh-worker partition spin-loop, or a schema-compile / query storm against a laptop-grade host), the process keeps `Up` and `docker ps` shows no health — but `/readyz` and `/meta` starve and CPU pegs ~100%. `restart: unless-stopped` never fires (it only triggers on *exit*), so the dead cube sits there until someone restarts it by hand. Give it a `/readyz` healthcheck so the wedge flips to `unhealthy` and surfaces.
- **Why:** the FE build page showed *"Unable to load meta data. Cube request timed out after 15s"* on `:11000`. The server proxy (`cube-proxy.ts`) returned 504 at its 15s ceiling on `GET /cube-api/v1/meta?extended=true`, while the cube container's logs were ~15 min stale and CPU sat at 102%. A node probe from inside the server container to `http://cube_api:4000/readyz` *timed out at 8s* — proving the cube, not the proxy or the network, was the hang. `CUBEJS_REFRESH_WORKER` was already `false` (the documented spin-loop guard, see `docker-compose.prod.yml`), so the trigger was the games-readiness probe firing dozens of concurrent per-game `/load` queries that compiled 6 tenants' schemas at once and never unwound. `docker restart cube-playground-cube-api` cleared it: plain `/meta` 200 in 4s (cold compile), `?extended=true` 200 in 47ms.
- **Signal:** any cube-backed surface (build meta, catalog, readiness) returns *"Cube request timed out after Ns"* / 504 while the rest of the app is fine. Confirm with `docker stats <cube> --no-stream` (CPU pegged, memory normal), stale cube logs (no recent request lines), and a `/readyz` probe that times out. Distinguish *wedged* (CPU 100%, restart fixes) from *slow Trino* (cube responds, queries are slow) — the readyz timeout is the discriminator.
- **Apply:** keep `CUBEJS_REFRESH_WORKER=false` **on the serving instance, permanently** — pre-aggregation builds belong in the dedicated `cube_refresh_worker` / `cube_refresh_worker_dev` containers (own event loop; a wedged refresh can't starve the API's `/readyz`/`/meta`). Corollary: with serving-side builds off, rollups that are *defined but never built* hard-fail every matching query with "No pre-aggregation partitions were built yet" — the worker container is mandatory once any model ships a `pre_aggregations:` block. Healthcheck the cube on `/readyz` (compose `healthcheck:` block — node `fetch` probe, generous `start_period` for cold compile). Avoid hammering the laptop cube with fan-out probes (the readiness sweep is the usual culprit). When meta times out, check the cube container's CPU/readyz *before* suspecting proxy/auth/header code — the wedge masquerades as an app bug.

### `second time provided was later than self` is a CubeStore clock jump — restart CubeStore, not Cube API
- **Rule:** `Internal: Execution error: Internal: second time provided was later than self` is Rust's `SystemTime::duration_since` error (the second arg is later than the first → negative duration). It surfaces from CubeStore's query-execution timeout path, **not** the Node Cube API. On Docker Desktop for Mac the VM wall clock jumps backward on sleep/wake; if CubeStore captured `SystemTime` references before the jump, its deadline math goes negative and every pre-aggregated query errors. Fix = `docker restart <cubestore>` to reset the in-process time baseline; pre-agg data is persisted to the volume and survives.
- **Why:** the `:11000` build page returned that error for `active_daily.dau` (an HLL `cardinality(merge(...))` measure served entirely from `prod_pre_aggregations.*` in CubeStore) while a Trino-passthrough recharge query on the same stack succeeded. Clock read *correct* at diagnosis time, yet errors began at a discrete instant (12:09:36Z) ~6h into a clean CubeStore start — a jump event, not gradual drift or startup corruption. Only 4 errors in a 14-min window. Restart cleared it: the exact query replayed through Cube API `/load` returned 200, 0 new errors after.
- **Signal:** "later than self" in the FE error or CubeStore logs; errors that begin at a sharp timestamp well after startup (grep `docker logs --timestamps <cubestore> | grep "later than self"`); pre-aggregated (CubeStore-served) queries fail while Trino-passthrough queries on the same stack work; container clock reads correct *now*. The split between pre-agg vs passthrough is the discriminator — it isolates the fault to CubeStore.
- **Apply:** restart CubeStore (not Cube API — a healthy Cube API masks this; `cube-playground-cube-api` was `Up (healthy)` throughout). A `/readyz` healthcheck won't catch it (readiness passes; only query-time time math fails), so it stays invisible until a query hits it — when a `:11000` query errors but meta/readiness are fine, suspect CubeStore time state after a laptop sleep before anything else.

## Deployment / Docker build

### The prod image builds from a clean checkout in network isolation — commit every build input, forward the proxy
- **Rule:** the production image builds from tracked files only, inside a sandbox with no direct internet. Every input a `RUN` step needs must be (1) committed to git and (2) reachable through the org proxy. Gitignored lockfiles, host-only env, and ambient registry config simply don't exist in the build context.
- **Why:** the prod deploy died in sequence, each blocker hidden behind the last. `COPY package-lock.json` failed — the 3 lockfiles were gitignored. Once committed, `apt-get` then `npm` hit "Network is unreachable" (build `RUN` steps have no internet; they need `HTTP_PROXY`/`HTTPS_PROXY` forwarded as build args). The first proxy fix used a dedicated `BUILD_HTTP_PROXY` that was **empty** in CI (`length: 0` in the trace) — fixed by falling back to the CI host's always-exported `http_proxy`/`https_proxy`. Then `npm ci` ERESOLVE'd (antd@4 vs `@ant-design/compatible` peer) because `.npmrc` (`legacy-peer-deps`) was never `COPY`'d into the deps stage. None reproduced on a dev box — it had internet, the lockfiles on disk, and global npm config.
- **Signal:** `COPY <file>: not found` during build; `Network is unreachable` / errno 101 / `ETIMEDOUT` in an apt/npm/pip `RUN`; an `npm ci` ERESOLVE that doesn't repro locally; "builds fine in dev compose" but fails in CI. A `BUILD_*` proxy var that traces as empty.
- **Apply:** un-ignore and commit lockfiles (`npm ci` requires them; verify `npm ci --dry-run` is in sync first). Wire proxy build args through a compose anchor with host fallback — `${BUILD_HTTP_PROXY:-${http_proxy:-}}` (nested default works in compose ≥ v2). Forward both upper- and lower-case (`HTTP_PROXY` + `http_proxy`) since tools split on convention. `COPY .npmrc ./` before `npm ci`. A green `docker compose build` on a laptop proves nothing about the sandboxed CI build — the only honest test is the CI pipeline (or `docker build` with networking disabled).

### tsconfig `rootDir`/`include` defines the emit layout — keep it aligned with the container's CMD and COPY paths
- **Rule:** a service that ships as `node dist/index.js` must have a tsconfig that emits the entry at exactly `dist/index.js`. `rootDir: "."` plus an `include` that pulls in `test/` makes `tsc` mirror the source tree under `dist/` (→ `dist/src/index.js`) and scatter compiled test files into the image — silently shifting the entry path and every relative COPY target the Dockerfile was written for.
- **Why:** server + chat-service both used `rootDir: "."` and `include: ["src/**/*", "test/**/*"]`, so `tsc` emitted to `dist/src/index.js` while the Dockerfile ran `node dist/index.js` and copied migrations to `dist/db/migrations`. The server container crashed with `MODULE_NOT_FOUND` and never became healthy; because `web depends_on server: { condition: service_healthy }`, the whole deploy aborted — a build that compiled cleanly still failed to ship. chat-service carried the identical latent defect, invisible only because nothing gates on its health.
- **Signal:** a container exits immediately with `MODULE_NOT_FOUND` / "Cannot find module …/dist/index.js"; `dist/` has an unexpected extra `src/` level or stray compiled `*.test.js`; a health-gated dependent service (web) aborts the deploy even though its own image built green.
- **Apply:** set `rootDir: "src"` and scope `include` to `["src/**/*"]` so the entry lands at `dist/index.js`; tests run via their own vitest config and need no `tsc` emit. Reproduce the prod boot locally — run the compiled output (or the image) and hit the healthcheck (`/api/health` → 200) — before pushing; a passing `tsc`/`vite build` says nothing about the runtime entry path. Audit sibling services for the same defect; the one nothing health-gates will be broken silently.

### `tsc` emits only `.js` — every non-TS asset a runtime path reads needs its own Dockerfile COPY

- **Rule:** the `build` step is `tsc`, which copies nothing but compiled `.js` into `dist/`. Any directory a runtime path reads by `readdir`/`readFile` — `.sql` migrations, `.yml` presets, `.json` seeds — must be COPY'd into the image explicitly, at the path the loader resolves relative to `dist`. If you add a new disk-read asset dir under `src/`, add the matching COPY in the same change.
- **Why:** the business-metrics registry loader (`getRegistryDir() → dist/presets/business-metrics`) and the dashboard-starter-pack loader both `readdir(..., '*.yml')` at runtime, but the Dockerfile copied only `dist/` + `src/db/migrations`. So the prod image shipped an empty `dist/presets/**`, and the metrics catalog + starter dashboards loaded **zero** entries — while local `npm run dev` (tsx runs from `src/`, where the YAMLs sit) looked fine. The migrations dir had the exact same hazard and was already special-cased; presets were the next instance of the class.
- **Signal:** a registry/catalog/list is empty *only* in the container, never under `tsx`/dev; the loader does `readdir`/`readFileSync` on a dir under `src/…`; the build is `tsc` (not a bundler that inlines assets); `dist/<that dir>` is absent or empty in the image. Bonus: a redeploy wipes entries that were written at runtime into a non-volume `dist/` dir.
- **Apply:** mirror the migrations precedent — `COPY --from=build /app/server/src/presets ./dist/presets` (and any future asset dir). For assets that are *also written at runtime* (scaffolded metrics, user dashboards), shipping the seed in the image isn't enough — point the loader's registry dir at the persisted `/data` volume and seed-copy from the baked presets on first boot, or every redeploy silently discards runtime-created entries. **For a single small asset with one consumer, prefer inlining it as a TS string constant** — it compiles into `dist` with the module, so there is no asset to copy and the bug class can't recur via a stale/cached deploy. The lakehouse snapshot DDL hit this exact ENOENT (`dist/lakehouse/segment-membership-ddl.sql`); a COPY-based fix was added but the error persisted across a deploy, so it was inlined as `segment-membership-ddl.ts` (`SEGMENT_MEMBERSHIP_DDL` + `DDL_TABLE_PREFIX_TOKEN`) and the COPY removed. Only `readdir`'d multi-file dirs (migrations, preset YAMLs) still need the COPY. Tell that you should have inlined: a targeted COPY lands but the ENOENT *persists* — the running image is stale/cached.

### Checked-in assets must not live under a `.dockerignore`d directory — the image silently ships without them
- **Rule:** before placing a git-tracked runtime asset (seed JSON, snapshot, fixture) in a directory, check `.dockerignore`. A dir excluded as "writable scratch" (`chat-service/runtime`) erases everything under it from the build context — including files you deliberately committed. The build succeeds, the container boots, and the feature quietly degrades to its no-asset fallback in prod only.
- **Why:** the pregenerated starter-questions seed was put in `chat-service/runtime/seed/` (next to the chat snapshot precedent), but `.dockerignore` excludes `chat-service/runtime` wholesale and the Dockerfile recreates `runtime/` empty. Prod served the template/static fallback while dev served the seed — spotted only by eyeballing the prod landing page showing the old question list.
- **Signal:** a file-backed feature works under dev/tsx but falls back in the container; the loader path contains a dir name that also appears in `.dockerignore`; the fallback is graceful so no error logs.
- **Apply:** keep committed runtime-read assets in their own dir (`chat-service/seed/`) with an explicit Dockerfile `COPY`, never under scratch dirs. When adding a seed/asset file, grep `.dockerignore` for every path segment, and verify with `docker build` + `ls` in the image (or at minimum trace the COPY chain) before calling it deployed.

### Don't gate the prod bundle on a whole-repo typecheck
- **Rule:** the production bundle step (`vite build` / esbuild) must not be chained behind `tsc --noEmit`. esbuild transpiles without type-checking, so type debt that never reaches the emitted bundle — false positives from loose `strictNullChecks`, stale test fixtures, third-party prop-type drift — will block an otherwise-shippable deploy. Keep typecheck a separate, non-blocking job.
- **Why:** `build: "tsc --noEmit && vite build"` failed CI on 73 pre-existing type errors, none of which affected the esbuild output. The deploy was blocked by debt entirely unrelated to the change being shipped; `vite build` alone succeeded in ~30s.
- **Signal:** prod build/deploy fails only at the `tsc` step, in files the current change never touched; the bundler step succeeds on its own; the error count is large and clearly pre-existing.
- **Apply:** `build: vite build`; keep `typecheck: tsc --noEmit` and a `build:strict` alias for local/PR use. If you want type safety in CI, add typecheck as its own job that can fail without blocking deploy — not a serial prereq of the bundle. Caveat: loose strictness masks real bugs too, so a clean typecheck still has value (just not as the deploy gate) — fix genuine type errors on their own track.

### Claude Agent SDK `bypassPermissions` fails closed when the container runs as root — set `IS_SANDBOX=1`
- **Rule:** when the chat-service runs the Claude Agent SDK with `permissionMode: 'bypassPermissions'` (so curated MCP tools run without an interactive prompt — there's no TTY in a container), the spawned Claude Code CLI passes `--dangerously-skip-permissions`, which the CLI **refuses under uid 0** unless `IS_SANDBOX=1` (or `CLAUDE_CODE_BUBBLEWRAP`) is in the child env. Prod containers run as root by default; dev machines don't — so it works locally and dies only in prod.
- **Why:** every chat turn died with the opaque `Claude Code process exited with code 1`. The child wrote nothing the SDK surfaced by default — the `query()` error carries only the exit code, and the SDK pipes child stderr to `options.stderr` only when that callback is set, otherwise discards it. After wiring an `options.stderr` capture and folding it into the thrown error, the real line appeared: `--dangerously-skip-permissions cannot be used with root/sudo privileges`. The CLI binary's own check is `permissionMode==='bypassPermissions' && getuid()===0 && process.env.IS_SANDBOX!=='1'` → `exit(1)`. Safe to bypass here because every builtin tool (`Bash/Read/Write/Edit/WebFetch/WebSearch`) is disabled, so the agent can only call the curated cube MCP tools — there is no shell/filesystem to "skip permissions" on.
- **Signal:** an SDK/CLI subprocess exits non-zero only in the container, never locally; the opaque error names just an exit code; the difference between envs is who the process runs as (root in image vs your user); `bypassPermissions`/`--dangerously-skip-permissions` is in play.
- **Apply:** set `IS_SANDBOX: '1'` in the env passed to `query({ options: { env } })` (not the parent process env — the SDK forwards the options.env bag to the child). Keep an `options.stderr` capture wired so the SDK's opaque exit errors always carry the child's real stderr — debugging a subprocess you can't see the logs of is otherwise guesswork. Alternative fix (heavier): run the container as a non-root `USER` with the data volume + Claude home chowned to it.

### A new auth-gated surface must honor the app-wide `AUTH_DISABLED` break-glass, or it fails closed while everything else is open
- **Rule:** when adding a route/service that enforces its own auth (a shared secret, a token check, a grant lookup), wire it to the same `AUTH_DISABLED` (or equivalent break-glass) switch every other surface reads. A new gate that ignores the global switch will keep denying after an operator has deliberately turned auth off everywhere else — an inconsistent posture that reads as "broken", not "secure".
- **Why:** `/internal/access` (the cube↔server auth bridge) was added after the app-wide `AUTH_DISABLED` synth-admin pattern and hardcoded `if (!CUBE_AUTH_INTERNAL_SECRET) 503`. Prod ran `AUTH_DISABLED=true` as its standing config, so login, the server authz gates, and the SSO config endpoint were all open — but this one route still demanded a secret that was never set, returning `503 internal_secret_not_configured`. It surfaced only on the `local` (minted-JWT) Cube workspace as "Unable to load meta data — auth API 503"; the `prod` workspace (`authMode:none`, never calls the route) looked fine, masking that the failure was one missed surface.
- **Signal:** one feature/surface 503s/401s/404s for "auth" reasons while the rest of the app is confirmed open; the failing surface is newer than the auth-disable mechanism; the gate references a secret/grant with no `if (authDisabled()) return` escape; a sibling code path with a different auth mode (`authMode:none`, public route) works, hiding the gap.
- **Apply:** grep for the break-glass flag (`AUTH_DISABLED`) when adding any gate and mirror its bypass — skip the secret check *and* return the same synthesized all-access principal the rest of the app uses (here: `allowedGames:['*']`, with the consumer treating `'*'` as "all"). Keep the bypass branch unreachable when the flag is off so real-auth behavior is unchanged. After adding a gate, test both flag states across *every* surface that hits it, not just the one with the permissive auth mode.

### A new compose service's image tag must match what's already cached on the network-isolated runner — don't pin a fresh tag
- **Rule:** when adding a service that runs from a pulled image (not built in-repo) to the prod stack, use the *exact* tag another stack already deploys on that runner. The deploy host reaches Docker Hub only through a slow org proxy and times out on cold pulls; a tag that's already in the local image cache resolves instantly, a fresh one forces a doomed `registry-1.docker.io/v2/` round-trip.
- **Why:** co-locating an in-stack Cube added `cube_api` + `cubestore`. The build steps (server/web/chat) reach Hub via BuildKit proxy args and their base layers were long cached, so they built fine — but a compose *image pull* is done by the Docker **daemon**, which on this runner couldn't even finish the registry ping (`Get "https://registry-1.docker.io/v2/": context deadline exceeded`). The deploy died *after* `down --remove-orphans` tore the old stack down → prod app offline. Root cause: I pinned `cubejs/cubestore:v1.6.46`, but the already-live sibling stack (`cube-api`) runs `cubejs/cube:latest` + `cubejs/cubestore:latest`, so *those* layers were cached and `v1.6.46` was not. Switching both to `:latest` made compose resolve from cache — `CACHED`, no `Pulling`, green.
- **Signal:** deploy fails at `docker compose up` on `<image> Pulling` → `context deadline exceeded` / registry timeout, while the in-repo *builds* succeed; the failing image is one newly added to the stack; a sibling stack on the same runner uses a different tag. Bonus hazard: the failure lands after teardown, so prod is down until a deploy succeeds.
- **Apply:** before adding a pulled-image service, check what tag is already running on the runner (`docker image ls` on the host, or read the sibling stack's compose) and match it exactly. Reproducibility-pinning is good practice but only *after* that tag is seeded on the host (pre-pull on the runner, or a reachable registry mirror) — otherwise the pin is what breaks the deploy. Keep co-located images on the same tag in lockstep. The "build is green locally" signal is worthless here: laptops pull from Hub directly; the sandboxed runner does not.

### Verify component props against the *installed* major version, not the API you remember
- **Rule:** before using a library component prop, confirm it against the version in the lockfile. Major versions rename props with no deprecation shim, and the wrong name often type-checks loosely (or the gate is off) so it fails silently at runtime, not at build.
- **Why:** an "activate to CDP" modal used antd v5's `<Modal open={…}>` but the project pins antd `4.16.13`, where the prop is `visible`. The modal simply never opened — no error, the trigger looked dead. Surfaced only while auditing the type errors during the deploy.
- **Signal:** a component renders nothing / does nothing with no console error; a boolean-controlled widget (modal, drawer, popover, tooltip) ignores its open state; the prop name matches current docs but the installed major version is older.
- **Apply:** check `package.json`/lockfile for the major version, then the prop name for *that* version (antd v4 `visible` → v5 `open`; similar churn in MUI, react-router, recharts). When `strict`/`strictNullChecks` is off, TS won't catch an extra/wrong prop — treat the installed-version check as manual.

### A `:ro` bind-mount sourced from a git **submodule** path is empty on the deploy runner — the parent push carries only a gitlink
- **Rule:** when a compose service bind-mounts files that live under a submodule path (`./cube-dev/cube:/cube/conf`), those files do NOT travel with a `git push` of the parent repo — the parent tracks only a 160000 gitlink pointer, not the submodule's blobs. On a network-isolated runner that can't fetch the submodule URL, the mount source resolves to an empty dir and Docker happily mounts nothing. A service that reads config/models from that mount boots but serves an empty/erroring surface.
- **Why:** the in-stack `cube_api` mounted `./cube-dev/cube` (a submodule → `metrics-catalogue.git` over GitHub SSH). `git ls-files cube-dev/` returned a single entry (the gitlink); the 52 model YAMLs were never in the parent tree, so pushing to the `second` (kraken) remote shipped no models. `cube_api` `/meta` returned `Error: Model files not found`; the server's background segment-refresh calls `suggestIdentities()` → `/meta` throws → caught → returns null → every `local`-workspace segment broke with the misleading `broken_reason: "no identity-field mapping for recharge"`. The error names identity mapping, but the real failure is two layers down: no models → no meta → no suggestion.
- **Signal:** a downstream feature reports a domain-specific failure (`no identity-field mapping for X`) but the cube/entity demonstrably exists in source; the failure is prod-only and post-deploy; the data source is a `:ro` bind-mount whose host path is under a submodule; `git ls-files <submodule-path>` shows ~1 entry; the deploy remote is network-isolated and the submodule URL is external. Probe the *source* surface directly (`curl .../cube-api/v1/meta`) — "Model files not found" beats guessing at the consumer.
- **Apply:** don't bind-mount runner-deployed assets out of a submodule path. Either vendor the files into the parent repo (de-submodule the subtree so the blobs are tracked and pushed — matches this repo's deployment-guide intent: "vendor `cube-dev/cube/`"), `COPY` them into the image at build, or run `git submodule update --init` on the runner only if it can reach the submodule URL. When a consumer error blames a logical resource, verify the resource's data source is actually present on the box before touching consumer logic — fixing the consumer (e.g. prefix-matching in the resolver) treats a symptom and can mask or worsen the real outage.

### Background jobs must derive workspace-scoped config from the artifact's own workspace, not the deployment default
- **Rule:** when a background/cron job processes an artifact that carries a `workspace` (segment, dashboard tile, alert), resolve workspace-scoped settings — the cube naming model (prefix vs game_id), Cube URL, token — from THAT artifact's workspace, not from `getDefaultWorkspace()`. A request handler gets the workspace from the request header; a background job has no request, so reaching for the global default silently couples every artifact to one workspace's config.
- **Why:** `resolveIdentityField` derived the cube prefix via `resolveGamePrefix(gameId)` → `getDefaultWorkspace()`. On prod the default workspace is prefix-model (`ballistar`), but segments are refreshed against the in-stack game_id cube. For a game_id segment the prefix should be null; instead it came back `ballistar`, so a manual identity override stored logical (`recharge.user_id`) got physicalized to `ballistar_recharge.user_id` and `/load` against the game_id cube failed. Latent because auto-suggest (no override) returned logical names that happened to match the raw cube — only an override tripped it. Fixed by adding `resolveGamePrefixForWorkspace(workspaceId, gameId)` and passing `segment.workspace` through.
- **Signal:** a feature works for the no-override / default path but breaks once a per-artifact override or a non-default workspace is involved; a background job calls a `getDefault*()` / `resolveGamePrefix(gameId)`-style helper while the artifact it's processing has its own `workspace` column; prefix/url/token come from different workspaces than the data being queried.
- **Apply:** thread the artifact's `workspace` into the resolver and branch on the *resolved* workspace's `gameModel`. Keep a default-fallback overload for callers with no workspace context (request handlers already have `req.workspace`). Regression-test with the default workspace set to one model and the artifact on another — a single-workspace test can't catch the divergence. If the job also executes `/load` or `/meta`, derive the Cube ctx (url+token) from the same workspace; deferring that is safe only while the global `CUBE_API_URL` happens to equal the artifact-workspace URL (e.g. all artifacts on the in-stack cube).

### Server-side Cube calls and the FE query path must mint the token under the SAME principal
- **Rule:** the principal in a minted Cube JWT (`userId` claim) decides whether `cube-dev`'s `checkAuth` can resolve the caller. The FE query path and every server-side `req.cubeCtx` Cube call must use a principal the target cube can actually resolve — and they must agree. The playground mints under the **service principal** (`playground`) for the FE query token (`cube-token.ts` → `resolveCubeTokenForWorkspace(ws, game)` with no userId), but `req.cubeCtx` (built in `workspace-header.ts`) mints under `user.email` so cube-dev can double-enforce per-user access. If the cube can't resolve that email, server-side calls fail while the FE query still works.
- **Why:** the segment row-picker checkbox vanished on dev `/build` — but only once real Keycloak auth was turned on. `/api/identity-map`'s suggester calls `/meta` through `req.cubeCtx`, whose token carried `userId: 'khoitn@vng.com.vn'`. The dev cube resolves users from a file/bridge that doesn't know that email → `/meta` denied → suggester's `try/catch` swallowed it → empty map → `hasIdentityFor('recharge')` false → no expansion-mode selection column. The cube **query** rendered fine because the QueryBuilder fetches its token from `/api/playground/cube-token`, which mints under `playground` (resolvable). So `/load` worked while a sibling `/api/*` came back empty — same symptom as the `x-cube-game` propagation bug, but a different cause: **token principal mismatch**, not a missing header. `AUTH_DISABLED=true` masks it (no email → `playground` everywhere); the real-auth cutover unmasks it.
- **Signal:** a tenant-scoped `/api/*` (identity-map, readiness, coverage) is empty while the cube query for the same page returns rows; the failing call goes through a server-side `req.cubeCtx`/`getMetaWithCtx` path; it started failing exactly when real auth was enabled; `curl`-ing `/api/identity-map` with the workspace+game headers but no app-JWT (service-principal path) succeeds, while the logged-in browser fails; the minted token's `userId` is an email the cube's auth store has never seen.
- **Apply:** for **schema introspection** (identity-map, meta-cache, readiness) prefer the service principal — it's cube *metadata*, not user-scoped data, so per-user enforcement buys nothing and only adds a failure mode. Either mint introspection tokens under `playground` explicitly, or make the cube's user store resolve every authenticated principal (seed emails, or have the bridge emit `'*'` for admins — see the `checkAuth` wildcard-expansion). Before the `:11000`/prod real-auth cutover, decide the cube principal deliberately (service-principal row vs per-user-email + resolvable grants); a green dev run under `AUTH_DISABLED` does NOT exercise this path. Dev keeps the bypass (`AUTH_DISABLED=true` in `.env.local`) by design — so this bug only surfaces on stack/prod, where it must be caught by a bridge smoke-test, not a local reload.

### CS ticket schema has stale metadata + dedup traps — read from `cs_ticket_new_master`, dedup before join
- **Rule:** `iceberg.cs_ticket` table has two `*_master` table pointers: an old `cs_ticket_master` with stale metadata, and the active `cs_ticket_new_master`. Always query the `_new_` version. Also: every ticket has multiple rows in both the master table (one per `ai_label`) and in `cs_ticket_map_ai_label` — **dedup before joining**, else ticket counts inflate. Use `row_number() OVER (PARTITION BY ticket_id ORDER BY <stable_col>) = 1` or a GROUP BY to collapse N rows into one per ticket per table before aggregating.
- **Why:** CS Care tab queries reached the wrong master table, serving 0 tickets; fixing the table name surfaced the second issue: ticket counts ballooned (1 ticket in the schema became 3–4 in the report because the multi-row-per-ticket structure fanned out during aggregation). Same pattern exists in `cs_ticket_map_ai_label` — the tab saw "3 tickets in High category" when there was only 1.
- **Signal:** CS-ticket-backed surfaces show zero data, or ticket/issue counts are implausibly high (often 3–5× reality); queries hit `cs_ticket_master` (old spelling) or omit a dedup before joining the label map.
- **Apply:** Hardcode `FROM iceberg.cs_ticket.cs_ticket_new_master` in every query touching that schema. For any surface that counts tickets or aggregate attributes by category/sentiment/rating: first `SELECT DISTINCT ON (ticket_id) ... FROM cs_ticket_new_master` or `GROUP BY ticket_id`, then join to the labeled attributes via that deduplicated set. Verify via spot-check: run the query and manually count tickets in a sample category — if query result > manual count, you missed a dedup.

### CS segment join coverage is ~8% and membership spans multiple identity namespaces — surface coverage honestly
- **Rule:** CS ticket data uses `PSID` or proprietary `contact_id` as the key, but segment membership uses `user_id` (internal identity). The join path (`cs_ticket → member_map → segment`) is incomplete — only ~8% of users have a valid join. Never silently drop contact failures; surface the coverage %, note what fraction couldn't be contacted, and warn the user that the watchlist is filtered to joinable members only.
- **Why:** CS Care tab computed "contacted members / total members" as a "coverage pct", but the denominator was segment size and the numerator was (much smaller) joined contacts. Coverage looked like 3–5%, which seemed like a data quality issue; actually, the schema's limitation meant 3–5% of the segment was even contact-reachable. Without surfacing the gap, the tab's "coverage" metric was meaningless — a user would think "only 5% were contacted" when it should be "we can only reach 5% of this cohort."
- **Signal:** a CS/contact-history surface shows very low coverage/contacted rates while the segment itself is healthy; queries join on an ID that's sparse (PSID, contact_id) in your data; the member-count metric doesn't match the segment's UI size.
- **Apply:** compute both `totalMembers` (segment cardinality) and `joinableMembers` (members with a contact_id found in the ticket master); display **both** and emphasize the joined subset in the watchlist. A "Coverage %" that divides by totalMembers is honest; a banner like "Watchlist shows joinable members only (N of segment size)" prevents misinterpretation. When the join rate is structurally low (platform-dependent support), note it once in the product (a help popover) rather than every time the metric appears.

### CS sentiment codes are 1=Negative/2=Positive/3=Neutral; ticket_rating is 1–5 double; vip_id is a flag not a tier
- **Rule:** when reading CS sentiment/rating codes, use the right column and invert expectations if needed. Sentiment in the schema uses 1 (Negative) / 2 (Positive) / 3 (Neutral) — not 0-based or alphabetical. Ticket rating is a DOUBLE 1–5 (fine to cast to INT for display). The `vip_id` column is a 0/1 flag (whether the ticket was tagged VIP), **not** a customer tier or profile id — don't try to join it to a tier table.
- **Why:** a CS Care tab draft showed "negative sentiment" when it should have shown "positive". Sentiment-aggregation logic treated 1 as the worst (true) but later rendering inverted the scale again (double-inversion bug); a rule to filter `sentiment < 3` ("less than neutral") was correct logic but unintuitive. Separately, on another project a similar schema has `vip_id = customer_vip_tier` (a foreign key) — don't assume.
- **Signal:** a sentiment-based filter has counterintuitive conditions (e.g. `sentiment >= 2` to show positive); a rendering maps high numbers to "bad" status; `vip_id` has a wide cardinality (not just 0/1) or a join to a tier table fails.
- **Apply:** add a comment in every query that decodes sentiment: `-- 1=Negative, 2=Positive, 3=Neutral`. For rating, ensure the `CAST(ticket_rating AS INT)` or display is consistent (1–5, not 0–4). Never assume `vip_id` is anything other than a flag — test the column's cardinality and NULL pattern before designing around it. Seed a test fixture with sentiment examples (1, 2, 3) and verify the rendered labels match the intended meaning.

### A relative Vite `base` breaks deep non-hash routes under nginx (OAuth callback)
- **Rule:** an SPA served from the site root must build with absolute `base: '/'`, not `base: './'`. A relative base emits `<script src="./assets/…">`; the browser resolves that against the CURRENT path, so on a deep route like `/auth/callback` it requests `/auth/assets/…`, which the nginx SPA fallback (`try_files … /index.html`) returns as `index.html` (`text/html`) → "Expected a JavaScript module but got MIME text/html" → the bundle never loads → the app never boots.
- **Why:** real-auth login on the stack got stuck on `http://localhost:11000/auth/callback?code=…` with a blank page; the server logged NO `POST /api/auth/keycloak/callback`. The OAuth code arrived but the SPA never mounted to exchange it. HashRouter masked the bug for every normal route (`/#/build` keeps `pathname=/`, so `./assets` still resolves to `/assets`); only the one real deep path — the OAuth callback — tripped it. Vite dev (`:3000`) also hid it: the dev server resolves `/assets` regardless of route, so login worked there. Fixed by `base: '/'`.
- **Signal:** a deep non-hash route (OAuth callback, anything not under `/#/`) shows a blank page only on the nginx/static build, never on vite dev; console says a module script got `text/html`; `curl /<deeproute>/assets/<bundle>.js` returns 200 `text/html` (the index fallback) instead of the JS; the server never sees the request the page was supposed to make.
- **Apply:** keep `base: '/'` for any root-served SPA; reserve `./` for genuinely path-relative embeds. When adding a real (non-hash) route, test it on the built/nginx image, not just vite dev — `curl` the route's HTML and confirm `<script src>` is absolute (`/assets/…`). If the app must serve under a path prefix, set `base` to that prefix, not `./`.

## Telemetry & Access Control

### Identity duality: sub vs email keys break silently when scopes don't align

- **Rule:** telemetry and artifacts key on Keycloak `sub` (user identity, always present after auth). Access grants + admin UI key on lowercased email (stable, indexable, user-visible). Canonical mapping = `user_access.kc_sub` (the single source of truth for email↔sub). Never key telemetry on email; email is a display-only join. Never assume a logged-in user's email is in the DB — invited-not-yet-logged-in users may have no `user_access` row, so email lookups can silently match nothing.
- **Why:** developer sandbox had artifacts stored by `sub` (Keycloak subject from token) but access grants keyed by email. Admin panel filters/counts on email, activity aggregator resolves email→sub via `user_access.kc_sub` before any sub-keyed read. During dev iteration before full seeding, a user was logged in (had a `sub` from Keycloak) but had no `user_access` row. Queries like `SELECT * FROM activity_events WHERE actor_sub = ?` (keyed on the logged-in sub) ran fine. But `SELECT email FROM user_access WHERE kc_sub = ?` (resolving the reverse mapping for display) returned NULL. The aggregator's email-to-sub resolver hit a miss, count routes returned 0 events for a user who had clearly acted, and the admin panel showed "0 queries" next to a user name. The bug only surfaced in the aggregator; the events spine itself was consistent (sub-keyed throughout). The fix is a prerequisite check: resolve email→sub via `user_access` BEFORE any sub-keyed read; if the mapping doesn't exist, the user either isn't in the system yet (pending invite) or has a corrupted identity state.
- **Signal:** a logged-in user's count/filter returns 0 or NULL in the admin panel or activity dashboard even though direct sub-based queries (e.g. `SELECT COUNT(*) FROM activity_events WHERE actor_sub = ?`) show rows; the reverse mapping (email→sub lookup) fails silently; email and sub seem unrelated in telemetry.
- **Apply:** always resolve email→sub via `user_access.kc_sub` as a prerequisite before any activity/artifact read. The resolver `resolveEmailToSub(email: string): string | null` must check the DB; if null, the user either has no access row yet (pending invite, hasn't logged in) or their identity is incomplete. Telemetry primary key is sub; email is a display join. Any admin route that filters on email must resolve it first. For tests: always seed both the `user_access` row AND the artifact keyed on sub, never just the email. Verified by: activity aggregator routes return correct counts; email→sub round-trip works for every active user; pending invites are visibly "no activity" (not silent 0).

### Visibility NULL→personal flip: legacy segments lose visibility silently after filtering enforcement added

- **Rule:** when a schema adds a visibility/scope column to an existing table and enforces it server-side (filtering rows on a null-safe COALESCE), the filter behavior changes for **legacy rows with NULL**. Existing rows that pass the old filter (no filter applied) may fail the new filter (NULL → default value). Treat the behavior change as intentional + communicate it; owners re-grant visibility to see rows again. Never backfill NULL silently — it's a data-integrity decision that belongs in operations, not sneaky DDL.
- **Why:** segments table originally had no `visibility` column. LIST routes returned all rows (no filter). After adding `segments.visibility` (nullable) + enforcing `WHERE COALESCE(visibility, 'personal') IN (allowedList)`, legacy segments with NULL visibility became invisible to non-owners (NULL→personal→personal segments are only visible to owner). Teammates who relied on viewing shared segments suddenly saw them disappear. The UI showed no error, just an empty list — users suspected data loss. The behavior is actually correct (personal segments should be owner-only), but the silent flip was disruptive. Owners could re-grant via the visibility selector, but they didn't know to look for it.
- **Signal:** rows that previously listed for everyone stop appearing after a filter is added to a LIST query; user reports "my segment disappeared"; a schema migration adds a nullable enum column and servers apply a COALESCE default; before+after counts don't match for the same filter criteria.
- **Apply:** when shipping a scope/visibility gate that changes legacy-row behavior, (1) document the impact clearly in the changelog (legacy NULL rows become owner-only; owners re-share via the visibility selector). (2) Optionally, provide a one-time migration script so data owners can opt-in to backfill visibility=shared if they want. (3) Communicate to users: "Your previously-shared segments are now visible only to you; click the visibility icon to re-share them." The DATA MIGRATION is the owner's decision; the server code change is just the enforcement.

### Client-supplied identity header = no isolation: the chat path bypassed apiFetch and the proxy trusted it

- **Rule:** ownership/identity must be **server-authoritative** — derived from the verified JWT on the request (`request.owner = claims.sub`), never from a client-supplied header the FE happens to send. A header the browser controls (`X-Owner-Id`) is advertising, not authentication; treat it as a dev/legacy fallback only when no verified principal exists. Every FE call to an owner-scoped service must carry the app JWT — if a client uses raw `fetch` instead of the shared `apiFetch` wrapper, it silently drops `Authorization` and the server can't see who's calling.
- **Why:** chat sessions on `:11000` (real Keycloak) were visible to every user. Two faults compounded: (1) the chat FE clients (SSE + REST) used raw `fetch` and sent only `X-Owner-Id`, never `Authorization` — unlike `apiFetch`, which attaches the bearer. So the gateway's `authenticate` middleware saw no token → `request.owner = 'anonymous'`. (2) The chat proxy's `resolveOwner()` **preferred the client `X-Owner-Id` over `request.owner`**, and that header defaulted to the literal `'dev'` for everyone (nothing ever set it per-user on login). Net: every user's chat traffic resolved to owner `'dev'`, all sessions shared one owner, and the per-owner 403 check in chat-service was a no-op (`'dev' === 'dev'`). Invisible in local dev (`AUTH_DISABLED=true` → everyone is `'dev'` anyway); only surfaced once distinct Keycloak subjects hit the same stack. Fixed by (a) a shared `chatHeaders()` helper that attaches the bearer to every chat fetch, and (b) flipping `resolveOwner()` to return `request.owner` first, falling back to `X-Owner-Id` only when `request.owner === 'anonymous'`.
- **Signal:** a per-user resource is visible across users only on the real-auth stack, never in `AUTH_DISABLED` dev; the owning service has a correct-looking `owner_id !== caller` 403 check that never fires; the owner column is the same literal value (`dev`) on every row; the FE feature talks to the gateway via hand-rolled `fetch(...)` with a custom header set instead of the shared authenticated client. **Read-endpoint variant:** a list/registry endpoint that *fails open* to "no `req.user` → return everything" (e.g. `GET /api/workspaces` returns all workspaces for anonymous callers so the picker can render pre-login) silently returns the **unfiltered superset** when its FE fetch is tokenless — a grant-restricted user sees options they aren't granted (a prod-only user still sees `local` in the workspace switcher), and the per-user filter that *does* exist server-side never runs because there's no user to filter against.
- **Apply:** route every FE→gateway call through a helper that attaches `Authorization: Bearer <appToken>` (chat uses `src/api/chat-auth-headers.ts`; segments etc. use `apiFetch`). On the gateway, resolve owner from the verified principal first; client identity headers are a fallback for the no-JWT (dev/test) path only, never an override. When you add a new owner-scoped FE surface, grep that its fetches carry the bearer, and add a proxy test asserting the verified `request.owner` wins over a mismatched client `X-Owner-Id`. SSE is fine — chat streams over `fetch` (not `EventSource`), so custom headers work. The same trap hit the two bare `fetch('/api/workspaces')` calls in `workspace-context.tsx` + `use-game-context.ts` — both now attach the bearer via `readAppToken()` so the grant filter applies. Rule of thumb: any endpoint whose handler branches on `if (!req.user) return <everything>` MUST be fetched with the bearer from the FE, or the fail-open branch becomes the production behavior.

### Game-id vocabulary split: server allows, cube denies the same granted game

- **Rule:** cube-playground speaks gds.config game ids (`cfm_vn`, `jus_vn`); cube-dev's `checkAuth` speaks canonical ids (`cfm`, `jus`) and folds aliases (`GAME_ALIASES`) on the **requested** game BEFORE the `allowedGames` membership test. Any list cube-playground hands cube-dev (the `/internal/access` bridge) must be **canonicalized** to match — keep internal gds vocab everywhere else, translate only at that one boundary.
- **Why:** a viewer correctly granted `cfm_vn` saw "User … not allowed for game cfm_vn" on the metric trend. Server-side `userCanAccessGame(user, 'cfm_vn')` passed (compares gds id vs gds grant), so the request reached the cube — but the bridge returned the raw grant `['cfm_vn']`, while checkAuth folded the requested `cfm_vn`→`cfm` and tested `['cfm_vn'].includes('cfm')` → false → denial. The error message echoes the raw requested id (`cfm_vn`), masking that the comparison happened in canonical space. The admin/`'*'` path hid the bug: cube expands `'*'` to canonical `SUPPORTED_GAMES`, so admins (and grant-fallback `'*'`) never mismatched. Only a non-admin with a concrete grant on an aliased tenant (`cfm_vn`, `jus_vn`) triggered it. Fixed by folding the bridge's emitted `allowedGames` through the same alias map (`internal-access.ts` `GAME_ALIASES`, mirroring cube-dev).
- **Signal:** server (workspace-header) lets a game request through but the cube denies it with "not allowed for game `<id>`" naming a game the user *is* granted; the denial only hits non-admin users with concrete grants (admins/`'*'` work); the denied id carries a country suffix (`_vn`) that cube-dev aliases to a bare canonical id.
- **Apply:** when two services exchange entity ids that each normalize differently, canonicalize at the boundary adapter, not in core logic — and unit-test the adapter with an aliased id (`['cfm_vn'] → ['cfm']`), not just canonical ones, or the test locks in the bug (the original bridge test asserted raw `['…','cfm_vn']`). Keep cube-playground's `GAME_ALIASES` in sync with cube-dev's `cube.js`; a new aliased tenant must be added to both.

### Porting a prod Cube model into a local tenant: overwrite loses local-only members + imports dead columns

- **Rule:** when porting cubes from the upstream/prod Cube repo (kraken/cube) into a local per-game folder, **diff members and Trino-verify columns — never blind-overwrite**. The prod cube for a game is NOT a strict superset of the local one in either direction: (a) local often carries tenant-specific measures the prod cube lacks (CFM `active_daily` had `wau`/`trailing_wau`/`trailing_mau` + the `dau_by_country_payer_daily` rollup, `user_recharge_daily` had `trailing_wpu/mpu`, `mf_users` had `churn_risk`) — overwriting drops them and silently breaks presets/business-metrics that reference them; (b) prod cubes copied from a sibling-game template model **columns that don't exist in this tenant's raw table** (CFM `etl_ingame_recharge` has no `country_code`/`os_platform`/`role_*`/`is_first_recharge`/`money_type` — local had correctly omitted them; prod's cfm_recharge models them anyway and would fail at query time, latent because the dashboard never hits that view).
- **Why:** the cfm-user360 port replaced 4 overlapping core cubes wholesale at first; a member-name diff (local vs ported) surfaced 1 lost dim in `mf_users`, 4 in `active_daily` (incl. the WAU/trailing family), 2 in `user_recharge_daily`, and that local `recharge` was *richer* (16 SDK dims). Resolution: keep local core cubes as the base, merge in only the genuine prod additions (just `engagement_segment` for mf_users), and prune view `includes` that reference non-existent columns (12 pruned). A Trino column inventory (`SHOW TABLES`/`DESCRIBE` per schema) is the ground truth, not the prod YAML's comments.
- **Signal:** a ported cube compiles green and `/meta` looks complete, but a preset/business-metric that worked before returns "member not found", or a view query 500s at Trino with "column does not exist" only for some tenants; prod YAML comments claim a column the live `DESCRIBE` doesn't show; member counts differ between the local file you replaced and the prod file.
- **Apply:** before overwriting, snapshot local member names (`grep '- name:'`), generate the ported version, `comm` the two sets, and merge missing-but-used members back. Build a per-schema Trino table/column inventory first and gate every cube's `sql_table` + every dimension/measure column against it. Strip the `<game>_` cube prefix + `<schema>.` qualifier deterministically (a text-level transform, not a YAML round-trip, to preserve comments/anchors). A behavior-log guardrail (≤31-day bound on raw `etl_*` event cubes, 1M–1.3B rows) must land in `cube.js` BEFORE importing those cubes, and must fail **closed** (unknown/negative/reversed dateRange → reject). Adding a new tenant = `GAME_SCHEMA` entry + dev `auth-users` allowedGames (incl. the `playground` service account, or the app can't query it) + a model folder.

### Chat query-artifact chart is decoupled from the query — LLM-authored data + free-text labels drift from the real members

- **Rule:** the inline chart on a chat `query_artifact` is **not derived from the linked Cube query** — it's a separate LLM-authored `ChartSpec` (rows + `encoding {category,value,series?}` + free-text column names). The query (all measures/dimensions/filters) only travels in the deeplink. So the chart can show fewer columns than the query and mislabel them. Never trust the chart's column names for display; resolve labels from `/meta`. Chart rows are keyed by Cube **member name** (the LLM pastes `preview_cube_query` output verbatim, and Cube `/load` keys rows by `cube.member`), which is the hook that lets the server attach a deterministic `columns[]` descriptor (label/dataType/kind) per column.
- **Why:** a whales query (`mf_users.ltv_total_vnd` + 5 dims, limit 100) rendered a single bar series labelled "revenue" — wrong on two counts: the metric is lifetime value not revenue, and the 5 dimensions were invisible. The label was LLM prose (`encoding.value` named "revenue"), and the chart only encoded 1 measure because a `bar`/`line` encoding structurally holds at most category+value(+series). The fix: server resolves every chart column key against `/meta` (`resolveMemberMeta`) into a `ChartColumn{key,label,dataType,kind}`, the UI renders table headers + axis/tooltip/legend text from that map (never the raw key or LLM name), the card defaults to the **table** for table-shaped results (`preferTableView`: >12 rows or ≥4 cols), and a manual X/Y/series picker (+ LLM picking the 2 most-relevant columns) drives a focused 2-column chart over the same ≤50 preview rows.
- **Signal:** a chat chart's series/axis is labelled with a friendly-but-wrong word (or a raw `cube.member` key), shows only one measure when the query has many dimensions, or the metric name in the chart doesn't match any member in the query; the data table view header reads `mf_users.ltv_total_vnd` instead of "Total LTV (VND)".
- **Apply:** keep chart rows keyed by member name (paste preview rows verbatim, all columns). Attach `columns[]` from `/meta` in `emit_query_artifact` and mirror the type on the FE `ChartArtifact`. Resolve all display text through `chart-column-labels.labelOf(buildLabelMap(columns), key)` with a humanised-leaf fallback for assistant-derived columns not in meta. When adding a new chart-rendering surface, thread the label map into headers/axes/tooltips/legend — a raw member key leaking into the UI is the smell. The chart is a *preview* (≤50 rows); the full result set is the deeplink → Playground, not the chart.

### Chat agent thrashes resolving members by grepping /meta → burns the turn budget and times out

- **Rule:** the explore agent must resolve natural-language terms → physical Cube members through `resolve_query_terms` (glossary over live-`/meta` fuzzy search) and filter-value casing through `list_dimension_values`, NOT by fetching the full `/meta` blob and eyeballing it. The disambiguator only pins the *metric* + core slots; when the agent augments that query with extra dimensions/filters it has no resolver unless it calls these tools. `get_cube_meta` is a fallback dump, not the first step.
- **Why:** a whale-leaderboard turn ran the full 120s per-turn timeout and was killed with no answer (`stop_reason` NULL, 0/0 tokens). The reasoning trace was pure trial-and-error: guessing `uid`→`user_id`, `time.event_date`→`recharge.recharge_date`, and `whale` vs `Whale`, against a single-line meta blob it complained it "can't Grep". Each guess was a separate round-trip; the clock ran out before `preview_cube_query`. The disambiguator had already returned `action:auto`, but the agent rebuilt the query by hand to add user-level dimensions the resolver hadn't pinned.
- **Signal:** a turn's reasoning is a long sequence of member-name and value-casing guesses; `get_cube_meta(full)` fetched then visually scanned; the turn ends with empty assistant text + `stop_reason='timeout'` (now visible — previously NULL/"legacy"); latency pinned near the per-turn timeout.
- **Apply:** route resolution through `resolve_query_terms` (returns member ref + kind + dataType + label + confidence per term, one call — resolve all needed dimensions/filters/time fields in one batch) and `list_dimension_values` (exact enum casing, capped + truncated flag). When adding a new query-building skill or agent loop, give it a member resolver before it can reach for raw meta. Separately: a timeout-abort ends the SDK stream with no `result` message, so persist the abort reason as `stop_reason` explicitly or the failure hides as a NULL "legacy" row and skews successRate.
- **Variant (2026-06-05, cfm_vn):** same timeout, different trigger — the agent legitimately needed cube *structure* (which cube has which measures/segments), but `get_cube_meta(compact)` on a member-rich game (cfm_vn: 24 cubes, ~95k chars) exceeded the Agent SDK's ~25k-token MCP output cap. The SDK dumps oversized results to a file; the agent then burned ~47s grepping a one-line JSON file, spawning an extraction subagent, and failing `python3` (exit 127 — not on the Monitor PATH) before the 120s `CHAT_TURN_TIMEOUT_MS` killed the turn. Fix: `get_cube_meta` now takes `cubes: [names]` for per-cube full detail (descriptions + segments, prefix-tolerant matching for prod-prefixed cube names) and degrades the unfiltered compact response to a name+count index when over a 60k-char budget. General rule: **any MCP tool whose response scales with schema/data size needs an explicit size budget and a narrowing parameter** — the SDK file-dump fallback is a turn-killer, never an acceptable path.

### Agent hunts for "where the data ends" with repeated empty previews → burns the turn on a freshness question

- **Rule:** when a Cube query for a recent date range returns 0 rows, the question is usually "how far behind is the pipeline?", not "is my query wrong?" — answer it with ONE `get_time_coverage` call (server-side walks 31-day windows backwards, returns the latest date with data), never with speculative re-previews on shifted ranges. The ≤31-day bound guard on behavior cubes makes blind probing actively pathological: when data lags more than a month, *every* "recent" probe the guard permits is guaranteed empty.
- **Why:** a currency-outflow turn (`etl_money_flow`, 1.35B rows) asked for "this month" (June) when the cube's latest partition was April. The agent ran ~8–10 `preview_cube_query` probes — June, `last_30d`/`last_7d` segments, May, then April — each a real Trino scan, and only escaped because a meta description happened to mention "verified Apr–May". It found the right answer but was killed by the 240s turn timeout while refining it. ~70% of the budget went to freshness archaeology.
- **Signal:** a turn's reasoning contains a chain of "Still empty. Let me try…" date-range mutations; multiple previews differ only in `dateRange`/segment; the cube has a bounded-date guard and the user asked about "this month"/"this week"; `stop_reason='timeout'` with the analysis nearly complete.
- **Apply:** route freshness through `get_time_coverage({member: <time dim>})` → re-anchor the range to `latestDate` → disclose "data through <date>" in the answer. The probe pattern (order-desc, limit-1, bounded windows) is cheap and guard-compatible. When building any agent that queries partitioned/lagging data, give it a freshness primitive before it ships — "empty result" is ambiguous between *no data ever*, *wrong query*, and *pipeline lag*, and an agent without the primitive will pay N queries to disambiguate what one targeted probe answers.

### Saved-defaults memory fill flips clarify→auto with last session's metric on a brand-new question

- **Rule:** cross-session/session memory may gap-fill *refinement* slots (timeRange, dimension, filters) freely, but must NOT fill *topic* slots (metric, intent, concept, entity) when the current message carries substantial unresolved text (≥3-word span the resolvers couldn't account for) — that text is probably the subject the user actually asked about. Filling the metric there deletes the metric clarification and upgrades `clarify`→`auto` with an unrelated saved metric.
- **Why:** "top currency outflow reasons by total amount spent" — no glossary term matched, so the per-turn metric slot was empty; the L3 user-prefs fill injected the user's habitual `active_daily.dau` at 0.92, `dropResolvedClarifications` removed the metric question, and the action flipped to `auto`. The skill's hard rule ("never second-guess `auto`") then *mandates* charting DAU for a money-sink question. The agent only recovered because it disobeyed the rule. The legit fill case — short slot-replies like "by country"/"theo quốc gia"/"last month" after a clarify — leaves at most tiny unresolved fragments, which is the discriminator.
- **Signal:** `disambiguate_query` returns `action:'auto'` whose pinned metric is semantically unrelated to the question; `warnings[]` contains "resolved from your saved defaults"/"resolved from session memory" while `unresolved[]` holds most of the message; agent reasoning starts with "the resolver returned X but the user wants Y".
- **Apply:** any memory/preference layer that back-fills slots into a resolver result needs a topic gate: classify slots as topic-bearing vs refining, and block topic fills when the message has unaccounted content (`hasSubstantialUnresolvedText` in disambiguate-memory-merge.ts). More generally — a "helpful default" that *suppresses a clarification* is an auto-routing decision, not a default; it must clear the same confidence bar as a real resolution of *this* message, not of last week's.

### Streaming chat re-renders the whole thread per token → UI (and the Stop button) freezes on long turns

- **Rule:** any component rendered once-per-message inside the chat thread MUST be memoized (`React.memo`) with referentially-stable props for committed turns. The stream store applies every SSE `token`/`thinking` delta as a synchronous `set()`, so the thread re-renders at token rate; without memo, every committed turn re-parses its full ReactMarkdown on every token — per-token cost grows with conversation length until the main thread saturates and input events (e.g. "Stop generating") starve.
- **Why:** on prod, a long research turn (10+ MCP tool calls, multi-turn history) made the Stop button unresponsive. `ChatMessageList` mapped all messages into un-memoized `AssistantMessage`/`UserMessage`; each parent re-render re-ran `mergeToolSections` + ReactMarkdown for every committed turn. Memoizing both components confines per-token work to the live `__streaming__` message (its `sections` array rebuilds each render, so memo correctly lets it through).
- **Signal:** chat UI gets progressively laggier as a turn streams (worse with more history); clicking Stop does nothing; DevTools performance trace shows long render tasks dominated by markdown parsing of unchanged messages.
- **Apply:** keep committed-message props shallow-stable: append-only `setCommittedMessages((prev) => [...prev, msg])` (preserves element refs), `useCallback` for chip handlers, flags like `showFollowups` derived so they don't flap during streaming. New per-message components (badges, chips, cards) inherit the same constraint — if a prop is rebuilt every render for committed turns, the memo silently dies. Regression guard: `chat-message-list-streaming-memo.test.tsx` counts markdown parses across a simulated token re-render.

### Background job resolves Cube schema without the artifact's workspace/game ctx → per-game cubes "don't exist"

- **Rule:** any server-side code that introspects Cube `/meta` (or compiles probe queries) on behalf of a stored artifact (segment, metric, card) MUST build the introspection ctx from the artifact's OWN `(workspace, game_id)` — never call the ctx-less suggester/meta helpers. Per-game cubes (event-level `etl_*`) exist only in that game's schema, and join-probe identity only compiles under that game's token; the ctx-less path silently introspects the DEFAULT game and reports "cube not found / no identity".
- **Why:** segment `dat bom` (cube `etl_game_detail`, game `cfm_vn`) was created fine on the Build page (route passes `introspectionCtx(req)`) but every refresh tick marked it `broken: no identity-field mapping` — `resolveIdentityField` called `suggestIdentities()` with no ctx, so the suggester saw the default game's meta where `etl_game_detail` doesn't exist. Second half of the same bug: refresh required confidence ≥0.9 while creation accepts ANY non-null suggestion — join-probe results (0.7) passed creation and failed refresh by construction.
- **Signal:** an artifact works on the page that created it but a cron/background job marks it broken with a "missing cube/member/identity" reason; the cube is per-game (`etl_*`, join-probe identity); the broken reason disappears when you hit the equivalent API route with the right `x-cube-game` header.
- **Apply:** in job code, build the ctx like `resolve-identity-field.ts#buildIntrospectionCtx` (service principal, segment's workspace + game, default-game fallback for game_id workspaces) and pass it through to `suggestIdentities`/`getMetaWithCtx`. Keep acceptance criteria in lock-step across the artifact lifecycle: whatever the creation surface accepts (any non-null suggestion), refresh/validation must accept too, or artifacts create-then-break. When testing code on this path, mock BOTH `getMeta` and `getMetaWithCtx`.

### Swallowed dependency failure degrades to *wrong-looking* behavior instead of an honest error (meta outage → off-topic clarify)

- **Rule:** when a resolver/pipeline silently skips a stage because a dependency fetch failed (`catch { x = undefined }`), the skip MUST leave a trace in the result (warning + flag) — and any user-facing fallback produced in that degraded state must say "dependency unavailable, retry", not pretend the input was ambiguous. A catch that only nulls a variable converts an infra outage into a phantom logic bug.
- **Why:** same starter-question chip auto-resolved at 17:17 but got a canned "Which metric should I show? Revenue/ARPU/…" at 17:41 (session 4929a3e9). Cube `/meta` was unreachable for ~4 min mid dev-stack restart; `disambiguate_query` caught the error, set `knownMembers = undefined`, the starter pass-through hit its `if (!meta || !knownMembers) return null` guard, and the glossary engine (which can't see raw `etl_*` members) produced an off-topic metric menu — zero trace of the real cause anywhere in the turn.
- **Signal:** identical input behaves differently minutes apart, around a service restart; the failing turn's output contains *defaults* (canned options, generic clarify) where the working turn had specifics; no error/warning anywhere despite the behavior flip — the absence of a trace is itself the signal that some `catch` ate the cause.
- **Apply:** `disambiguate-query.ts` now records `metaError`, always pushes a `cube meta unavailable (…)` warning, returns `metaUnavailable: true`, and when the metric is also unresolved replaces the canned menu with a bilingual "data model temporarily unavailable, try again" clarification (no chips). New degraded-mode code should follow the same shape: catch → record reason → annotate result → make the fallback honest. Regression guard: `test/tools/disambiguate-meta-unavailable.test.ts`.

### Bounding the FIRST time dimension ≠ bounding the PARTITION column — guarded cubes 500 anyway

- **Rule:** when programmatically adding a `timeDimensions` bound to satisfy a behavior-cube guard ("must bound log_date/dteventtime within 31 days"), pick the cube's PARTITION time dimension (`log_date`/`dteventtime`), not `dimensions.find(type === 'time')`. A cube with several time dims (cros `etl_register`: `register_time` first, `log_date` second) gets a bounded-but-wrong-column query that the guard still rejects.
- **Why:** every cros etl_* starter candidate failed tier-1 with `query-error` while cfm_vn etl cubes sailed through — cfm's etl cubes expose exactly one time dim (`log_date`), so "first" and "partition" coincided by luck. The shared picker (`disambiguate-starter-passthrough.ts#timeDimensionOf`) also builds the RUNTIME clicked-chip query, so shipped chips on multi-time-dim cubes would have 500'd in prod.
- **Signal:** queries that DO carry a `timeDimensions.dateRange` still get the "must bound …" 500, only on some games' event cubes; the failing cubes list 2+ `type: time` dimensions in `/meta` with the partition column not first.
- **Apply:** `timeDimensionOf` prefers `/\.(log_date|dteventtime)$/` among time dims, fallback first. Regression guard: "bounds the PARTITION time dimension" test in `test/tools/disambiguate-starter-passthrough.test.ts`. Any new query-shaping code (probes, draft scaffolds, card runners) must reuse this picker rather than re-deriving "first time dim".

### Trino DATE column declared `type: time` in Cube YAML → every time-dim select 400s ("actual date")

- **Rule:** a Cube time dimension whose physical Trino column is DATE must wrap it: `sql: "CAST({CUBE}.col AS TIMESTAMP)"`. Cube's timezone conversion wraps time dims in `AT TIME ZONE`, which Trino rejects on bare dates with `Type of value must be a time or timestamp with or without time zone (actual date)`.
- **Why:** all cros coverage probes on `etl_login/etl_logout/etl_register.log_date` and `user_active_monthly.log_month` 400'd with that message while the same YAML shape worked on cfm (whose physical columns are timestamp-compatible). The YAML is not enough to spot it — same declaration, different upstream column type. Repo precedent: `tf/user_recharge_daily.yml` already carried the cast.
- **Signal:** `/load` 400 `line N:M: Type of value must be a time or timestamp… (actual date)` only when the time dimension is SELECTED (probes, granularity queries); pure `dateRange` filters may still pass. Per-game: same cube name fine in one game, broken in another.
- **Apply:** cast in the dimension SQL (cros etl_login/etl_logout/etl_register `log_date`, user_active_monthly `log_month`). When adding/onboarding game cubes, probe one time-dim SELECT per cube before shipping; don't infer health from a sibling game's identical YAML. Note: the dev "local" workspace cube (:4000, `cube-playground-cube-api-dev` container) caches the compiled schema — restart that container (not `cube-playground-cube-api` on :17001) for model edits to take effect.

### Refresh worker pegs CPU → container "unhealthy" without crashing (missing-source pre-agg, then sheer partition volume)

- **Rule:** the refresh scheduler's sweep cost is O(total partitions) of CPU-bound SQL planning every `CUBEJS_SCHEDULED_REFRESH_TIMER` (default 30s) — independent of whether anything needs building. Two ways to starve the event loop: (a) a pre-agg whose source table doesn't exist for that game retries forever, (b) enough partitions (~600: 20 pre-aggs × monthly from 2024-01-01 × 6 games) that one sweep takes longer than the sweep interval. Either way the HTTP server stops accepting connections — healthcheck `fetch /readyz` times out with the accept backlog full (512) and hundreds of CLOSE_WAIT sockets.
- **Why:** ptg `marketing_cost.yml` referenced `std_marketing_cost_all_channels_by_game` (56 game schemas have it; ptg has only `etl_*` tables) — worker ran 99.6% CPU, 1,909 consecutive healthcheck timeouts (~19h). Deleting it dropped sweeps from 18min to ~2min, but 2min sweeps on a 30s timer still re-saturated within minutes of "healthy": fixing the loudest error ≠ fixing the load. "Unhealthy" was starvation, not a crash, so nothing restarted or alerted either time.
- **Signal:** container `(unhealthy)` with healthcheck `ExitCode: -1` / "exceeded timeout" (not 4xx/5xx); in-container `fetch 127.0.0.1:4000` gets `UND_ERR_CONNECT_TIMEOUT` despite port LISTENing; logs repeat `Refresh Scheduler Interval … was not finished` every cycle — with a `Table '…' does not exist` error for case (a), or near-silent logs at 100% CPU for case (b).
- **Apply:** (a) deleted `cube-dev/cube/model/cubes/ptg/marketing_cost.yml`; before adding a pre-agg to a scaffolded per-game cube, verify the source exists in THAT game's schema (`SELECT table_schema FROM game_integration.information_schema.tables WHERE table_name = '…'`). (b) `CUBEJS_SCHEDULED_REFRESH_TIMER=300` in both compose files + `build_range_start` 2024→2025 + every rollup converted to `<name>_batch` + `rollup_lambda` union (fresh tail served live from Trino). Timer gotcha: `CUBEJS_REFRESH_WORKER=true` SHORT-CIRCUITS Cube's `refreshWorkerMode` env chain (`shared/dist/src/env.js`) — with the boolean set, the numeric timer is never read and the interval silently stays 30s ("was not finished with 30000 interval" keeps logging). Set ONLY the numeric timer: it both enables worker mode and carries the interval. Recreate (not just restart) the worker for env changes — and recreate via `npm run stack -- up -d --no-deps cube_refresh_worker`, NOT raw `docker compose -f …`: the env lives in `.env.docker.local` which only the stack runner passes, raw compose silently blanks `CUBEJS_DB_HOST` and every source query dies with node-fetch `Invalid URL`. Same trap hits the DEV pair (`cube_api_dev`/`cubestore_dev`, devcube compose file) — raw compose also drops the wrapper's `CUBESTORE_TAG=v1.6.46-arm64v8` arch pin, recreating cubestore on the amd64 `latest` image which Rosetta-wedges into a ZOMBIE: container `running`, port 3030 refused, **zero log lines ever** (`docker logs` empty is the tell; PID1 = `/mnt/lima-rosetta/rosetta ./cubestored`). ### cubejs/cube itself can run amd64-under-Rosetta on Apple Silicon — 15× slower sweeps, plus boot zombies

- **Rule:** `cubejs/cube:latest` IS multi-arch, but the local image cache can hold only the amd64 variant (pulled before the arm64 manifest existed, or under a forced platform). Compose then silently runs the worker under Rosetta — `docker compose up` warns "requested image's platform (linux/amd64) does not match" and PID1 becomes `/mnt/lima-rosetta/rosetta /usr/local/bin/node …`. Emulation multiplies every CPU-bound sweep: per-tenant schema compile measured 5–35 s under Rosetta vs 0.7–2.4 s native; a full 8-tenant sweep 3.2 min vs ~13 s. On a 300 s timer the native sweep leaves the event loop free ~96% of the time; the emulated one starves `/readyz` through most of every sweep.
- **Why:** the prod-stack refresh worker sat `(unhealthy)` at ~100% CPU with 33 consecutive healthcheck timeouts, ONE log line in 26 min, and zero CubeStore writes — the 300 s timer + lambda-rollup fixes were already in place and didn't help. Two failure modes were observed on the same amd64 image across recreates: (1) full zombie — `running`, 0% CPU, zero log lines ever (same shape as the cubestore zombie above); (2) sweep saturation — logs only "listening", then 100% CPU forever: the scheduler IS sweeping (silently — see the log-level lesson below), but Rosetta stretches each sweep until consecutive sweeps merge into one continuous event-loop block (so tight even SIGUSR1's inspector never starts). The serving `cube_api` died the same way an hour later: a games-readiness `/load` burst (the known fan-out culprit, see "wedged in-stack Cube" above) wedged it permanently at 100% CPU / 98-probe failing streak, where a native cube absorbs the same burst fine. Switching the same version to native arm64 fixed all of it.
- **Signal:** compose prints the platform-mismatch warning on recreate; `cat /proc/1/cmdline` in the container starts with `/mnt/lima-rosetta/rosetta`; "Compiling schema completed" durations in the tens of seconds; healthcheck `ExitCode: -1` timeouts with no errors in logs.
- **Apply:** pull the native variant of the SAME version by digest (no drift): `docker pull --platform linux/arm64 cubejs/cube@sha256:<index-digest>` (the local image ID under containerd IS the index digest), then recreate via `STACK_PLATFORM=linux/arm64 npm run stack -- up -d --force-recreate --no-deps cube_refresh_worker` (same for `cube_api`). Verify with `cat /proc/1/comm` → plain `node`. Prod is untouched — its runner is amd64-native; this is purely a local image-cache problem.

### A healthy refresh worker is SILENT at info log level — don't read quiet logs as "scheduler not running"

- **Rule:** Cube's logger filters by internal message class, not the `"level"` field serialized into the JSON line. "Refresh Scheduler Run", "Compiling schema", and "Query started/completed" are trace-class: at `CUBEJS_LOG_LEVEL=info` a perfectly working refresh worker logs NOTHING after the boot "listening" line — sweep after sweep, build after build, in total silence. Only the first sweep compiles schemas (compiler LRU cache); warm sweeps are cheap I/O, so even CPU sampling can miss them.
- **Why:** chasing the Rosetta wedge above, an info-level worker showed zero scheduler lines and an 11%-max CPU over 6 min and was nearly declared broken ("trace→sweeps, info→never sweeps" fit four containers in a row). Cube source disproved it: `startScheduledRefreshTimer()` runs unconditionally in the server-core constructor, and `CUBEJS_SCHEDULED_REFRESH_TIMER=300` parses to 300 s regardless of log level. CubeStore's ingestion log settled it — a fresh `TableImportCSV("temp://prod_pre_aggregations.…_batch…")` job landed mid-"silence".
- **Signal:** worker logs contain only "🚀 Cube API server … listening" yet the container is healthy. That is the NORMAL healthy state, not a stalled scheduler.
- **Apply:** to verify sweeps, don't grep worker logs at info — check CubeStore for ingestion jobs (`docker logs <cubestore> | grep TableImportCSV`; fresh `prod_pre_aggregations.*_batch…` imports prove builds) or temporarily recreate the worker with `CUBEJS_LOG_LEVEL=trace` to watch the scheduler directly. Corollary for the saturation lesson above: "near-silent logs at 100% CPU" does not mean the scheduler never started — at info level it was sweeping the whole time, invisibly.

### One bundled query per UI surface = one guarded measure blanks EVERYTHING (Members-tab enrichment)

- **Rule:** never bundle a behavior-cube measure (date-bound-guarded `etl_*`) into the same Cube query as identity-cube (`mf_users`) columns. The guard rejects at COMPILE time, so the whole query 400s and every column backed by it renders `—` — which reads as a permission bug ("admin sees only 3 of 6 columns") when it's a query-shape bug.
- **Why:** the tiered Members tab fetched `mf_users.lifecycle_stage` + `last_active_date` + `etl_game_detail.matches` in ONE enrichment query; the cfm model's "must bound log_date/dteventtime within 31 days" guard killed it, blanking Stage/Last active alongside Matches. Any preset memberColumn on an event cube MUST carry `boundTimeDimension` (and a window-suffixed label, e.g. "Matches (30d)"). ### A cube with no time dimension can never be bounded — every query is a full-history scan that 504s

- **Rule:** any cube whose SQL UNIONs raw event tables (or otherwise scans partitioned history) MUST expose a `type: time` dimension AND push it into the scan via `FILTER_PARAMS` — applied to **both** the event timestamp and the partition column (`log_date`), or Trino reads every partition regardless of the query's date range. A cube without a time dimension cannot even *receive* a date bound: callers physically can't write a bounded query, and it silently bypasses the cube.js behavior-bound guard (which keys off cube names / time-dim fields).
- **Why:** `ordered_event_funnel` — the cube purpose-built for funnel questions — was a UNION of 4 `etl_ingame_*` tables with `ROW_NUMBER()` over all history and only a `step_name` FILTER_PARAMS. Every `/load` exceeded the gateway's 15s budget (504), in all 6 games, always. A chat turn asking exactly the question the cube was built for burned ~50s on funnel 504s, fell back to hand-assembling the funnel from etl_* cubes (hitting the 31-day guard twice more), and was killed by the 240s turn timeout with the full answer sitting unwritten in its reasoning. The FE funnel-builder was *also* broken — `run-funnel.ts` queries `<cube>.ts` per the template convention, but the YAMLs never exposed `ts`.
- **Signal:** a specific cube 504s with "Cube request timed out after 15s" on even trivial measures while sibling cubes answer fine; the cube's `/meta` entry lists no `type: time` dimension; the behavior-bound guard never fires for it despite it fronting `etl_*` tables.
- **Apply:** add the `ts` time dimension + double FILTER_PARAMS pushdown (`.ts.filter('ts')` and `.ts.filter('log_date')`) per `docs/ordered-funnel-cube-template.md`; register the cube in `BEHAVIOR_VIEWS` and its time field in `TIME_DIM_FIELDS` in `cube-dev/cube/cube.js` so unbounded queries fail in ~7ms with an actionable message instead of a 15s 504. Verified May-2026 bounded funnel: cold 26s (first compile + Trino), warm 0.1s. Audit any new cube wrapping event tables for the same hole.

### A timed-out agent turn used to discard minutes of correct work — salvage instead of blanking

- **Rule:** when the per-turn timeout aborts the SDK stream, `assistantText` is empty (the `result` message never arrives) even though reasoning + tool results captured real findings. Salvaged turns are never response-cached (write gate requires `end_turn`); user cancels are never salvaged.

### An early-return tool path that skips its memory write makes every follow-up context-blind

- **Rule:** any short-circuit branch in a tool that normally persists session state (here: `disambiguate_query`'s starter-question pass-through vs `writeMemoryFromResult`) must leave the SAME state trail as the normal path before returning. A fast path that skips the write doesn't fail on ITS turn — it fails on the NEXT turn, far from the cause.

### Recharts callback payload shapes differ per chart family — Funnel names tooltip items by nameKey→row-index, not dataKey

- **Rule:** a recharts `Tooltip`/`Legend`/`LabelList` callback that works for one chart family is NOT safe to share with another — the `name`/`value` it receives is family-specific. Cartesian charts (Bar/Line/Area) pass the series `dataKey` string as `name`; `Funnel` passes `getValueByDataKey(entry, nameKey, index)` — i.e. the `nameKey` column value, falling back to the **numeric row index** when the rows have no such column (default `nameKey` is `'name'`). Any shared formatter that assumes a string column key (and calls string methods on it) crashes only on the funnel.
- **Why:** the chat query-artifact funnel renderer reused the shared `tooltipFormatter` (`labelOf(labels, name)` → `humaniseMember` → `key.includes('.')`). Funnel rows are keyed by Cube member names with no `name` column, so recharts handed the formatter `name = 0` and hovering threw `TypeError: e.includes is not a function` inside the error boundary ("Error occured while rendering", June-2026). Every other chart type rendered the same artifact fine, which misdirected the diagnosis toward the data.
- **Signal:** a chart-type switch crashes for exactly one family while the same spec/data renders everywhere else; a minified stack shows a string/array method failing inside a tooltip/label render frame (`*.includes is not a function`); the failing chart is one of the "shape-special" recharts families (Funnel, Pie, Radar, Treemap) sharing a formatter written for cartesian charts.
- **Apply:** give shape-special charts their own formatter and set `nameKey` explicitly (`<Funnel nameKey={spec.encoding.category}>` in `assistant-chart-section.tsx`) so the tooltip names entries by the step label; harden shared label resolvers to accept `string | number` and coerce (`labelOf` in `chart-column-labels.ts`). When wiring a new recharts family onto a shared formatter, log/assert what `name` actually arrives as for THAT family before shipping — don't extrapolate from Bar/Line.

### Ordering a list by a completion-time column hides rows that haven't completed yet

- **Rule:** any `ORDER BY <nullable-completion-timestamp> DESC ... LIMIT n` query must `COALESCE` the sort key to a creation timestamp. SQLite sorts NULL as the smallest value, so DESC puts in-flight rows LAST — past the LIMIT they vanish entirely. Rows are invisible exactly during the window the user most expects to see them (right after creating them).
- **Why:** `listSessions` ordered by `last_turn_at DESC LIMIT 20`; `last_turn_at` is only stamped when the first turn COMPLETES (`incrementTurnCount`), and first turns can run minutes. A user with 27 sessions started a chat, navigated away mid-turn, and the new session was unfindable in the sidebar through any number of refreshes (June-2026) — it "appeared after a while" only because the turn finally finished. A first turn that errors before completion left the session buried forever. Looked like an ownership/refresh bug; the FE refetch machinery (`gds-cube:chat-session-changed`, `cache: no-store`) was all working — the SERVER was excluding the row.
- **Signal:** "my new X doesn't show up in the list until much later / after refreshing"; the row exists in the DB with the sort-key column NULL; list endpoint has `LIMIT`; FE refetch events fire correctly.
- **Apply:** `ORDER BY COALESCE(last_turn_at, created_at) DESC` in `listSessions` + `listSharedSessions` (`chat-store.ts`). When adding any list endpoint sorted by an activity timestamp, write the test "just-created row with NULL activity + LIMIT-worth of older rows → new row is first".

### A new Cube view YAML is invisible until the dev container restarts — `/meta` still 200s without it

The dev Cube (`cube-playground-cube-api-dev`, :4000) mounts the VENDORED `cube-playground/cube-dev/cube` dir read-only (`docker-compose.devcube.yml` → `./cube-dev/cube:/cube/conf:ro`) — NOT the sibling `cube-dev-old`. - **Why:** enabling jus_vn Member 360 added `views/jus/user_360.yml`, but the member page 400'd `Cube 'user_recharge_timeline' not found` (all 4 jus 360 views, not just one). The container had been up 24h — it compiled before the file existed. - **Signal:** a query errors `Cube '<view>' not found for path '<view>.<field>'` for a view you just authored; `/meta` returns 200 but a JWT-authed `/meta` dump doesn't list the view's cube; the Cube container's uptime predates the file's mtime; editing the sibling `cube-dev-old` instead of the vendored `cube-dev/cube` (wrong dir entirely) produces the same "not found".
- **Apply:** confirm the live mount with `docker inspect <cube-container> --format '{{range .Mounts}}…'` before debugging schema; restart the container (or let the `dev:all` watchdog recreate it) after model edits; verify with a minted-JWT `/meta` grep for the new view, not just an HTTP 200. The `member360-coverage` surface (admin Data-coverage tab) now makes this visible — a just-added-but-unloaded view shows `blocked` until the reload.

### A URL-encoded route param fed straight into a Cube equals-filter matches zero rows for ids with special chars

- **Rule:** react-router v5 `useParams()` returns route params **raw / still-percent-encoded**. Any param that came from an `encodeURIComponent(...)` link and is then used as a Cube filter value (or any exact-match key) MUST be `decodeURIComponent`'d first. A value containing `@`, `+`, `/`, `#` arrives as `%40`/`%2B`/… and `equals` against the stored literal never matches.
- **Why:** the per-member 360 page (`member-360-view.tsx`) filtered `user_profile.user_id equals <uid>` with the raw param. The Members-tab link correctly `encodeURIComponent`s the uid; vopenid uids (`<n>@vng_vie.win.163.com`, jus_vn / cfm) become `<n>%40...` in the hash URL, and the page sent that literal to Cube → every panel rendered `—` / "No data" (June-2026). It hid for a long time because ballistar/cfm uids in earlier testing were plain numerics (no escape chars), so the bug only bit once a game with `@`-bearing uids (jus_vn) got a 360. Cube returned HTTP 200 with `data:[]`, and the coverage probe (unfiltered) showed the view as `ready` with rows — so "data exists but this member shows nothing" pointed at the filter value, not the model.
- **Signal:** entity detail page is all blanks while the cohort/coverage view confirms the table has rows; the Cube `/load` request-log shows the filter value with a `%40`/`%2B`/`%2540` (double-encoded in the querystring) where you expect a literal `@`/`+`; only ids containing special characters are affected, numeric-only ids work. Note: external `curl` against the `/cube-api` proxy without the FE session returns empty `data` for everything — don't use it to "confirm no data".
- **Apply:** `decodeURIComponent` the uid once at the top of the component (guard with try/catch → fall back to raw on malformed) and use the decoded value for both the live query and the precompute cache key. Fastify path params are already decoded server-side, so the cache route is fine — the gap is FE-side only. When a new entity page keys off a `useParams` id, decode before any equality use.

### Selecting a Cube measure as a dimension 400s — and the same view name has a different grain per game

Any code that picks "a member to query" from a panel's column list must carry the member's `kind` and branch the query shape. (b) The **same logical view name does not have the same grain across games** — verify each game's columns against that game's actual cube, don't reuse another game's panel definition wholesale.
- **Why:** rolling Member 360 out to cros/tf, the data-readiness probe built `{ dimensions: [member] }` for whatever the panel's first column was. cros/tf device/IP panels are per-user **aggregate rollups** whose only useful columns are `count_distinct` measures (`distinct_devices`, `distinct_ips`) — so the probe sent a measure in `dimensions` and the panel classified `error` (400). ### A logical-member Cube probe that fails safe to `now()` silently mislabels a stale mart as fresh on a prefix workspace

- **Rule:** any `/load` helper that probes a member directly (e.g. MAX-of-time-dim freshness) must **physicalize** the member (`physicalMember(member, gamePrefix)`) before the call — `loadWithCtx`/`cubePost` post the body verbatim, they do NOT translate logical→physical. On a prod prefix workspace every probe would 404 → fall back to today → stamp the weeks-stale gameplay mart "as of today", the exact misread the feature exists to prevent (June-2026). - **Signal:** a feature that "works" only on the local game_id workspace; a fail-safe default (today / 0 / empty) that masks a probe error rather than surfacing it; a `/load` for a logical member name on a workspace whose `/meta` only lists prefixed cubes; green local tests with `gamePrefix: null` and no prefix-workspace case.
- **Apply:** add a prefix-workspace unit test that asserts the loader receives the **physical** member; treat any new direct `/load` as needing physicalization unless it's provably game_id-only; and when an error path defaults to a plausible value, ask "if this fires constantly on prod, does anyone notice?" — if not, log loudly or fail closed instead of defaulting.

### A column in `/meta` + schema exists ≠ populated — RANK over NULL ties whole population silently

- **Rule:** before authoring a mart that ranks or filters on a column, verify the column is actually populated in the target game's source table with `SELECT COUNT(col) WHERE col IS NOT NULL FROM source`. A column appearing in `/meta` and existing in the Trino schema can be 100% NULL. A RANK/ORDER BY on an all-NULL column silently ties all rows at the same rank, so a "top-N" filter becomes a whole-population match with no error or empty result — it looks available and passes surface validation.
- **Why:** jus_vn playbooks 06/09 ranked by `ingame_fighting_power` to select top-10 and #1 players. The column exists in `mf_ingame_roles` and appears in `/meta`. The mart compiled and `/load` returned results. But `ingame_fighting_power` is 100% NULL across mf_users, mf_ingame_roles, and std_ingame_user_active_daily (verified: 0 of 1.69M rows populated). `RANK() OVER (ORDER BY fighting_power DESC NULLS LAST)` tied all 33,543 anchor-day users at rank 1; `WHERE rank <= 10` then returned all 33,543. A playbook verdict flipped from unavailable to "available, cohort = entire VIP base" — the exact silent-whole-base-match failure the design warned about. Caught during a cohort-distribution probe before the sweep, not in a demo.
- **Signal:** a RANK/percentile/filter mart returns cohort size = entire qualifying population (e.g., "top-10 leaderboard" = 33k users, same as the base); spot-checking rank values shows all records tied at rank 1 / same bucket; the ranking column is present in `/meta` and the compiled SQL is syntactically valid, but the *data* doesn't exist.
- **Apply:** add a pre-mart-ship cohort-distribution probe that fetches the cohort and asserts (a) cohort-size is plausible relative to the filtering base (e.g., a "top-10" cohort should be ~10, not the whole 866k), and (b) for rankings, spot-check that `row[0].rank != row[1].rank != row[2].rank` (if all ranks are identical, the ranking key is all-NULL or constant). A direct Trino count of non-null population (e.g., `SELECT COUNT(DISTINCT rank_col) FROM source`) before modeling flags the all-NULL case early.

### Cube `/sql` returns SQL capped by the default rowLimit — wrapping it in an INSERT silently truncates the cohort

- **Rule:** when you take the compiled SQL from Cube's `/sql` endpoint and run it directly (e.g. wrap it in `INSERT … SELECT … FROM (<compiled>)` to land a full cohort in a warehouse), STRIP the trailing `LIMIT n` (and `FETCH FIRST n ROWS ONLY`) first. Cube appends its default rowLimit (10000) to the compiled SQL regardless of whether the query carried an explicit limit — so the raw compiled SELECT silently caps results.
- **Why:** the segment-membership lakehouse writer compiles `{filters, dimensions:[identity]}` via `/sql`, gets back `… GROUP BY 1 ORDER BY 1 ASC LIMIT 10000`, and wraps it in a cross-catalog INSERT. Without stripping, every cohort >10k members landed exactly 10000 rows with no error — a paying-user segment of 7,829 looked fine, but an 8.3M-member segment would have been truncated to 10k and looked "done". The whole point of the feature (full membership, no `MAX_UID_LIST` cap) was silently defeated by Cube's default limit. Also: `/sql` returns **parameterized** SQL `[sqlText, params]` with positional `?` placeholders — inline params with a quote-aware scanner before wrapping, never naive string-replace.
- **Signal:** a "full cohort" write/export lands a suspiciously round count (exactly 10000, 50000); the count matches Cube's default page size rather than the segment's true `total:true` size; large cohorts and small cohorts both "succeed" but large ones plateau at the same number.
- **Apply:** after compiling, end-anchor-strip `/\s+LIMIT\s+\d+(\s+OFFSET\s+\d+)?\s*$/i` and the `FETCH FIRST … ROWS ONLY` form, then assert the landed count equals Cube's `total:true` count for at least one large segment. Note the compiled SELECT references game tables **bare** (`FROM mf_users`) — run it with the session catalog+schema Cube uses (`game_integration` + `GAME_SCHEMA[game]` from cube.js) so the refs resolve.

### Cube `/sql` for a pre-agg-served query is CubeStore dialect against a phantom table — not pasteable into the source DB
- **Rule:** the SQL Cube returns from `/sql` (and shows in the query-builder "Generated SQL" tab) is the SQL for the engine that will *execute* it. When a query matches a pre-aggregation that's `external: true`, that engine is **CubeStore**, so you get CubeStore dialect — backtick identifiers, `to_timestamp(...)` — against a `preagg_*` rollup table that exists only in CubeStore. Pasting it into a Trino/Presto editor fails. To get runnable **source** (Trino) SQL for such a query, reconstruct it from the `/sql` response's pre-agg metadata; do not naively swap quotes (the rollup table still won't exist in the source DB).
- **Why:** a marketing query on `game_key_metrics` matched the `key_metrics_by_source_daily_batch` rollup, so the SQL tab emitted `` SELECT `…__cost_vnd` FROM preagg_cfm.game_key_metrics_…_batch `` — `backquoted identifiers are not supported` in Trino, and even quote-fixed the `preagg_cfm.*` table doesn't exist there. The `/sql` response *does* carry everything to rebuild source SQL: `preAggregations[0].sql` is the rollup's own source SELECT (real `cons_*` tables, Trino dialect), `aggregationsColumns` is the additive re-aggregation recipe (`sum(...)`), `tableName` is the rollup table to re-point the FROM at, and `matchedTimeDimensionDateRange` is the real date range. **Trap:** `preAggregations[0].sql`'s params come back as partition sentinels `['__FROM_PARTITION_RANGE','__TO_PARTITION_RANGE']`, NOT real dates — inline those naively and the subquery filters on `from_iso8601_timestamp('__FROM_PARTITION_RANGE')` (invalid). Resolve sentinels from `matchedTimeDimensionDateRange` first.
- **Signal:** generated/`/sql` SQL has backticks or `to_timestamp(` and a `FROM preagg_*`/`prod_pre_aggregations.*` table; `external: true` in the `/sql` response; Trino error `backquoted identifiers are not supported; use double quotes`. If you "fixed" the quotes you'd next hit `table not found`.
- **Apply:** reconstruct (see `src/utils/rollup-source-sql.ts`): re-point the CubeStore exec query's FROM at `(preAggregations[0].sql)` as a subquery, translate dialect (`` ` ``→`"`, `to_timestamp(`→`from_iso8601_timestamp(`), resolve partition sentinels to `matchedTimeDimensionDateRange`, inline params (`src/utils/inline-sql-params.ts`). **Guard to additive rollups** — re-aggregating partials is only correct for `sum/min/max/count`; bail to the raw source SELECT (rollup grain) for `merge(`/HLL count-distinct, lambda/union rollups (`lambdaQueries` non-empty), or any CubeStore-only construct that survives translation. Equivalence holds by construction (rollup rows ≡ source SELECT output; re-aggregation ≡ the query). Caveat: rebuilt SQL reads *live* source tables, so it can differ from a stale rollup.

### Readiness probe: gate on /sql routing + CubeStore partitions, NEVER on usedPreAggregations (lambda masks it)

- **Rule:** to judge whether a pre-aggregation is actually serving, do NOT read `usedPreAggregations` off a `/load` body. EVERY rollup-bearing cube in this model exposes its rollup via a `rollup_lambda` (`union_with_source_data: true`), and Cube reports `usedPreAggregations: {}` (empty) for lambda unions — the result blends the sealed CubeStore rollup with a live source tail, so Cube won't credit one pre-agg. The honest gate is two-step: (1) a `/sql` DRY-RUN — if `preAggregations[].tableName` is present and `external:true`, the compiled query routes to a rollup (`FROM preagg_*`), not raw source; (2) cross-check that table against CubeStore `system.partitions.active > 0`. `built` = routed AND active; `from-source` = compiled SQL reads the raw source (no rollup matched the grain); `unbuilt` = routed but no active partition.
- **Why:** an earlier hardening "asserted usedPreAggregations non-empty = built". In a lambda-only model that reads EVERY rollup as passthrough no matter how completely it is built — a fleet-wide false negative. Live proof: after a full jus_vn build, `/load` returned `usedPreAggregations: undefined` while the `/sql` plan showed `external:true, FROM preagg_jus.active_daily_dau_by_country_payer_daily_batch` and CubeStore held 3 ready / 3 active partitions (297 rows). The field the probe asserted is the one field that is structurally always-empty here. (This is the SAME trap already documented above under "rollup_lambda … reports EMPTY usedPreAggregations" — the readiness probe re-walked into it.)
- **Signal:** readiness shows `built=0 / fromSource=N` (everything amber) immediately after a confirmed successful build; the count flips with VPN/Trino reachability rather than with builds; a `/sql` dry-run of the same query disagrees with `/load` (`external:true, FROM preagg_*` vs empty `usedPreAggregations`). A SUBSET reading `from-source` after a full build = a grain mismatch: the probe must query at the rollup's own `granularity` (a monthly rollup probed at `day` routes to source — `preagg-model-registry.ts` now carries `granularity` and `buildProbeQuery` matches it).
- **Apply:** `preagg-readiness.ts` `probeOne` issues a `/sql` dry-run (cheap, compile-only — also removes the cold-Trino latency `/load` carried), extracts planned preaggs via `extractPlannedPreaggs`, and verifies materialisation via `findPreaggByTableName().activePartitions`. Gated on `CUBESTORE_INTROSPECT_ENABLED`; when off (prod hosts that can't reach :3306) it degrades to /sql-routing-only (a planned rollup reads `built` without the partition check — still stricter than any-200). When matching a dry-run `tableName` to a CubeStore base, the dry-run already reports the bare logical name — do NOT re-strip it with the physical-name suffix stripper or every verdict degrades to `not-built`. Introspection queries: `docs/cubestore-cache-and-eviction.md` (lowercase `system.*`, no `SUM(boolean)`/JOIN).

### A triggered build won't appear in sweep history unless recorded at the trigger — log-inference loses it

- **Rule:** record an on-demand (Build button) pre-aggregation build as a durable history row AT THE TRIGGER (window close), from the build's own worker logs — do not rely on the scheduled-sweep log-inference path to backfill it. That path only collects builds bracketed inside a `Refresh Scheduler Interval` window and attributes them to a game by a `CREATE TABLE preagg_<game>.` table match; a scoped build sweeps every 20s (scattering the work across many tiny windows) and the strict table match drops completion lines whose table is only on `targetTableName`. Net: the operator sees "nothing rebuilt — every refresh key matched" right after a build that clearly created partitions.
- **Why:** the live build panel (`aggregateBuildEvents`) folds the same logs liberally and provably captured a jus_vn build (8/9 rollups), but it is ephemeral (10-min linger, lost on restart). NOTE: the exact strict-parser drop cause wasn't reproduced against a live failing sample (logs wiped post-restore) — the `targetTableName` gap is grounded in code; record-at-trigger is the authoritative fix regardless.

### A multi-query batch that `await`s all parts in one go discards the parts that SUCCEEDED when any one fails — settle each independently

- **Rule:** the tier runner ranked a cohort into top/middle/bottom by `await`ing three Cube queries in a single array literal. The middle tier uses a deep `OFFSET ≈ cohort/2` — on a 7.1M-user segment that `ORDER BY ltv … OFFSET 3.59M LIMIT 50` cannot finish inside the 30s per-tier timeout, so it threw, the outer `try/catch` returned `null`, and the WHOLE block was discarded — even though top+bottom (offset 0) returned in ~5s each WITH names. Net effect: the giant segment's Members tab was structurally un-nameable; every hourly refresh repeated the same all-or-nothing wipe and kept months-old nameless tiers. Fix: wrap each tier query in a `settleTier` that returns `null` on failure; store whichever tiers succeeded; only return `null` (keep prior block) when ALL fail. A missing tier just omits its tab — the FE already builds tabs only from non-empty tiers (`tierOptions`) and reads pools with `?? []`, so partial tiers needed zero FE change.
- **Why:** the deep OFFSET is the cost, not cold Trino — proven by timing the three queries with a warm cluster (top 5.0s, bottom 5.5s, middle FAILED at 30.0s). Raising the timeout doesn't help: a 3.5M-row sort+skip in Trino is genuinely slow and fragile. The middle tier of a whale-skewed base is also low-value (median LTV is ~0, same as bottom), so losing it occasionally costs almost nothing — whereas losing top+bottom costs the entire feature.
- **Signal:** an enhancement that works for small inputs and silently no-ops for large ones; a refresh logs `status=fresh` and recomputes `uid_count` but a sub-artifact's `computed_at` never advances; a batch of N independent remote calls where one slow member (deep pagination, a fat partition, one unreachable shard) makes the whole batch yield nothing. Any `const [a,b,c] = [await f(), await g(), await h()]` where the callees can fail independently is this bug waiting to happen.
- **Apply:** for independent remote sub-queries that each enhance (not gate) the result, isolate failures per item and keep partials; reserve the null/empty return for total failure so a transient blip never overwrites good prior data. Prefer avoiding deep `OFFSET` entirely on multi-million-row cohorts (a measure-threshold `WHERE ltv <= median LIMIT 50` avoids the skip scan) if the middle tier ever needs to be reliable rather than best-effort.

### An exported function defaulting a parameter to a value typed by an unexported type in another module fails the build only under declaration emit (TS4076)

- **Rule:** the Care scheduler's exported `listDueCareSegments(nowMs, window = parseCareWindow())` inferred `window`'s type from `parseCareWindow()`, whose return type was an interface (`Window`) declared but **not exported** in the member-360 scheduler module. `tsc` type-checks fine and the dev server runs fine, but the production image builds with `declaration: true` — and the `.d.ts` for the Care module cannot name a type that lives unexported in another module → `error TS4076 ... cannot be named`, deploy fails. ### A shared compiler that throws a low-level TypeError on malformed input sends an agent into a budget-burning self-correction spiral

- **Rule:** the advisor's `predicate_compile` / `propose_cohort` tools hand the model's predicate tree straight to `predicateToSql`. On a malformed node (missing `kind`, or a group missing its `children` array) the compiler fell through to `groupToSql` and crashed on `group.children.length` → the tool surfaced a raw `Cannot read properties of undefined (reading 'length')`. Fix shape-validation in the shared core, not per-tool, so all callers (and the API boundary) benefit. ### A "client disconnected" abort wired to the REQUEST stream aborts every POST immediately — the request closes when its body is read, not when the client leaves

- **Rule:** the /load proxy added a disconnect-aware abort (free Cube/Trino when an orphaned query's client goes away) via `req.raw.on('close', () => ac.abort())`. For a request with a body (POST), Node ends and auto-destroys the IncomingMessage the instant Fastify finishes reading/parsing the body — emitting `'close'` *before* the upstream /load answers. So the abort fired on every POST /load within milliseconds, `forward()` caught the AbortError, and returned its timeout-shaped body `{"error":"Cube request timed out after 110s"}` — a 504 in ~2ms, not a real 110s wait. GET /load has no body to consume so its request stream stays open until the response, slipping past the bug. Net effect: the chat-service (POST via `load-cube-rows`) had ALL its Cube queries killed, while browser surfaces (/build, /segments, /ops — GET via the SDK) worked, so it read as "charts don't render in chat" rather than "chat data is down." Fix: listen on the **response** stream — `reply.raw.on('close', () => { if (!reply.raw.writableFinished) ac.abort(); })` — which fires on actual socket close and the `writableFinished` guard makes a normally-completed response a no-op.
- **Why:** in Node's HTTP model the request `'close'` means "the inbound stream is done" (body fully read), NOT "the peer disconnected." Conflating the two is a classic footgun that only bites requests that carry a body, so it hides from any test/surface that uses GET. The misleading 504 text (the proxy's own timeout message, reused for any AbortError) sent diagnosis toward "Trino is slow" when the latency (single-digit ms) already disproved a timeout.
- **Signal:** a POST endpoint returns a *timeout* error in milliseconds; the same query works as GET but not POST (or from a browser but not a server-to-server caller); a disconnect/abort feature that worked in manual GET testing breaks a POST-only consumer; an error body whose wording implies a long wait paired with a tiny measured latency.
- **Apply:** detect client disconnect on `reply.raw` ('close' + `!writableFinished`), never on `req.raw`. When an abort-shaped failure appears, check the *latency* first — a "timeout" that returns instantly is an abort, not a timeout — and trace the abort source before assuming the upstream. Reproduce a POST-path bug with a real body over HTTP (GET probes will pass and mislead).

## Segment Snapshots & Lakehouse

### Predicate-free mf_users projection + Trino JOIN avoids per-segment join-rooting
- **Rule:** when materializing per-user state for many segment members, write a cross-catalog query that projects mf_users **without** the segment's predicate filters, then JOIN in Trino to the segment's membership snapshot — never apply segment scope inside the Cube query. This keeps the Cube query predicate-free (scanned once per game+snapshot_ts, cached within the tick) and the JOIN in Trino routes rows exactly to the segment's members (scan-wide, write-narrow).
- **Why:** early state-snapshot design scoped mf_users by the segment's predicate inside Cube (`cubeSql` with segment-defined filters), producing per-segment queries. For a segment with predicate referencing a different cube (e.g., recharge events, a `WITH recharge AS (…) WHERE … IN (SELECT uid FROM segment_membership@ts)`), the mf_users projection inherited wrong scope filters and returned wrong state values. Decoupling the projection (predicate-free) + the scope (JOIN to membership in Trino) fixed both: mf_users gets the full user set, Trino JOINs narrows it to the segment's actual members.
- **Signal:** state snapshots for a segment show wrong dimension values (ltv_vnd, ingame_name) that don't match the user's live record; different segments with overlapping members show different state values for the same uid; a segment whose predicate is defined over a non-mf_users cube (recharge, gameplay) shows empty state columns.
- **Apply:** always project mf_users predicate-free in Cube → compile once per game at tick start, cache the SQL. Route the membership list (from `segment_membership_daily@snapshot_ts`) and the state projection separately to Trino, then JOIN them: `INSERT segment_member_state_daily SELECT snapshot_ts, segment_id, uid, … FROM (mf_users projection) JOIN (membership@ts) ON uid WHERE …`. The projection's SELECT is dimension-only (dimension:[identity, ltv_vnd, …], measures:[]) — no Cube pre-agg routing, just a predicate-free scan.

### Segment KPIs reuse card-runner (runScopedKpi), never derive ratios/thresholds/foreign-cube KPIs from per-user state
- **Rule:** when persisting segment-level KPIs as a time-series, use the **same Cube query execution path** (`runScopedKpi` from the card-runner) that the Insights tab uses. Never compute KPIs by aggregating per-user state rows (e.g., dividing per-user payer counts by total members → paying_rate) — per-user state is dimensional, not additive, so derived ratios are wrong. Non-additive measures (paying_rate_30d, arppu_vnd, whales_count) *must* come from Cube, which applies the correct metric definition and handles cross-cube joins.
- **Why:** early KPI materialization aggregated per-user state snapshots directly: computed `COUNT(CASE WHEN is_paying_30d='yes' THEN 1 END) / COUNT(*) → paying_rate`. This silently produced wrong numbers whenever the segment's predicate didn't include the recharge cube — a "all users" segment scored everyone as a payer if ANY user in the game was paying (the denominator was segment size, numerator was the global payer count from the state snapshot). The card-runner's `runScopedKpi` scopes every query to the segment's **predicate filters** (not just membership), so non-additive measures stay correct. Insights tab + KPI time-series now read from the same source; no drift.
- **Signal:** segment KPI time-series show implausible numbers (paying_rate > 100%, arppu > lifetime value, whales_count > member_count); these numbers differ from the Insights tab for the same snapshot; a segment whose predicate doesn't mention the revenue cube still shows non-zero payment metrics in the KPI series.
- **Apply:** wire the KPI writer (`segment-kpi-writer.ts`) to call `runScopedKpi(segment.cube, spec, scope: {predicate, segment, snapshot_ts})`, NOT to aggregate per-user state columns. The writer produces one row per (snapshot_ts, segment, metric), value = Cube answer (or NULL if the cohort is empty). Verify persisted KPI == Insights KPI for the same snapshot via a manual test before shipping.

### Snapshots are as-of values — downsample by last-in-bucket, never sum; mixed-cadence collapses cleanly to coarser, steps toward finer
- **Rule:** when downsampling snapshot time-series to a coarser granularity (e.g., 15m capture → request for 1h data), aggregate by **last-in-bucket** (the most recent snapshot in that hour), never by summing or averaging multiple points. Snapshots are as-of values (the state *at* that moment), not flows — a sum would be meaningless. Mixed-cadence requests (some periods captured 15m, others daily) collapse cleanly to the coarser cadence in the response via `downsample-snapshots.ts`; stepping toward a finer cadence than captured renders carry-forward (repeat the last known value).
- **Why:** KPI time-series intended to downsample 15m revenue snapshots to 1h by summing (e.g., hourly = sum of 4 fifteen-min values). Revenue snapshots are segment-membership KPI snapshots, each stamped at a moment — a member counts once per snapshot regardless of cadence. Summing would overcount (4 snapshots × same members = 4× the revenue). Downsample by taking the *last* snapshot in each hour instead — that 1h row's value represents the segment's KPI state *at the end of that hour*.
- **Signal:** downsampled KPI values are 4× too large (or N× too large for N-per-bucket); downsampling from 15m to 1h increases the total when it should stay flat; a metric's trend looks discontinuous when mixing data from different cadences (a jump when switching from daily to 15m captures).
- **Apply:** in `downsample-snapshots.ts`, bin by the requested granularity bucket, keep `last(snapshot_ts)` per bucket. For mixed-cadence windows, track the effective granularity (coarsest cadence present) in the response header + signal in the data that finer cadence is unavailable. If the client requests finer granularity than captured, fill gaps via carry-forward (repeat the last known snapshot's values until the next real snapshot).

### LIVE-VALIDATION GATE: state writer assumes Cube /sql aliases as cube__field — offline parsing never catches wrong Cube dialect
- **Rule:** the state-snapshot writer compiles a predicate-free mf_users projection via Cube's `/sql` endpoint, expecting the response to carry aliases in the form `<member>` → `<cube>__<field>` (e.g., `mf_users.ingame_name` → `mf_users__ingame_name`). This is the standard Cube SQL alias format. If a custom Cube dialect or a future Cube version changes the alias form, the state INSERT will land **ZERO rows silently** — the column names in the SELECT won't match the column list in INSERT. No error, no warning, no dead-letter; the snapshot runs and reports success with `rowCount: 0`, and the monitor board shows a gap. Validate this assumption against the *live* Cube /sql endpoint before deploying.
- **Why:** the state writer (`segment-member-state-writer.ts`) parses `/sql` response's compiled SQL string via regex to extract the SELECT list (expecting `alias1, alias2, …` where alias = `cube__field`). It then uses these alias names to build the INSERT column list. If the parsing is wrong or Cube's alias format drifted, the mismatch is invisible until a data gap surfaces in monitoring. The fix is not code — the code is correct — but a smoke test: before deploying to prod, run the writer on a live segment and `SELECT COUNT(*) FROM segment_member_state_daily WHERE snapshot_ts = (most recent)` — confirm rows > 0. If 0, debug the alias parse vs the actual `/sql` response.
- **Signal:** segment state snapshots run (no errors in logs) but `segment_member_state_daily` stays empty or has 0 rows for recent snapshots; `console.warn("dead join…")` messages in the state writer logs (a fallback signal for when the Trino INSERT succeeds but returns 0 rows); the state-distribution endpoints return empty histograms even though members exist in `segment_membership_daily`.
- **Apply:** add a pre-deploy smoke-test script (`scripts/validate-state-writer.ts` or similar) that: (1) picks a prod segment with members, (2) calls the state writer directly, (3) asserts `rowCount > 0`. Run this in CI/CD before shipping or as a manual gate. Document the assumption (Cube alias format = `cube__field`) inline in the writer code. If a future Cube version changes aliases, the test fails loudly before it silently corrupts data. Monitor for 0-row snapshots in prod as a standing alert.

### Percentile / top-N spend measures degenerate without population scoping — always bind to payers
- **Rule:** a percentile cutoff over a spend measure (recharge, revenue) MUST be computed over a scoped population (e.g., recharge > 0). An unscoped percentile where free users dominate the distribution yields a cutoff of 0, selecting 100% of members. The cutoff query AND the segment membership query must share the same population scope — they're two passes over the same data, and if one is scoped and the other isn't, the cohort size diverges from the preview estimate. For multi-row-per-user marts (jus `mf_users` dual identity namespace), the cutoff must also be taken over the per-user-merged grain (via `split_part(user_id,'@',1) + max() + GROUP BY`, mirroring the cube's SQL), or it double-counts and cutoff ≠ cohort.
- **Why:** (a) percentile scoping: a "top 25% spenders" segment ran two passes — the cutoff against all users (includes free), yielding percentile 0 and cutoff = ₫0 (selects everyone), then the membership query ran against recharge > 0 (only payers), yielding ~58k members instead of the full user base. The two counts disagreed because the population wasn't aligned. (b) identity merge: jus's `mf_users` has two rows per user (VIET + INTL namespace); when computing a percentile cutoff, the raw query returned per-row percentiles (one per user per namespace), double-counting the spend. The fix: collapse to per-user max via `split_part + max + GROUP BY`, matching the cube's final grain. Once fixed, cfm_vn cutoff ≈757k₫ matched the prior measurement within 5%; jus_vn ≈6.9M₫ also within 5%.
- **Signal:** a 'top X%' segment returns ~100% of the population; a jus cohort's displayed size is ~2× the expected membership based on prior measurements. The `/api/segments/resolve-cutoff` preview estimate (estCount) diverges significantly from the final segment size after creation.
- **Apply:** always pass a `PopulationRef { filter, identityMerge }` when building a percentile cutoff. The filter (e.g., `recharge > 0`) must be the same as what the segment's membership query will use. For jus, set `identityMerge: 'split_part_max'` to collapse multi-row identity into per-user rows before percentile calculation. Test: create a "top 25% spend" segment, compare its size to the `estCount` from the `/resolve-cutoff` preview — they must be within 1–2% (allow for small refresh delta).

## Advisor & Diagnostic Rail

### Advisor data-gate compared SQL column names to exposed Cube member names — silently withholding levers whose data exists
- **Rule:** a lever's `requiredCubes` token is a Cube **member** name (`cube.dimension` / `cube.measure` — the exposed name), NOT the raw SQL column it maps from. `user_gameplay_daily.yml` computes `clan_cur`/`ladder_level_cur` in SQL but exposes them as dimensions `clan_id`/`ladder_level`; `etl_lottery_shoot` exposes `distinct_players` not `unique_players`; `ccu_by_server` exposes `server`/`server_peak` not `server_id`/`peak_ccu`. The data-gate (`lever-library-index.ts` `resolveLeversForGame`) does `requiredCubes.filter(c => !members.has(c))` against the live `/meta` member set — so a token written as the SQL column never matches, the lever is withheld, and the advisor tells the user to "unlock the missing cube" for data that is already present and served.
- **Why:** the FPS + MMORPG lever libraries were authored from the SQL/source schema, not the exposed Cube model, so 5 tokens across both libraries named non-existent members. The existing test built its member set FROM `requiredCubes` (`for (const c of lv.requiredCubes) s.add(c)`) — self-referential, so it could never detect the drift. Verified by probing live `/cube-api/v1/meta` for cfm_vn/jus_vn: the cubes and members existed; only the names mismatched. Fixing the tokens unlocked 4 cfm levers + the jus server-health lever.
- **Signal:** the advisor's recommendation says "missing cube / not available in this game's schema" and lists withheld levers with their `missingCubes`, but `/cube-api/v1/meta` for that game DOES contain the cube — just under a different member name (the SQL column vs the exposed dimension). `recommend_actions` returns zero cited candidates on a game that clearly has the data.
- **Apply:** never derive a test's "available members" from `requiredCubes`. Assert `requiredCubes ⊆ a real /meta snapshot` (see `server/test/fixtures/lever-gate-meta-members.json` + the naming-drift tripwire in `genre-lever-library.test.ts`), with a tiny documented `KNOWN_ABSENT` allowlist for genuine gaps (no guild data; a member added to source but not yet deployed to the tunnelled Cube). When adding a lever, copy the member name from `/meta`, not from the cube's SQL. Genre honesty still holds: a member truly absent from the model stays withheld — don't fabricate it.

### A "Keycloak 500" in the FE was a dead local API server (Node-version skew), not auth
- **Rule:** the FE auth call (`/api/auth/me`) failing surfaces in the UI as an auth/"Keycloak 500", but the local API server is reached through the vite proxy (`/api` → `:3004`), and when that target is down the proxy returns 500 — identical symptom, unrelated cause. Before touching Keycloak config, confirm the server is actually listening: `lsof -iTCP:3004 -sTCP:LISTEN` and `curl :3004/api/auth/me`. Local dev runs `AUTH_DISABLED=true` (in `.env.local`, which Node `--env-file` loads AFTER `.env`, later-wins) → synth admin, no real KC flow at all.
- **Why:** after rebuilding `better-sqlite3` for Node 22 to fix tests, the still-running dev stack was on Node 24; a crash dropped the server out of its `listen` (process alive under `tsx watch`, but nothing bound to `:3004`), so every `/api` call proxied to a dead target → 500. KC was reachable and bypassed the whole time. A clean restart of the stack under the repo's pinned Node (22) fixed it.
- **Signal:** UI shows a 500 on load attributed to auth/Keycloak; `curl localhost:3004/...` returns `000` (connection refused) while the `tsx`/dev-all processes still appear in `ps`; remote KC `/.well-known/openid-configuration` returns 200 (reachable). The repo declares Node 22 (`.nvmrc`/engines) but a dev process is on another major.
- **Apply:** keep the whole dev stack on the repo's pinned Node — after any native-module rebuild (`better-sqlite3`), restart `dev:all` under `nvm use 22` so the running process ABI matches the rebuilt binary. Diagnose "auth 500" by checking the proxy target's liveness first; only inspect KC after the server is confirmed up.

### Chat-service dropped X-Cube-Game / X-Cube-Workspace on server calls → diagnose ran on the wrong game, "cube not found" for present data
- **Rule:** every chat-service → server call must forward `X-Cube-Game` and `X-Cube-Workspace`, not just owner id + cube token. The server resolves the ACTIVE Cube model from those headers (`introspectionCtx` reads `x-cube-game`; the workspace-header plugin reads `x-cube-workspace`). With the game header absent the server falls back to `loadGamesConfig().defaultGameId` — a different game's model — so per-game cubes like `billing_detail` resolve to `Cube 'billing_detail' not found` and population metrics return the wrong game's numbers (observed: payers 8,047 instead of cfm_vn's 238,111). The cube token's `game` claim is NOT used for model selection; only the header is.
- **Why:** `chat-service/src/services/server-client.ts` `buildHeaders` set only `X-Owner-Id` + `X-Cube-Token`. `ToolContext` already carried `gameId` + `workspace`; they just weren't forwarded. Confirmed by replaying the server `/api/advisor/diagnose` call with token-only (no `X-Cube-Game`) — reproduced the exact wrong-game numbers + `billing_detail not found`; adding the header fixed it.
- **Signal:** the advise/diagnose rail narrates `Cube '<name>' not found` for a cube that `/cube-api/v1/meta` clearly exposes for that game, and/or population values that don't match a direct game-scoped Cube query. A direct server call WITH `x-cube-game` works; the same call without it reproduces the failure.
- **Apply:** when smoke-testing a turn, send `x-cube-token` + `x-cube-game` + `x-cube-workspace` (the turn endpoint requires the token). Beware the **message-keyed turn cache**: re-running the identical message replays the pre-fix (broken) answer — vary the message text or send `x-bypass-cache: 1` to see the effect of a code change. Isolate "wrong data" bugs by reproducing against the server endpoint directly with/without each header before touching the engine.
