Data Connector · Worked example

Reconstructing ballistar's Cube model from a live Trino connection

The Data connector connects live to the warehouse, profiles a raw schema, and proposes a draft Cube data-model — the bootstrap stage that feeds Coverage (reconcile) and the Drift Center (repair). Below: the real ballistar tables behind today's hand-authored cubes, run through the actual live pipeline (profileTable → inferSchema → scaffoldCubeModel → toYaml), shown next to the cubes a human curated — so you can see exactly what the agent gets right, what it flags for review, and what only a human (or the LLM-enrichment pass) can add.

    Read this as the triage canvas. Of 20 raw columns, the agent resolves 16 with high confidence (collapsed/auditable),
    surfaces 2–4 for a quick accept/reject, and leaves the genuinely human work — joins, derived ratios, segments, business
    naming, pre-aggs — as guided suggestions (golden-query seeding shows which members appear in real saved queries;
    LLM enrichment proposes names). The DA never starts from a blank file.
  


  
  
std_ingame_user_active_daily active_daily — at scale

  
43 columns, 561K rows, profiled in 8.5s. Shows the heuristic at width — and one honest limitation.

  


  

    Honest limitation caught live: the agent found no single-column primary key — correct, because this table is
    one-row-per-user-per-day and the hand-authored cube uses a synthetic composite_pk = CONCAT(user_id, '__', log_date).
    A composite key can't be inferred from column stats alone, so the triage step prompts the DA to define it. The 12 inferred
    measures (the total_*_active / *_role_level bigints) and 4 time dims land correctly;
    fighting_power + client_ip fall under 0.6 → review.

How this was produced — real code, real data, no mocks. Live read against

    trino-profiler.profileTable() → raw-schema-inference.inferSchema() →
    cube-model-scaffolder.scaffoldCubeModel()/toYaml(). Same path the
    POST /api/onboarding/generate endpoint runs; staging + approve-write add the gated persistence.

    
Bounded + read-only: one aggregate stats query per table (count / approx_distinct / min / max) + one LIMIT-8 distinct
    sample per column; 20s statement timeout; credentials redacted from all output.
