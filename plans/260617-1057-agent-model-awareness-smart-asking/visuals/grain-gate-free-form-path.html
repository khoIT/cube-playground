chat-service · agent grain rule · P5 follow-up

The grain gate lives in the engine.
The agent often doesn't go through the engine.

A “rank top-N by metric” turn resolves the metric down one of two paths, and the model picks which. The deterministic grain gate — drop per-head ratio metrics (a sum ÷ a head-count) when ranking individuals — sits only on the engine path. Extending it to the free-form path closes the cases where the agent offers metrics without ever calling the engine. Below: where that gap is real, and where it isn't.

Why the gap exists
The smoke run showed the agent took the free-form path for “top spenders” — it auto-answered and never called disambiguate_query. So the gate never ran. Coverage today = only the turns where the model chooses the engine, which is the minority.

In which situation extending the gate actually helps

Situation Path taken Extend the gate? Why
Individual ranking, model offers metrics directly “top spenders” → offer_choices chips, ratio in set Free-form YES — the gap Engine never called → no code gate fired. A ratio (ltv/arpdau) can reach the chips. Only enforcement here is a prompt sentence. This is the case worth fixing.
Individual ranking, model calls the engine “rank players by ?” → disambiguate_query Engine No — already covered P5 gate runs in metricOptions(); ratios already dropped. Duplicating on the free-form path is redundant here.
Group ranking “top countries by ARPU” Either No — ratios are valid A per-head average is well-defined over a group. Gating here would wrongly remove a correct option. The gate must stay grain-aware, not blanket-ban ratios.
Grain unknown / no entity resolved “top VIP players” (no concept match) Either No — fail-safe Can’t prove it’s an individual → don’t exclude anything (same rule both paths). Excluding on a guess risks dropping a valid metric.
Metric already named, or no ratio in the candidate set “top spenders by revenue” Either No — moot Nothing to gate: the gate only ever removes ratio candidates, and there are none (or the metric is pinned).

net  Extending the gate to the free-form path buys exactly one scenario — the ▶ row — but it's the most common live path for ranking questions, and the only one where a wrong-grain metric can actually surface to a user. The other four are already correct (engine-covered, or correctly not gated). Mechanism if we do it: apply the same refKind==='ratio' + isIndividualEntity filter to the metric set the agent passes into offer_choices (validate-and-strip at the tool boundary), so the rule is code-enforced on both paths — not guidance on one. On cfm_vn, that drops ltv/arpdau/arppu but keeps arpu (it's a per-user measure, mf_users.arpu_vnd, not a ratio).
