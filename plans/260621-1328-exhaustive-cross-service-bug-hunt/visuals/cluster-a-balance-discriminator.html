Cross-service bug hunt · Cluster A · Findings #1 #2 #14

One discriminator, copied three times — minus the is_error clause

The canonical detector in claude-runner.ts classifies a failed turn on is_error === true || subtype !== 'success'. Three peripheral one-shot calls copied only the subtype half. The live LiteLLM gateway reports a drained-balance 400 as subtype:'success' with is_error:true — so the copied guards are all false on a real exhaustion.

The live gateway balance-400 message shape (verified)
{ type: 'result', subtype: 'success', is_error: true, api_error_status: 400,
  result: 'Your credit balance is too low to access the Anthropic API' }
A subtype && subtype !== 'success' check evaluates false here — the balance branch never runs.

The discriminator — before vs after

Before

subtype-only — misses the gateway shape

// copied into salvage, segment-brief, title
if (m.type === 'result') {
  if (m.subtype && m.subtype !== 'success'
      && isBalanceExhaustedError(m.result)) {
    reportKeyBalanceExhausted(...);   // never reached
  }
  result = m.result;  // ← captures the error string
}
⚠️
Drained key never reported (failover ladder defeated); in salvage the raw billing error is persisted + streamed as the user's answer.
After

shared canonical detector

// anthropic-key-failover.ts — single source of truth
const balanceText = balanceErrorTextOf(m);
if (balanceText) {
  reportKeyBalanceExhausted(...);
  return ''; // or throw — per site
}
if (m.type === 'result') {
  if (isFailureResultMessage(m)) return '';
  result = m.result;
}
✅
Both SDK shapes (result + assistant-echo) caught; drain reported → rotation advances; no failure text ever leaks as content.

The fix is DRY: promote balanceErrorTextOf + add isFailureResultMessage to anthropic-key-failover.ts (beside isBalanceExhaustedError + reportKeyBalanceExhausted). claude-runner and all 3 sites import the one detector — the discriminator can no longer drift. Regression tests assert the exact subtype:'success'+is_error:true shape is caught.

On finding #2: a skeptic correctly narrowed it — parseBriefResponse's JSON.parse blocks the error text from being served as a brief body, so the real bug is the failover-defeat + mis-attributed 502, not a content leak. The fix addresses both.
