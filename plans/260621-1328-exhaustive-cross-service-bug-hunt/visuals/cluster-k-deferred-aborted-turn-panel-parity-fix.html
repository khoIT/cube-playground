1 · Aborted-turn handling (R31 HIGH · R38 LOW)

The bug had two halves: the server persisted a partial on one abort path but not the other, and the client only committed a turn on done — so an aborted turn's partial answer vanished the instant the status flipped.

server · before graceful abort (loop ends) → appendTurn persists partial ✓  ·  thrown AbortError → catch emits turn_aborted+done, returns before persist ✗ partial lost on refresh
server · after both paths → appendTurn persists partial (+ flush observability FKs, increment count) ✓ refresh-consistent
client · before commit effect fires only on status === 'done'; isStreaming excludes aborted ✗ partial disappears mid-session
client · after commit on transition into done OR aborted, in both surfaces ✓
user Stop store cancel() wiped entry to idle (partial gone) → now keeps partial as aborted; empty turn still falls back to idle ✓ keeps what was generated
