Cross-service bug hunt · Cluster F · Findings #10–13

Open-month rollup partition never seals — the timezone trap

Four games' recharge rollups set build_range_end to a bare MAX(timestamp). The column is timezone-aware and queries arrive in Asia/Saigon (UTC+7), so the computed end sits 7 hours ahead of UTC now. CubeStore refuses a partition that ends in its own future — so the current month never builds, and every current-period revenue query falls through to a full Trino scan.

Why CubeStore rejects the open partition
UTC now
+7h ahead
MAX(pay_time) @ UTC+7
CubeStore: "Internal: second time provided was later than self" → partition discarded
Before

uncapped MAX

build_range_end:
  SELECT MAX(pay_time)
  FROM etl_ingame_recharge
  • end = latest pay_time, +7h ahead of UTC now
  • CubeStore rejects the open-month partition
  • current month never seals → full etl_ingame_recharge scan (seconds, not ms)
After

LEAST cap (cfm-proven pattern)

build_range_end:
  SELECT LEAST(MAX(pay_time),
               current_timestamp)
  FROM etl_ingame_recharge
  • end clamped to never exceed UTC now
  • CubeStore accepts the partition
  • current month seals → rollup serves from CubeStore (ms)
