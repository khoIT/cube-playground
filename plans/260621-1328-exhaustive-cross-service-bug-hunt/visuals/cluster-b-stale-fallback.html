Cross-service bug hunt · Cluster B · Finding #3

"Served stale on Trino error" — code that could never run

Four movement endpoints (kpi-trend, movement, state-distribution, state-distribution-trend) document graceful degradation but hard-502 on any Trino blip. The fix turned out to be two-part — and writing the missing key alone was still not enough.

The bug, in two layers

1
The :stale key was never written
Success path wrote only cacheSet(cacheKey, payload); the catch read cacheGet(cacheKey + ':stale') — a key no endpoint populated. Always null → straight to 502.
2
Even writing :stale wouldn't help — the TTL trap
Fresh and :stale share one 10-min TTL and one write time, so they expire together. While fresh is warm the handler returns at the top (fresh-hit short-circuit) and never reaches the catch; once fresh expires, :stale is gone too. The fallback is unreachable either way.

The fix — write both keys AND read stale past its TTL

Before
// success
cacheSet(cacheKey, payload);
// error
const stale =
  cacheGet(cacheKey + ':stale');  // always null
if (stale) return {...stale, stale:true};
return reply.status(502)...
After
// success — one helper, all 6 endpoints
cacheSetWithStale(cacheKey, payload);
// error — read ignores the fresh TTL
const stale =
  cacheGetStale(cacheKey + ':stale');
if (stale) return {...stale, stale:true};
return reply.status(502)...
⚠︎ Why the TTL-bypassing read is the load-bearing half

cacheGetStale returns the entry regardless of age — so the :stale mirror outlives the fresh window and is actually present when a catch runs after the fresh entry expired. Without it, cacheSetWithStale alone is cosmetic: the entry it writes would have expired at the exact moment the fallback needs it. The regression test advances the clock 11 min (past the 10-min TTL), forces a reader error, and asserts stale:true is served — it fails against the write-only fix.

All six endpoints unified on the helper pair

kpi-trend movement state-distribution state-distribution-trend snapshot-ledger snapshot-coverage

Scope note: the two snapshot endpoints (which the finding considered "surviving") shared the same latent TTL trap, so routing them through cacheSetWithStale + cacheGetStale makes their documented stale-on-error behavior actually work too. One helper pair removes the drift that caused the original divergence.
