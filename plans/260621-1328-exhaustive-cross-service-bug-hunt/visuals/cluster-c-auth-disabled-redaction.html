Cross-service bug hunt · Cluster C · Finding #4

Tokenless redaction is a no-op under AUTH_DISABLED

The redaction gate that strips payer/CS/VIP columns from tokenless callers keys on Boolean(req.user). Under AUTH_DISABLED every request — including a bare curl with no token — is decorated with a synthetic admin, so the gate is always open.

Live on: local dev + playground host (AUTH_DISABLED, non-prod NODE_ENV, VPN-reachable)  ·  Prod real-auth: not affected (boot guard blocks AUTH_DISABLED+prod)

Broken

Anonymous curl, no Authorization header

  • 1
    Request hits the onRequest hook
    GET /api/segments/<uuid>/members — no token, no X-Owner.
  • 2
    AUTH_DISABLED branch runs
    request.user = devUser() — synthesized role:'admin', every feature on. Set unconditionally, for any caller.
  • 3
    Members route reaches the gate
    redactSensitiveMembers(payload, Boolean(req.user)) → Boolean(req.user) is true.
  • 4
    Gate returns payload untouched
    if (authenticated) return payload; — the SENSITIVE_MEMBER_FIELD strip never runs.
🔓
Leak. Full per-player dossier served tokenless: billing_lifetime, ltv_vnd, arppu, vip, csat, cs_ticket_detail. Members route doesn't even call guardSegment → any segment UUID, cross-workspace.
Fixed

Gate on a real-token-verified signal

  • 1
    Add request.tokenVerified decorator
    Defaults to false. A new request flag distinct from identity presence.
  • 2
    Set true on the real-JWT branch only
    After verifyAppJwt + active grant resolves: request.tokenVerified = true. The AUTH_DISABLED branch leaves it false.
  • 3
    Gate keys on the new signal
    redactSensitiveMembers(payload, req.tokenVerified) — synthetic dev admin no longer counts as authenticated.
  • 4
    Tokenless callers get redacted rows
    Sensitive columns stripped for any caller without a verified token, in every mode.
🔒
Sealed. Prod token-holders still see full data (unchanged). Prod tokenless still redacts (unchanged). AUTH_DISABLED curl now redacts (fixed) — the tokenless capability serves only non-sensitive columns, as designed.

Why this is a fix, not a behavior reversal: the gate's own comment claims "req.user is set only when a real token verified" — false under AUTH_DISABLED. The change aligns the code with its documented intent. The cross-workspace pull on the members route (no guardSegment) is left intentional — the tokenless "CS handed the URL" capability model is by-design; redaction is the correct boundary, not blocking access.
