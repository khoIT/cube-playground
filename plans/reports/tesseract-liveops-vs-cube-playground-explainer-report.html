Tesseract in real life — and why it is not cube-playground

A plain-language, worked walkthrough of thangLV's Tesseract platform (v0.3) through one LiveOps scenario — then a contrast with cube-playground, the holistic exploration tool, and where the two meet.

The one idea everything hangs on

At ~750,000 events/sec arriving and ~15,000 reads/sec going out with a 500 ms deadline, you cannot re-read a player's raw event history — not when an event arrives, not when a decision is asked for.

Every design choice in Tesseract is a consequence of that single constraint. Trace the chain:

can't re-read raw events (too slow at read · too costly at write) └─► must store a pre-computed, incrementally-updated number per player └─► a nightly batch number alone is too stale for LiveOps └─► so add a live stream-updated number on top └─► now TWO independent writers must combine at read, with no raw data └─► ∴ the aggregation must be MERGEABLE └─► ∴ the metric-algebra taxonomy — non-mergeable metrics are batch-only

The scenario we'll follow

Game Dragon Empires. Player p_8842 is playing right now. The Promotion engine wants to decide, in-session and in milliseconds: "Is this a high-value, currently-active player I should show a bonus-bundle offer to?"

That decision is a machine call, thousands of times a second, against fresh state — exactly what Tesseract is for. No human is reading a dashboard here.

A raw event — the atom

Everything starts as canonical events: one fact about one player at one moment. The fields are fixed; the payload varies by type.

(game=DragonEmpires, player=p_8842, object=sku_goldpack, event_time=10:12, event_id=content-hash, type=purchase, payload={amount: 4.99, item_cat: currency})
Dimensions vs events: a metric's "dimensions" are just fields lifted out of the event to group by. lifetime_spend has dimensions (game, player) → one total per player. Add item_cat as a dimension and you get spend split by currency / cosmetic / … — same events, finer buckets. The player is always the anchor, because hot state is keyed (game, player, object?, metric).

The merge is the whole trick. The gateway never re-scans events — it reads two small pre-aggregated numbers and combines them by the metric's algebra: sum → add, last-value → keep newest, sketch → union. Tonight's batch recomputes BASE from full history, advances as_of, wipes the now-superseded LIVE, and measures how far it had drifted — so any stream error self-heals within one batch period. The served value is always one coherent answer stamped with its freshness: eventual (as-of) consistency.
Why exact-money reads skip LIVE. When the dispatched offer's value must be exact (real reward / price), the gateway reads BASE only — authoritative, accepts being a few hours stale — rather than the fresher-but-estimated merged value. Freshness for eligibility; exactness for money.

The LiveOps gotcha worth telling product teams: the things they most want in real time — funnels, sessions, exact unique counts, leaderboards — are exactly the holistic ones. Real-time rules must lean on additive + last-value + sketch proxies (spend totals, last-seen, approximate distincts, timeout-based "active"). Anything needing exact ordering / uniqueness / ranking comes from the nightly BASE, or from exploration.

Pushing the design — two fair objections

Both come up immediately, and both have honest answers that sharpen where the line actually sits.

"Exactly how many unique items does this player own — why not just keep a store and track it?"

You can — but only by storing the player's full set of distinct item-types (you need the set to dedupe). That breaks two things the platform is built on:

  • It isn't mergeable. To combine a batch base set with a live stream set you'd re-merge the actual sets at read — the raw re-aggregation the whole design exists to avoid.
  • State grows unbounded. A growing set per player × 30M players blows the cost ceiling the hot store is sized for. Bounded per-player state is what keeps it affordable.

The warehouse already holds every event, so batch computes the exact number overnight and serves it from base. "Batch-only" really means "don't pay real-time prices for an answer that doesn't need to be sub-second."

Subtlety: if ownership is +1/−1 add/remove and each item is unique, the count owned is just an additive counter — that is real-time. Only distinct types (where the same type recurs) is holistic.

Litmus test: does the number need exact and sub-second and per-player-at-scale — all three at once? You can almost always drop one: exactness can wait (batch), or sub-second can approximate (HLL ±2%).
"Is this player in the top 1% of spenders this week — what if users can wait a few seconds?"

Two good answers — and the second is the interesting one.

1 · Decompose it (no waiting needed). "Top 1%" hides a population statistic — the 99th-percentile weekly-spend cutoff (say $500). That barely moves minute-to-minute. Precompute the cutoff on a schedule; then the per-player check is just weekly_spend ≥ cutoff — and weekly_spend is a real-time additive metric. A slow global number + a fast per-player compare = real-time, exact-enough.

2 · Or add a "warm" tier (this is what the seconds buy you). Relaxing latency to seconds unlocks a middle tier between the hot store and nightly batch: query the derived-fact layer (Iceberg / Trino) on demand for a fresh, exact, cross-player answer — viable at low QPS. That warm tier is literally cube-playground's scan-on-read engine doing operational duty.

Hot · p99 ≤ 500msTesseract hot store · mergeable metrics · ~15k QPS
Warm · secondson-demand Trino over derived-fact · holistic, low QPS
Batch · nightlySpark base · exact, any algebra

So the real boundary isn't a wall — it's a spectrum of latency tiers. v0.3 specifies the hot and batch ends; the warm tier is a reasonable extension (and a good question for thangLV).

In one line: Tesseract is the live scoreboard for machines; cube-playground is the film room for humans. Speed vs breadth — not competitors.

Same data, opposite question — watch the routing

Run two real questions and see which plane each belongs to, and why.

Q1 · "Right now, is p_8842 a high-value active player, for an in-session offer?"
→ Tesseract · point read, <1ms, pre-materialized, fresh, per-player
A machine needs one player's decision-ready state, thousands of times a second, inside a frame budget. Tesseract already holds the answer — it just merges base+live and stamps it. cube-playground could compute this but would scan events in Trino over seconds — fine for a human, fatal for an in-session offer.
Q2 · "Last month, what's D7 retention of gold-pack buyers, split by acquisition channel and country?"
→ cube-playground · scan-on-read, exact, arbitrary dimensions, holistic OK
A human exploring — cross-player, arbitrary slicing, exact, order-dependent retention math. Holistic and ad-hoc: impossible to pre-materialize per player. Tesseract is structurally the wrong tool; this is exactly what a semantic layer over the warehouse is for.

The one-line distinction

Tesseract stores the answer to a known question and keeps it fresh, so a machine can grab it instantly. cube-playground answers any question you can think of by scanning when you ask. The first trades breadth for speed; the second trades speed for breadth. Neither replaces the other — and Tesseract explicitly refuses the exploration job (no Tesseract dashboards · DA/BI explore the processed layer via a gateway). And they run in sequence: you explore holistically in cube-playground to decide which events and what thresholds are worth serving, then register the chosen few in Tesseract for real-time — see tab ③.

Where the two meet

They share one thing: the definition of a metric or segment. They should never share data/state. So the integration is definition-flow, not data-flow — far simpler than the two-way sync it first looks like.

The relationship is a sequence, not a sync

cube-playground is the understand-and-decide phase; Tesseract is the runtime phase. You explore the data holistically to choose what is even worth operationalising — then only the chosen, mergeable few cross the line into real-time serving.

1Explorecube-playground · full fidelity, any slice, holistic — understand behaviour & distributions
▸

▸
3Registerpublish declaration → Tesseract registry · serveability lint checks it's mergeable
▸
4ServeTesseract maintains it in real time → LiveOps acts in-session
Concrete link to the demo: the $50 in tab ①'s eligibility rule is not arbitrary. You would discover it in cube-playground — explore the spend distribution, find the 90th-percentile whale cutoff ≈ $50 — then freeze that number as the constant in the Tesseract real-time rule. Exploration finds the threshold; Tesseract enforces it at runtime. The same holds for which events to ingest and which metric to maintain: holistic analysis upstream decides the narrow, mergeable set Tesseract commits to serving.

…and a spectrum, not a wall

The two systems aren't only sequential — at serve time they're the two ends of one latency continuum. A decision picks its tier by how fast it must answer and how exact/holistic it must be. The middle "warm" tier is where the boundary genuinely blurs: cube-playground's scan-on-read engine, doing operational duty.

Important · load-bearing

Cube is the on-ramp, not the runway

QPS is a property of the tier, not the metric. The warm plane (Cube → Trino → Iceberg) is where humans discover and compose — it never carries the ~15k-QPS operational load. The only true high-QPS path is hot. So "how does warm serve thousands of QPS?" has one honest answer: it doesn't — it decides what gets promoted to hot, which does.

1 · How users explore what's serveable. Make the Tesseract Metric Registry a lens inside Cube's catalog: every modeled member is badged registered? · algebra class · freshness tier · serveable-at-QPS — the operational-serveability lint (Seam 1, below). Browsing that federated catalog is "what's available in warm to combine into a Tesseract condition." Explore-only members stay visibly distinct from operationalised ones, so an author can't compose a segment that physically can't be served.

2 · How thousands of QPS actually get served — never from warm; three paths, in order of preference:

  1. Promote to hot — register the metric (mergeable, or a materialised flag); one declaration → executors builds it into the hot KV; the Gateway serves it as a point read (p99 ≤ 500 ms). The real ~15k-QPS path.
  2. Cube pre-aggregations — Cube Store serves rollups at high analytical concurrency, sub-second (Cube's own claim — not sub-ms), but refresh-interval stale and analytical, not transactional. Powers the LiveOps console — cohort grids, segment sizes, KPIs many operators watch at once — not the in-session decisioning path (per-key point reads, live, ~15k QPS). The disqualifier is access-pattern + freshness, not latency.
  3. Raw Trino — genuine ad-hoc only; tens of QPS, seconds. The "let me check before I register" query.
Cross-tier conditions — the "top 1% this week AND active now" pattern. The percentile is non-mergeable ⇒ can't be live. So precompute it warm/nightly into a hot-servable flag (is_top_1pct = true); the operational check is then two cheap hot reads: flag AND active. Collapse the holistic part into a hot feature, combine cheaply with live state — and Cube is where you discover you must.

Seam 2 — exploration over Tesseract's derived-fact layer

Tesseract's authoritative base persists as Iceberg on MinIO — the same Trino stack cube-playground already queries. Pointing Cube at it is new model YAML + one Trino catalog entry, not a new pipeline. The payoff: analysts explore the same numbers the operational systems act on — single lineage, no reimplementation drift.

Read the derived-fact Iceberg layer (SQL-scannable), never the hot KV store (point-lookup only, on the latency-critical serving path). Surface the as_of stamp as a time dimension so analysts know the freshness boundary.

What NOT to do

  • Don't mirror hot state or two-way-sync data — wrong target, and it re-creates the multi-source-of-truth drift Tesseract spent two review rounds eliminating.
  • Don't stand up a competing registry — one canonical declaration. cube-playground authors into Tesseract's registry for operational metrics; owns exploration-only metrics alone.
  • Don't build deep integration yet — Tesseract is a target (ADRs decide Jul–Sep 2026, v1 is one game, no self-serve UI). Align the definition model now; wire later.

Highest-leverage move

Get cube-playground's metric/segment model into the ADR-003 (metric declaration) conversation before its 2026-08-15 decide-by — so the registry's declaration schema is co-designed to be authoring-friendly, and the two systems can't diverge. Especially: does the registry model entities/joins, or is it flat per-(player, object, metric)? If flat, cube-playground's semantic model is the only place relationships live — making "cube-playground = semantic layer, registry = compiled projection" not just convenient but necessary.
