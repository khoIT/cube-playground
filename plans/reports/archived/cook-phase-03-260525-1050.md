## Idempotency Strategy

- `llm_calls` + `tool_invocations`: **INSERT OR IGNORE** on the table-level UNIQUE constraints already defined in the migration (`UNIQUE(turn_id, step_index)` and `UNIQUE(turn_id, tool_use_id)` respectively). Chose IGNORE over REPLACE because replays should not overwrite an already-committed row — the first write wins and the dupe is silently dropped.
- `sdk_events`: plain **INSERT** (append-only). Duplicate SDK messages from retries are retained as observability signal rather than collapsed.

## Truncation

- `content_json`, `args_json`, `payload_json`: capped at 64 KB (64 × 1024 JS string length).
- `result_summary`: capped at 4 KB.
- Truncated strings are suffixed with ` [truncated]` so readers know the value was cut.
- `truncate()` exported from `observability-store.ts`; used directly in the recorder.

## Pattern Alignment

- Follows `chat-store.ts` exactly: `db.prepare(...).run(...)` on every call, no ORM, no async, no connection pool.
- `insertLlmCall` / `insertToolInvocation` / `insertSdkEvent` take a typed row shape from `types.ts` (`LlmCallRow`, `ToolInvocationRow`, `SdkEventRow`).
- Read helpers (`listLlmCallsByTurn`, `listToolInvocationsByTurn`, `listSdkEventsByTurn`) exported for phase-06 debug API reuse (DRY).

## Error Containment

Every INSERT in `LlmTraceRecorder` is wrapped in `try/catch` → `console.warn`. The runner already wraps observer calls in its own try/catch (phase-02), but the recorder provides an independent defence layer per constraint #5.
