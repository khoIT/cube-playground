The legacy /load pass overload

What happens when ANOMALY_DETECTOR_DRIFT_ONLY is turned off — and why it was on by default in dev. A look at how the anomaly badge gets populated, why ~1-in-5 metric reads fail, and how much of it is actually a problem.

Verdict: transient & self-healing — but wasteful and fragile. Not an outage; a design asymmetry.

01Two detectors, one file — and only one has guardrails

The detector ships in two modes. The newer SQLite mode (Inbox) was built with caps. The older legacy mode (badge) was never retrofitted. Flipping DRIFT_ONLY off re-activates the un-guarded one.

runDetectorTick() SQLite mode → Inbox / /api/anomalies
  • ✓Curated watchlist. Iterates ANOMALY_METRICS — ~1–3 metrics/game (8 total), not the registry.
  • ✓Budget cap. ANOMALY_QUERY_BUDGET_PER_TICK = 20 loads, then stops.
  • ✓Per-game mutex. A second tick won't re-enter an in-flight game.
  • ✓Cadence. Every 15 min — light, predictable.
scanGameLegacy() Legacy mode → badge / anomaly-state.json
  • ✕Full registry. Walks all 73 getAllBusinessMetrics() per game.
  • ✕No budget cap. The queryBudget() guard lives only in SQLite mode.
  • ✕No concurrency cap. Sequential await load() — a long marathon, not a pool.
  • ✕Cadence. Every 6h, but each pass is a sustained cold-query hammer.
02Why it's ~306 queries, not 73

Ratio metrics (DAU/MAU, ROAS, retention…) fire two reads — numerator + denominator — aligned by date. So the load count is bigger than the metric count, and it multiplies across every configured game.

Across the 3 games currently in ANOMALY_DETECTOR_GAMES=ballistar,jus,ptg → up to ~306 sequential cold reads in a single legacy tick. Games where most refs don't resolve (e.g. ptg) skip the unresolved metrics, so the real count per game is lower — but the resolved-yet-Trino-cold ones are exactly the ones that fail.

03It's a sustained marathon, not a thundering herd

Every read is await-ed before the next — so at most one query is in flight. The damage isn't a connection flood; it's minutes of back-to-back cold Trino scans (3.5–15s each) that exhaust pools and start dropping connections (fetch failed).

04Why drift-only was the dev default

One early return is the whole story. In drift-only mode the cheap /meta reconciliation runs (feeds Drift Center) and the function bails before the per-metric /load loop that touches Trino.

// anomaly-detector.ts · scanGameLegacy()
const { meta, token, unresolvedIds } = await runDriftReconciliation(game, 'detector'); // cheap /meta — feeds Drift Center
if (!meta || !token) return {};

if (driftOnly()) return {};  // ◀ DRIFT_ONLY=1 stops here. No Trino. The safe dev default.

const metrics = getAllBusinessMetrics();           // ◀ all 73
for (const metric of metrics) {
  if (unresolvedIds.has(metric.id)) continue;
  const numRes = await load(plan.numerator, token);     // ◀ cold Trino scan, no budget cap
  if (plan.denominator) await load(plan.denominator, token); // ◀ ratios double it
  ...
}

The reason it existed. Dev Trino is slow and intermittently unreachable. Drift-only removes the data-source dependency entirely — the detector still does useful work (ref reconciliation, drift history) without ever hitting Trino. It was the pragmatic dev posture, not a bug.

What flipping it off reactivated. The full uncapped registry walk — a path that predates the SQLite mode's guardrails and never inherited them.

The asymmetry, in one line. The safe path (SQLite) has a curated watchlist + budget + mutex. The path that actually writes the badge (legacy) has none of them. Drift-only was masking that gap.

So this isn't new breakage. It's pre-existing behavior the default was hiding. The CCU work didn't cause it — it just turned the lights on.

06How bad is it, honestly
cosmetic▲ hereoutage

Low-to-moderate. Annoying, not dangerous.

  • ✅ Self-healing. Failures are transient and retried on the next tick — no persistent bad state.
  • ✅ Not a flood. Sequential, so it won't take Cube down with concurrent connections.
  • ✅ Infrequent. The legacy pass is gated to ~6h, not every minute.
  • ⚠️ Wasteful. ~150–300 cold queries to populate a badge that only needs a handful of metrics.
  • ⚠️ Partial data. ~20% of badge cells are stale/missing each pass because their read failed.
  • ⚠️ Trino pressure. Minutes of back-to-back cold scans every cycle — real load for a low-value output.
07If we want to keep it on — three ways to make it safe
OPTION 1 · recommended

Cap the legacy path

Port the SQLite mode's queryBudget() guard into scanGameLegacy(). Closes the exact asymmetry — smallest, most surgical change.

Effort: low · removes the marathon
OPTION 2

Badge from the watchlist

Feed the badge from the curated ANOMALY_METRICS list instead of the full registry. Fewer reads, all of them cheap. Changes badge coverage semantics.

Effort: low–med · narrows coverage
OPTION 3

Scope the games

Drop ANOMALY_DETECTOR_GAMES to just the CCU games (jus,ptg). Stop-gap — cuts the burst by removing ballistar, doesn't fix the uncapped walk.

Effort: trivial · env-only, partial
