# Red-Team Plan Review — Cube Advisor (Failure Mode Analyst / Flow Tracer)

- Reviewer lens: hostile Failure Mode Analyst; verification role Flow Tracer
- Date: 2026-06-03
- Plan: `plans/260603-1842-cube-advisor-architect-briefing-console/`
- Method: traced button → POST /runs → spawn claude → stream-json → SSE → ideas.json → ingest → dedup → store → render; grep-verified claims against cube-playground.

Verdict: the happy path is plausible; the failure path is largely unhandled. The plan treats the spawned-claude subprocess as a well-behaved function that always writes valid JSON and exits cleanly. Production reality is the opposite. cube-playground already solved most of these problems in `chat-service` (bounded ring buffer, event-loop monitor, AbortController-driven timeout, the Agent SDK rather than raw CLI parsing) — and this plan re-derives a weaker version while ignoring those solutions.

## Finding 1: ideas.json is the single load-bearing artifact, and every way it can fail to exist is unhandled
- **Severity:** Critical
- **Location:** Phase 3 §Requirements ("On child exit 0 → read+hand off `ideas.json` to ingest, mark run `done`"); Phase 4 §TDD step 4 ("malformed run quarantined").
- **Flaw:** The contract is "exit 0 ⇒ ideas.json is present and valid." Nothing enforces that the child actually wrote the file. The model is non-deterministic: it can narrate a successful run, exit 0, and never call the Write tool; it can write to a different path (the prompt says `runs/<id>/ideas.json` but the child's cwd is `../cube-playground`, per Phase 3 §Arch `spawn('claude', …, { cwd })` — so a relative `runs/<id>/` resolves under cube-playground, not the advisor repo); or write truncated JSON if killed mid-write.
- **Failure scenario:** User clicks Generate. 12 minutes pass, deep-research runs, child exits 0. Backend does `fs.readFile(runs/<id>/ideas.json)` → ENOENT (model wrote a summary to stdout instead, or wrote under the wrong cwd). Phase 3 says exit 0 ⇒ ingest; ingest gets no file. If this throws inside the exit handler (async, no surrounding catch shown anywhere in Phase 3/4), the run row is never transitioned and is stuck `running` forever — which then trips the single-run lock (Finding 4) and bricks the app until manual DB surgery.
- **Evidence:** Phase 3 only branches on exit code, never on file existence/validity *before* deciding `done` vs `failed`. Phase 4 quarantine covers *malformed* JSON but presupposes the file was read successfully. cwd ambiguity: Phase 3 §Architecture `spawn('claude', [...recipeArgs], { cwd })` with Phase 1 §Architecture stating "cwd at repo root" (= cube-playground), while the prompt (Phase 5) tells the model to write `runs/<id>/ideas.json` — two different roots, no absolute path pinned.
- **Suggested fix:** Make the orchestration prompt write to an **absolute** path injected by the backend (`{{IDEAS_OUTPUT_ABSPATH}}`), not a relative one. After child exit, classify on a 4-way matrix: (exit 0 + file present + schema-valid) ⇒ done; (exit 0 + file missing) ⇒ failed `no_output`; (exit 0 + invalid JSON) ⇒ failed `malformed`; (exit ≠ 0) ⇒ failed `child_error`. Wrap the entire exit handler in try/catch that *always* transitions the run row out of `running` in a `finally`. Add a test: stub-claude that exits 0 without writing the file ⇒ run ends `failed`, lock released.

## Finding 2: Backend parses raw `claude -p` CLI stream-json, but cube-playground proves the durable integration is the Agent SDK — the plan picks the brittle path and its fixture is self-referential
- **Severity:** High
- **Location:** Phase 3 §Arch (`stream-json-parser.ts`), Phase 3 §Risk ("schema may evolve → parser is tolerant … covered by a captured fixture"); fixture sourced in Phase 3 §TDD step 1 "captured from Phase 1 probe."
- **Flaw:** cube-playground does NOT parse `claude -p` stdout. `chat-service/src/core/claude-runner.ts` consumes the **Agent SDK** message stream programmatically (`result` message with `session_id`, `total_cost`, structured `assistant`/`tool_result` events; abortSignal support). The advisor instead shells out to the CLI and re-implements line-delimited JSON parsing by hand. The CLI stream-json envelope is an undocumented, version-coupled surface; the plan's only safety net is one fixture captured from the user's *current* `claude` version (2.1.161). A `claude` CLI upgrade silently changes event shapes; the "tolerant parser" only skips *unknown* event types — it cannot detect that the event it *thought* it understood changed meaning. The fixture is also circular: Phase 1 produces it, Phase 3 tests against it, so the test only ever asserts "the parser parses the thing we captured," never "the parser handles what the *next* claude version emits."
- **Failure scenario:** User runs `claude` auto-update. Next briefing: the `result`/`assistant` envelope nesting shifts. Parser silently yields zero progress items (all events "unknown"), SSE shows a dead progress log for 12 minutes, then ideas.json *does* land (model still wrote it) — so the run "succeeds" with a UX that looked frozen. Or worse: the exit-detection line moves and the runner never sees exit, relying solely on the 15-min timeout to kill every run.
- **Evidence:** `chat-service/src/core/claude-runner.ts:166,196,260-271` — existing integration reads SDK `result.session_id`/`result.conversation_id` from structured messages, not parsed CLI stdout. `grep child_process|spawn` across `chat-service/src` returns only comments — no production CLI-spawn exists to model this on.
- **Suggested fix:** Strongly consider the Agent SDK (`@anthropic-ai/claude-agent-sdk` or whatever `chat-service` already depends on) instead of CLI stdout parsing — it's already a proven dependency in this monorepo and gives structured events + abortSignal + cost for free. If CLI spawn is required for MCP inheritance (validate in Phase 1 whether the SDK inherits `~/.claude.json` MCP auth too — that may moot the whole CLI decision), then pin the `claude` version in `run-config`, assert it at boot, and fail loudly on mismatch rather than silently mis-parsing.

## Finding 3: better-sqlite3 is synchronous — the child-exit ingest transaction blocks the event loop while SSE is streaming, and cube-playground already has a monitor that fires on exactly this
- **Severity:** High
- **Location:** Phase 4 §Implementation step 3 ("ingest … in a transaction"); Phase 4 §Arch dedup/rank "pure functions"; Phase 3 §Requirement "output streamed incrementally (no buffering whole run in memory)."
- **Flaw:** `better-sqlite3` is fully synchronous (verified dependency: `server/package.json:19`, `chat-service/package.json:19`). The Phase 4 ingest = validate (zod over N ideas) + dedup (fingerprint normalization over the whole prior backlog) + N upserts, all inside one synchronous `db.transaction()`. That transaction runs on the single Node thread. While it runs, the SSE route cannot flush bytes, the health check stalls, and a concurrent `PATCH /api/ideas/:id` cannot interleave. cube-playground explicitly instruments this hazard: `chat-service/src/index.ts:127` ships `startEventLoopMonitor` precisely because "synchronous DB work blocks the single thread and starves cheap polls (the 504 cause)." The advisor plan adds zero mitigation and no monitor.
- **Failure scenario:** Backlog grows to a few hundred ideas over months. A run finishes; ingest dedups every candidate against the full backlog synchronously. For the ~hundreds-of-ms the transaction holds the thread, any other request (the frontend's post-`done` `GET /api/ideas` refetch, a second tab's health poll) blocks. Single-user app so impact is bounded, but the "no buffering / incremental streaming" promise is violated at exactly the moment of run completion, and any future multi-client use degrades sharply.
- **Evidence:** `chat-service/src/index.ts:120-135` (event-loop monitor exists specifically for sync-DB-blocking); `server/package.json:19` / `chat-service/package.json:19` better-sqlite3 is sync by design (no async API).
- **Suggested fix:** Keep ingest transactions small and bounded; index the fingerprint column (Phase 2 schema lists no index on `fingerprint` despite dedup keying on it — add it). Port `startEventLoopMonitor` from chat-service. Run validate+dedup (pure, no I/O per Phase 4) *outside* the transaction; keep only the upserts inside it. Document that ingest is a deliberate synchronous critical section and that the run is already `done` from the user's perspective before it starts (so a brief stall is acceptable), OR resolve the race in Finding 5 first.

## Finding 4: Single-run lock + timeout kill leaves the run row and the lock in an inconsistent state on the unhappy path — the lock outlives the run
- **Severity:** Critical
- **Location:** Phase 3 §Requirement "one run at a time (reject concurrent with 409)"; "child killed on timeout"; "on non-zero/timeout → mark `failed`."
- **Flaw:** The lock's nature is unspecified: is it the presence of a `runs` row with `status='running'`, or an in-memory flag? Both fail differently. (a) If the lock is the DB `running` row and the exit handler throws (Finding 1) or the server crashes mid-run, the row stays `running` and every future `POST /api/runs` returns 409 forever — no recovery path, no "stale run reaper" mentioned anywhere. (b) If the lock is an in-memory boolean, a server restart drops it but leaves the DB row `running` and the child orphaned (Finding 6), so the two views disagree. The timeout path is asserted ("timeout kills child") but the ordering — kill child → wait for exit event → transition row → release lock — is not specified, and `child.kill()` is async/best-effort; `SIGTERM` to a `claude` mid-MCP-call may not terminate promptly, so the lock may release before the zombie actually dies.
- **Failure scenario:** Run #1 hits the 15-min timeout. Backend sends SIGTERM, marks run `failed`, releases lock. The `claude` child ignores SIGTERM (blocked in a network MCP fetch) and keeps running for another minute, still holding the OAuth MCP session and possibly still writing `ideas.json`. Meanwhile user, seeing "failed," clicks Generate again → run #2 spawns → two `claude` children now race to write overlapping `runs/<id>/` outputs and double-spend tokens. Inverse case: exit handler throws → row stuck `running` → app permanently returns 409 until someone hand-edits SQLite.
- **Evidence:** Phase 3 lists no stale-run reaper, no startup reconciliation (mark dangling `running` rows `failed` on boot), and no `SIGKILL` escalation. Contrast chat-service `stream-registry.ts:14-21` which has an explicit lifecycle (`finish` + TTL sweeper) and `findRunning(sessionId)` reconciliation — the advisor reinvents the lock without that hygiene.
- **Suggested fix:** Define the lock as the DB `running` row + an in-memory PID handle, reconciled on boot (any `running` row with no live child → mark `failed`/`interrupted`). Timeout = SIGTERM, then SIGKILL after a grace window, and only transition the row + release lock in the child's actual `exit` event handler (in a `finally`), never optimistically. Add a manual "force-clear stuck run" endpoint or a stale-run reaper (running > timeout+grace ⇒ failed).

## Finding 5: Race between child-exit ingest and a concurrent status PATCH / refetch — read-modify-write on idea status with no serialization
- **Severity:** High
- **Location:** Phase 4 §Arch ingest "upserts ideas by fingerprint … merge (bump `run_last`, keep earliest `run_first`)"; `PATCH /api/ideas/:id` "sets status + writes `idea_status_log`"; Phase 6 GenerateButton "refreshes cards on `done`."
- **Flaw:** Ingest and PATCH both read-modify-write the `ideas` table, and the plan gives ingest a *merge* (read existing row → compute → write) rather than a single atomic upsert statement. If a PATCH lands between ingest's read and write, one clobbers the other. Concretely: an idea the user just `dismissed` gets re-ingested by a run whose `dedupVerdict` says `new` (fingerprint guard merges it) and ingest's merge resets its status, un-dismissing it — directly violating the success criterion "Re-runs don't re-pitch already-dismissed ideas." The plan's merge spec only mentions `run_first`/`run_last`, never "preserve user-set status," so status handling on merge is undefined.
- **Failure scenario:** Run completes, ingest merges a candidate that fingerprint-matches a previously *dismissed* idea. Ingest's merge writes the row without preserving `status='dismissed'` (status not in the merge field list). The dismissed idea reappears as `new` in the next briefing. The dedup loop the whole product is built around is silently broken.
- **Evidence:** Phase 4 §Arch merge rule enumerates only `run_first`/`run_last`; status preservation is absent. No transaction isolation or row-versioning between PATCH and ingest is specified. better-sqlite3 sync helps single-threaded atomicity *within* one statement but the merge is multi-statement (read then write).
- **Suggested fix:** Specify merge semantics explicitly: on fingerprint match, **never** downgrade a user-set terminal status (`dismissed`/`snoozed`/`accepted`/`shipped`) back to `new`. Do the merge as a single `INSERT … ON CONFLICT(fingerprint) DO UPDATE` that only touches `run_last`/`payload_json` and leaves `status` untouched when already set by a human. Add a dedup test: pre-existing `dismissed` idea + re-ingest with `new` verdict ⇒ stays `dismissed`.

## Finding 6: Timeout kill leaves orphaned/zombie children; no process-tree kill, and `claude -p` likely spawns its own MCP subprocesses
- **Severity:** High
- **Location:** Phase 3 §Risk "Orphaned child processes on crash → ensure kill on server shutdown + run timeout."
- **Flaw:** This risk is *named* but not *designed*. `child.kill()` kills only the direct `claude` process, not the process tree. `claude` with MCP connectors spawns/holds child connections (MCP servers, network sessions). On timeout or server shutdown, killing the parent leaves descendants reparented to init (orphans) still holding OAuth sessions and possibly mid-write to `runs/<id>/`. The plan has no `detached`+`process.kill(-pid)` group-kill, no `SIGKILL` escalation, and no shutdown hook implementation — only a sentence in §Risk.
- **Failure scenario:** 15-min timeout fires on a deep-research run that's spawned MCP helper processes. Backend kills `claude`; the MCP fetch subprocess survives, finishes its Confluence pull, and a half-written `ideas.json` appears *after* the run was marked `failed`. Next run reads a stale/partial file. Repeated timeouts accumulate orphaned processes that exhaust the user's machine over a day of use.
- **Evidence:** Phase 3 mentions kill only in prose, no implementation step covers process groups or signal escalation. No `server.ts` shutdown hook is listed in Phase 2/3 file lists. (chat-service uses the SDK's `abortSignal` to unwind cleanly — `claude-runner.ts:223,254-257` — which the CLI-spawn approach forfeits.)
- **Suggested fix:** Spawn with `{ detached: true }` and kill the whole group via `process.kill(-child.pid, 'SIGTERM')`, escalate to `SIGKILL` after grace. Register a `process.on('SIGINT'/'SIGTERM'/beforeExit')` shutdown hook that kills any live run group and marks its row `interrupted`. Write `ideas.json` to a temp name and atomically rename on the child's clean completion so a late/partial write can never be ingested.

## Finding 7: SSE replay buffer, server restart mid-run, and reconnect are asserted but not designed — and the plan ignores the bounded-ring solution already in the repo
- **Severity:** High
- **Location:** Phase 3 §Arch "SSE route holds the live run's emitter; late subscribers get a replayed buffer of progress-so-far then live tail"; Phase 6 `useRunStream` / GenerateButton.
- **Flaw:** "Replayed buffer of progress-so-far" with no bound = unbounded in-memory growth over a 15-min run emitting potentially thousands of tool-use lines (deep-research × ~10 candidates). cube-playground already solved this with a **bounded ring buffer + offset math + TTL sweeper** (`stream-registry.ts:19-21,33-38`), and the advisor plan neither reuses it nor bounds its own buffer. Worse: the buffer is in-memory only. On server restart mid-run (better-sqlite3 + nodemon dev reload is common), the emitter and buffer vanish but the child (orphaned per Finding 6) and the DB `running` row persist. The reconnecting browser's `EventSource` auto-reconnects to `GET /api/runs/:id/stream` for a run whose emitter no longer exists → the route has no live emitter and no replay source → either hangs or 404s, and the user has no path back to a run that may still be producing ideas.json.
- **Failure scenario:** Dev edits backend during a run (nodemon restarts) → run row `running`, child orphaned, browser EventSource reconnects to a dead stream, GenerateButton stays disabled (reflects 409 from the still-`running` row, Finding 4), user is wedged with no progress and no way to start over.
- **Evidence:** Phase 3 §Arch describes the replay buffer in one sentence with no size bound, no persistence, no reconnect/`Last-Event-ID` handling. cube-playground's superior pattern: `stream-registry.ts:19-21` (`STREAM_REGISTRY_RING_SIZE` default 2000, `MAX_TURNS` 100, TTL sweeper) and the existing replay endpoint `server/src/routes/chat.ts:241-249` (`?from=<offset>`). The advisor reinvents this weaker.
- **Suggested fix:** Port/adapt the chat-service ring-buffer registry (bounded events + offset + TTL). Support `?from=<offset>` replay and emit SSE `id:` lines so the browser's `Last-Event-ID` reconnect resumes cleanly. On boot, reconcile orphaned `running` rows (Finding 4) so a restarted server doesn't leave the UI wedged.

## Finding 8: Cost/latency — a full run is deep-research × ~10 candidates; the 15-min hard timeout will kill legitimate runs, destroying the work and the spend
- **Severity:** High
- **Location:** Phase 3 §Requirement "hard timeout (configurable, default 15 min); child killed on timeout"; brainstorm §Risks "full run = deep-research × ~10 candidates + repo reads → minutes + meaningful spend."
- **Flaw:** The plan simultaneously claims (a) deep-research per candidate across ~10 candidates × 2 categories (Phase 5: "top-5 candidates per category" = up to 10, each fan-out researched) + full repo scan + Confluence + Outlook + WebSearch, and (b) a 15-min hard kill. Deep research alone is multi-minute per topic; 10+ of them serialized inside one session realistically blows past 15 minutes. The timeout then kills a run that was *working*, after spending real tokens, producing nothing ingestable (Finding 1), and leaving the user worse off than no feature. There is no partial-result capture: a run killed at minute 15 with 8/10 ideas already researched discards everything because ideas.json is written once at the end.
- **Failure scenario:** First real briefing on a cold backlog: model scans repo, pulls Confluence, runs deep-research on candidate 7/10 at the 15-min mark. Timeout SIGTERMs the child. No ideas.json (or a truncated one). Run marked `failed`. User spent meaningful API cost for a blank dashboard, retries, hits the same wall. The product's headline success criterion ("One click → within minutes, 3 researched ideas/category render") is unachievable under these two constraints together.
- **Evidence:** Phase 5 §Requirement "top-5 candidates per category … deep-research per candidate"; brainstorm line 64 acknowledges "minutes + meaningful spend" but the plan never reconciles that estimate against the 15-min number — no measured baseline exists (Phase 1 captures latency for a *single* Confluence fetch, not a full fan-out run).
- **Suggested fix:** Measure a real full-run wall-clock in Phase 1/Phase 5's "guarded real run" *before* fixing the timeout; set the default from data, not a guess. Have the prompt write `ideas.json` **incrementally** (append each idea as researched, or checkpoint after each candidate) so a timeout still yields a partial, ingestable briefing. Make the timeout a soft warning + a higher hard kill. Reduce default fan-out (e.g. research top-3, list the rest shallow) to fit a realistic budget. Surface running token cost in the SSE progress (the SDK `result.total_cost` gives this for free — another reason to prefer Finding 2's SDK path).

## Finding 9 (Medium): Idempotency claim contradicts dedup-by-fingerprint; fingerprint collisions and dangling dedupVerdict ids unhandled
- **Severity:** Medium
- **Location:** Phase 4 §Requirement "ingest is idempotent (re-ingesting same run changes nothing)"; §Arch fingerprint = "normalized title+problem tokens"; dedupVerdict `duplicate-of:<id>`.
- **Flaw:** Two distinct failure shapes. (a) Idempotency vs merge: "re-ingest is a no-op" only holds if the run id is recorded and re-ingest short-circuits on it; but the merge logic keys on *fingerprint*, and re-ingesting bumps `run_last` — which is a write, not a no-op. The claim and the mechanism contradict unless run-level idempotency is explicit (it isn't in the steps). (b) Fingerprint over "normalized title+problem tokens" is a lossy hash: two genuinely different ideas with similar titles ("Improve cube query caching" vs "Improve cube query latency") can collide and get wrongly merged, silently dropping a real idea. (c) `dedupVerdict: duplicate-of:<id>` trusts an LLM-emitted id; if the model hallucinates an id that doesn't exist in the backlog, the backend has no specified behavior — orphan reference, possible crash on a join, or a "duplicate" that points to nothing and is silently dropped from ranking.
- **Failure scenario:** Model marks candidate B `duplicate-of:idea_042`, but `idea_042` was purged or never existed. Backend either throws during ingest (whole run quarantined, Finding 1 cascade) or silently excludes B from ranking → a real idea vanishes with no trace.
- **Evidence:** Phase 4 §Arch defines fingerprint as token-normalized (lossy) and trusts `dedupVerdict` ids; §TDD step 2 tests "identical-by-fingerprint" merge but no test for *false-positive collision* or *dangling duplicate-of id*.
- **Suggested fix:** Make idempotency explicit at run granularity (skip if run already ingested) separate from cross-run fingerprint merge. Validate every `duplicate-of:<id>` against existing ids during ingest; on miss, downgrade to `new` and log a warning rather than dropping. Add the fingerprint a normalized + length/structure component to reduce collision, and add a collision test (two different ideas, similar titles ⇒ NOT merged). Keep manual un-merge/dismiss as backstop.

## Finding 10 (Medium): Plan-handoff path target inconsistent with stated convention; idempotency-by-slug can silently overwrite a different idea's brief
- **Severity:** Medium
- **Location:** Phase 8 §Requirement/§Arch — writes to `cube-playground/plans/reports/advisor-<date>-<slug>-brief.md`, "new file only, never overwrites," yet "idempotent per idea (re-accept updates the same brief)."
- **Flaw:** "Never overwrites" and "re-accept updates the same brief" are contradictory by definition. More dangerously, the filename is `advisor-<date>-<slug>` keyed on a *title slug*, not the idea id. Two different ideas with the same slugified title (or the same idea re-accepted on a different `<date>`) produce either a collision (overwrite a different idea's brief) or a duplicate file (breaking the "same brief" idempotency). The path guard is correct in spirit but the identity key is wrong. Also, the plan writes into `plans/reports/` — but cube-playground's plan convention (per the repo's own docs and the `plans/<timestamp-slug>/` dirs observed) puts briefs/brainstorms under timestamped names; an `advisor-` prefix in the shared `reports/` dir is a new naming the cube-playground side hasn't agreed to, and a misconfigured `CUBE_PLAYGROUND_DIR` would silently write into the wrong repo.
- **Failure scenario:** Architect accepts idea "Improve cube caching" twice across two days → two files (`advisor-260603-…` and `advisor-260604-…`), violating "same brief." Or two distinct ideas slugify identically → second accept overwrites the first's brief, losing content despite the "never overwrite" promise.
- **Evidence:** Phase 8 §Arch filename `advisor-<date>-<slug>-brief.md` keyed on date+slug, while §Requirement claims per-idea idempotency — id is absent from the filename. Verified the target dir exists and is shared: `plans/reports/` contains many human/agent reports.
- **Suggested fix:** Key the filename on the stable idea id (e.g. `advisor-<ideaId>-<slug>-brief.md`), making re-accept genuinely target the same file (resolves the overwrite/idempotency contradiction). Store the written path on the idea row so the UI link and re-accept are deterministic. Validate `CUBE_PLAYGROUND_DIR` resolves to a real cube-playground checkout (check for a sentinel file) before writing, in addition to the traversal guard.

## Cross-cutting observation

The strongest single recommendation: **this monorepo already built the hard parts** — a bounded SSE ring buffer with offset replay and TTL (`chat-service/src/core/stream-registry.ts`), an event-loop monitor for sync-DB stalls (`chat-service/src/index.ts:127`), and a structured Agent-SDK integration with AbortController-driven cancellation, cost, and session reconciliation (`chat-service/src/core/claude-runner.ts`). The plan re-derives weaker versions of all three (unbounded buffer, no monitor, hand-rolled CLI stdout parsing). Before Phase 3, decide explicitly: reuse the SDK + registry patterns, or document *why* the CLI-spawn + bespoke parser is required (the only defensible reason is MCP-auth inheritance — which Phase 1 must prove the SDK *can't* do, or the CLI choice is unjustified).

## Unresolved questions

1. Does the Agent SDK (already a dependency in `chat-service`) inherit the user's `~/.claude.json` MCP connectors when run programmatically? If yes, Findings 2/6/8 collapse toward "use the SDK" and most of the subprocess-lifecycle risk evaporates. Phase 1 should test SDK-MCP inheritance, not only CLI-MCP.
2. Is the single-run lock DB-backed or in-memory? The whole of Finding 4's failure surface depends on this and the plan never states it.
3. What is a real full-run wall-clock and token cost? Without a measured baseline the 15-min timeout (Finding 8) is a guess that likely kills legitimate runs.
