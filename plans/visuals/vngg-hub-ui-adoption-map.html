Same brand, same font — opposite stacks. Converge at the token layer.

cube-playground and VNGG Hub UI already share the exact brand orange #f05a22, Inter, and a data-theme light/dark model. But Hub UI ships as Tailwind v4 + tailwind-variants + shadcn-style copy-in, while cube-playground is Ant Design + cube ui-kit + inline style={{var()}} with no Tailwind. So full adoption is a stack migration. The high-leverage move is to converge the token contract and mirror component anatomy — not lift-and-shift.

1 Foundation is already shared

Brand #f05a22 + Inter + data-theme are identical. Convergence starts from agreement, not a rewrite.

2 Token taxonomy is the gap

cube uses flat soft/ink status pairs; VNGG uses a state-aware fill matrix (intent × variant × state). Adopt the structure.

3 Mirror, don't import

The CLI hard-requires Tailwind v4. Replicate component specs in cube's inline-var stack; reserve real adoption for greenfield.

adopt-as-token structure portable as pure CSS vars mirror-anatomy rebuild spec in cube's stack skip / defer not worth swapping now Use the radius + theme toggles ↗ — the VNGG & converged samples respond live.

02

Token taxonomy — flat pairs → state-aware fill matrix

cube-playground status colors are flat 2-token pairs (--success-soft / --success-ink). VNGG generates a 3-axis matrix: --color-bg-fill-{intent}-{variant}-{state}. The same intent renders solid / subtle / ghost / flat across default / hover / pressed — so a button, a chip, and a banner all draw from one coherent ladder. This is the single most portable thing to adopt: it's pure CSS variables, no Tailwind required.

The free upgrade — multi-axis theming knobs

cube-playground already ships data-theme. VNGG layers more product-wide knobs on the same mechanism: data-radius (rounded / standard / smooth / luxury), data-emphasis (medium / strong), data-component-theme (brand / neutral). Each is a global restyle with zero component edits. Use the radius toggle in the header — every VNGG & converged sample on this page reshapes live. That's the payoff of a token-driven contract.

05

Adoption roadmap — leverage ÷ effort

Layered, not lift-and-shift. Tiers 1–2 are pure CSS and bank most of the visual consistency; Tier 3 is the bulk of component work; Tiers 4–5 are policy.

1

Converge the token taxonomy

Re-express cube's semantic tokens in VNGG's structure (--color-content-*, --color-bg-fill-{intent}-{variant}-{state}, 12-step scales). Brand + font already match; soft/ink slots in as the subtle column. Pure CSS — the recent single-token-layer refactor makes this a contained remap.

low · CSS only
2

Adopt the extra theming axes

Add data-radius / data-emphasis alongside the existing data-theme. Product-wide restyle knobs, no component edits.

low
3

Mirror component anatomy (~18 overlapping)

Rebuild the spec for Button, Table, Metric-card, Tabs, Segmented, Dialog, Toast, Alert, etc. in cube's inline-var stack — same variants/states/sizing, no Tailwind. Gate each on the existing visual-regression suite.

medium · iterative
4

Greenfield rule

Any NEW standalone app / micro-frontend starts on VNGG Hub UI directly (org standard, real component adoption + MCP scaffolding) — not cube's bespoke stack.

policy
5

Don't swap now

recharts (fine as-is), Ant Design wholesale (too costly), and the Tailwind-v4 requirement (blocker for in-place adoption) all stay. Re-evaluate only if the org mandates standardization.

hold

Open questions before committing

1 · Mandate or opt-in? Is there an org directive to standardize on Hub UI, or is convergence discretionary? Decides whether Tier 1 is “nice-to-have” or “required”.
2 · Tailwind appetite? Willing to introduce Tailwind v4 alongside Ant (unlocks real component vngghui add), or stay inline-var + mirror (Tier 3)?
3 · Canvas identity? Keep cube's warm cream as a deliberate signature, or align to VNGG's cool neutral grey for cross-app consistency?
