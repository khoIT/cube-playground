# mf_users pre-agg coverage optimization

Cut the Trino load generated by `mf_users` queries that bypass CubeStore rollups. Verified
against the live `gio-trino` buffer (19 Jun, 15:06–15:32 GMT+7): `mf_users` = top cost, 222
Trino queries, **17 ≥5 min (max 8.2 min)**, ~55 GB scanned. `activity/login` is already
cheap (std daily table) and is OUT of scope.

## Evidence (verified, do not re-litigate)

- HEAVY aggregate (`ltv_total_vnd`+`user_count` by `churn_risk`, `whales` seg): `/sql` →
  `external:false`, `preAggregations:[]` → **rollup miss**, raw Trino.
- COHORT (`ltv_total_vnd` by `country`×`install_date`): `/sql` → `external:true`, hits
  `preagg_cfm.mf_users_ltv_by_install_cohort_batch` → **rollup works** for cohort shapes.
- `user_count` = `count_distinct` exact, non-additive (`cube-dev/cube/model/cubes/cfm/mf_users.yml:256`);
  rollups carry only `user_count_approx` (HLL, mergeable).
- **The ≥5 min queries are member-grain detail** (`user_id, ingame_name, ltv_total_vnd,
  lifecycle_stage, last_active_date …` per user), rooting in / joining
  `etl_ingame_moneyflow`. Dimension rollups can NOT serve these (they return rows per user).
- No `originalSql` pre-aggs exist anywhere in the model.

## Two distinct problems

| # | Workload | Shape | Fixable by | Effort | Value |
|---|----------|-------|-----------|--------|-------|
| P1 | **Member-detail** (segment member-list, member-360, Pull-API enrichment) | per-user projection, joins raw moneyflow | materialize the snapshot path (originalSql / user-grain rollup) OR fix join-root OR upstream column | high | **high** (the 8-min queries) |
| P2 | **Segment sizing / aggregates** (counts + LTV by churn_risk/payer_tier/lifecycle) | group-by dims | a segment-oriented dimension rollup | low | medium |

## Key decisions (LOCKED by user 19 Jun — do not re-litigate)

1. **Counts → APPROX.** Use `user_count_approx` (HLL, ~2% error, mergeable) for segment-sizing
   rollups so ONE rollup serves many dim combos. Exact membership *pulls* keep using the
   identity projection (already cheap), so approx is acceptable for sizing/preview.
2. **P1 direction → DO ALL THREE (A+B+C).** (A) `originalSql` pre-agg materialising the
   mf_users snapshot in CubeStore; (B) fix the join-root so member queries read the snapshot
   table not `etl_ingame_moneyflow`; (C) push any missing fields upstream onto the snapshot.
   A/B land in-repo this effort; C is raised to the data team where a field is genuinely
   absent (flag per-game, don't fake).
3. **Rollout → ALL 8 GAMES.** Proceed to all games carrying `mf_users` regardless of current
   prod idleness (phase-02 pre-flight gate is informational only, not a stop).

## Success criteria

- The member-detail and segment-sizing shapes that today miss → `external:true` (CubeStore),
  verified by compiled `/sql` not just `usedPreAggregations`.
- Heaviest mf_users Trino query class (≥5 min) eliminated or reduced >5×.
- Rollup build cost bounded (sweep stays «hourly cadence); no future-seal/non-additive traps
  (see docs/lessons-learned.md + [[cube-rollup-authoring-rules]]).
