# Phase 05 — Chat-service awareness + verification

**Priority:** P0 (the payoff) · **Status:** ☐ todo · **Depends on:** cubes from 01–04

## Overview
A cube that exists isn't useful until the chat's diagnose/advise rail knows it exists, can query
it, and frames genre-relevant questions around it. The chat-service auto-discovers cubes via
`/meta` (no code change), but the *diagnostic grounding* and *starter questions* are driven by a
pre-generated per-game knowledge seed and a business-metrics catalog that must be refreshed.

## Key insights (from chat-service scout, file:line)
- **Auto-discovery (no code change):** `chat-service/src/core/cube-meta-cache.ts:40-64` fetches
  `/cube-api/v1/meta` with `x-cube-workspace` + `x-cube-game`; caches 60s. **`std_*`-prefixed cubes
  and views are stripped** (`stripAgentInvisibleCubes`, line 62) → our cubes are named without
  `std_`, so they appear automatically.
- **Query path:** `chat-service/src/tools/emit-query-artifact.ts:81-180` validates measures/dims
  against live `/meta`; needs each queried cube to expose a `type: time` dimension for time-range
  windows. Member resolution is fuzzy + workspace-tolerant (`nl-to-query/member-resolution.ts`).
- **Diagnostic grounding (must refresh):** `get_topic_knowledge` reads
  `chat-service/seed/game-topic-knowledge-seed.json` (per game×topic → verified questions + metrics).
  Regenerated by `npm run knowledge:pregenerate`. This is what tells diagnose/advise "jus's data now
  supports economy/garden/npc/role questions."
- **Starter questions:** `starter-question-templates.ts` fire automatically when required members
  exist in `/meta` (field-name match) — new cubes light up matching templates with no code change.
- **Business metrics catalog:** `server/src/routes/business-metrics.ts` auto-validates metric
  formulas against each game's `/meta`; new metrics referencing our cubes register per-game on next
  `/coverage` call. Optional but recommended for economy/engagement KPIs.
- **No per-game lever allowlist / hard data-gate exists** — the gate is the skill prompt + the
  knowledge seed. So refreshing the seed IS the mechanism that makes new surfaces diagnosable.

## Requirements
- Every new cube (01–04) verified present in `/meta` for `jus_vn`, with a time dimension.
- `game-topic-knowledge-seed.json` regenerated for `jus_vn` so diagnose/advise/explore reference
  the new economy, engagement (garden/npc), session, and role surfaces.
- Genre-relevant business metrics added/validated (economy net-flow, garden engagement rate,
  npc engagement rate, avg session min, per-role ARPPU) so `metric_explain` + coverage monitor expose them.
- A diagnose + explore smoke test against jus_vn passes (LLM finds and queries the new cubes).

## Implementation steps
1. **Per-cube meta check** (incremental, after each phase): `GET /cube-api/v1/meta` with
   `x-cube-workspace: local`, `x-cube-game: jus_vn`; confirm cube + measures + a time dim appear and
   are NOT stripped.
2. **Regenerate knowledge seed:** from `chat-service/`, run `npm run knowledge:pregenerate` (scoped
   to jus_vn if the script supports it); review the new jus_vn entries for the new surfaces; commit
   the updated `seed/game-topic-knowledge-seed.json`.
3. **Business metrics:** add genre KPIs referencing the new cubes to the catalog; hit
   `/api/business-metrics/coverage?game=jus_vn` and confirm they resolve `available` (not `draft`).
4. **Smoke test (host dev stack):** run `/diagnose "why did jus_vn recharge dip"` and
   `/explore "garden harvest by crop level last 14 days"` turns; confirm the rail calls
   `get_cube_meta`/`get_topic_knowledge`, picks the new cubes, and `emit_query_artifact` validates.
   (Reuse the `starter-question-batch-verify` skill harness if helpful.)
5. **Window disclosure:** ensure garden/npc (2-week) and money_flow (May-15 lag) caveats are present
   in the seed entries so the rail discloses rather than misreads.

## Success criteria
- All 01–04 cubes visible in jus_vn `/meta` with time dims.
- `get_topic_knowledge(game=jus_vn)` returns entries naming the new economy/engagement/role cubes.
- A `/diagnose` turn autonomously queries a new cube and returns a grounded, genre-aware answer.
- New business metrics show `available` for jus_vn in the coverage monitor.

## Risks
- Cube accidentally named with `std_` prefix → silently stripped from `/meta`. **Mitigation:** lint cube names in 01–04; this phase's meta-check catches it.
- Missing time dimension → `emit_query_artifact` can't build time-range queries. **Mitigation:** every cube exposes `log_date` as `type: time`.
- Knowledge seed regen burns gateway LLM key. **Mitigation:** run via local subscription auth (see memory: seed generation uses local subscription auth), not the gateway key.
- Serving Cube instance needs restart to pick up new rollups (DEV_MODE=false, no hot-reload). **Mitigation:** restart cube_api + worker after model reload before chat verification.

## Generalization
This whole plan is the per-game recipe. For the next game: re-run schema introspection
(`scripts/trino-query.mjs`), mirror phases 01–04 against that game's tables, build its own code
dictionary (phase 04), then run this phase. The `cube-dev/scripts/onboard-game-cube-model.mjs`
generator can scaffold the boilerplate.
