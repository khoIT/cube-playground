# Cube Data-Model Viewer

Interactive, all-products visualization of the Cube semantic layer: each product's
cubes as nodes, joins as edges (labeled with key-mapping + cardinality), and a
click-through drawer showing the full schema (dimensions, measures, segments,
descriptions). The data-driven successor to the static `../cfm_data_model.svg`.

## Use it

Two ways:

- **Via the gateway** (served alongside `/docs`): http://localhost:16000/model-view/
  The gateway mounts this directory read-only at `/model-view` (volume
  `./model-viewer:/app/model-viewer:ro`). Adding the mount the first time needs a
  container recreate: `docker compose up -d --force-recreate cube_gateway`. After
  that, regenerating data only needs `docker compose restart cube_gateway` (or
  nothing — assets are read fresh per request).
- **Standalone** — double-click `index.html` (no server needed; data is loaded as a
  plain `<script>`, libs are vendored under `lib/`, nothing is fetched at runtime).

Features:

- **Product picker** — switch between `cfm_vn`, `cros`, `jus_vn`, `tf`, `vga`, `ballistar_vn`.
- **Cluster boxes** — cubes are grouped into labeled, color-coded containers (User hub,
  Role bridge, Session/Behavior events, Recharge, Activity, Identity mapping, Profile),
  the interactive analogue of `../cfm_data_model.svg`. Drag a cluster box to move the group.
- **Curated positions** — by default the page uses the hand-arranged positions in
  `layouts.js` (clusters placed on a conceptual grid; see *Layout positions* below).
- **`Auto: TB/LR`** — re-run dagre auto-layout (ignores the curated positions); toggles
  direction. Useful as a fresh starting point before re-arranging.
- **`Save layout`** — **personal** default: persists to this browser's localStorage, which
  the load order prefers, so it loads on your next visit (this browser only).
- **`Publish to all`** — **server** default for **everyone**: POSTs the layout to the gateway
  (`/model-view-api/layout/<product>`), which overwrites `layouts.js` on disk — instantly the
  default for all users, no commit/redeploy. Requires `MODEL_VIEWER_WRITABLE=true` (on in dev,
  off in prod by default — it's an unauthenticated write, safe only behind network isolation).
  In prod the written file is overwritten by the next rsync deploy, so **commit `layouts.js`
  (via `Export`) for a durable team default.**
- **`Export`** — downloads `<product>-layout.json` to commit into `layouts.js`.
- **`Reset`** — discards your local save and returns to the committed/published `layouts.js` (or dagre).
- **`☀ Light` / `🌙 Dark`** — toggle the colour theme; your choice persists (localStorage).
- **Pan / box-select** — left-drag pans the canvas; **Shift + left-drag** draws a
  selection rectangle, then drag any selected node to move the whole group. Scroll zooms.
- **Click a cube** → opens the right drawer (joins, dimensions, measures, segments) **and**
  lights up that cube's connected arrows with their key-mapping labels, dimming the rest.
  Edge labels are hidden by default (clutter-free); click empty canvas to deselect.
- **Hover a cube** → description tooltip.
- **Search box** — highlight cubes by name; dims the rest.
- **"highlight a view"** — highlight the cubes a 360 view composes.

Position precedence: **localStorage (your Save) → `layouts.js` (Publish / committed) → dagre fallback**.
Three levels of "default": **Save** = just you (localStorage); **Publish to all** = everyone, live
(server writes `layouts.js`); **Export + commit** = everyone, durable across deploys (repo).

## Regenerate the data

The graph (nodes, edges, key-mappings, schema) is parsed straight from the cube
YAML — the join graph lives only there, not in Cube's `/v1/meta` API. After
editing anything under `../cube/model/`, regenerate:

```bash
python3 gen_model_graph.py
```

This rewrites `model-graph.json` (canonical) and `model-graph.js` (the
`file://`-safe loader the page reads via `window.MODEL_GRAPH`).

## Layout positions

Curated box positions live in `layouts.js` (`window.LAYOUTS`), generated by
`gen_layouts.py`. It arranges each cluster on a conceptual grid (hub centre, session
above, behaviour right, recharge / activity / identity around) and grids the cubes
inside each cluster — so the diagram reads clearly instead of relying on auto-layout.

```bash
python3 gen_layouts.py        # rewrites layouts.json + layouts.js
```

Tune the arrangement by editing `DEFAULT_ANCHORS` (per-cluster `(col, row)`) or
`PER_PRODUCT` overrides in `gen_layouts.py`, then re-run. Or arrange by hand in the
browser and hit **Save layout** — the downloaded `<product>-layout.json` is exactly the
`window.LAYOUTS[<product>]` value, so paste it into `layouts.js` to commit it as the default.
Run `gen_layouts.py` again after adding cubes so new ones get a position (else they fall
back to dagre).

## Files

| File | What |
|---|---|
| `gen_model_graph.py` | Parses `../cube/model/{cubes,views}/**/*.{yml,yaml}` → graph JSON |
| `model-graph.js` / `.json` | Generated graph data (committed so the page works as-is) |
| `gen_layouts.py` | Curated cluster-grid positions → `layouts.{js,json}` |
| `layouts.js` / `.json` | Curated box positions (`window.LAYOUTS`); committed |
| `index.html` | Single-file React + React Flow viewer (no build step) |
| `lib/` | Vendored React, React Flow, dagre, htm (proxy-safe, no runtime CDN) |

## Notes

- **Clusters** are a heuristic grouping for readability, not a semantic guarantee.
  etl event cubes split into *session* (direct `*__mf_users` join, e.g. login/logout)
  vs *behavior* (via the role bridge); everything else buckets by name keyword. Tune
  `cluster_of()` in `gen_model_graph.py` if a cube lands in the wrong box.
- Edges are de-duplicated per cube pair; the exact per-cube join list (both
  directions, with SQL key-mapping) is in each cube's drawer.
- Re-vendor libs (e.g. version bump) from a network with CDN access:
  `react@18`, `react-dom@18` (umd), `reactflow@11` (umd + style.css), `dagre@0.8.5`, `htm@3.1.1`.
